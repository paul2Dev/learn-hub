Agentic Workflow — Cum Lucrează Claude Autonom

De la un singur prompt la zeci de pași autonomi — cum funcționează bucla percepe-planifică-acționează și cum să o ghidezi corect.

Ce este un Agentic Workflow?

Un agentic workflow este modul în care Claude rezolvă task-uri complexe autonom, în mai mulți pași, fără să ceară confirmare la fiecare acțiune. Claude citește fișiere, editează cod, rulează comenzi, observă rezultatele și iterează — singur — până ajunge la obiectiv.

Diferența față de o conversație normală:

  • Conversație normală — Tu scrii un prompt, Claude răspunde, tu citești, tu decizi pasul următor.
  • Agentic workflow — Tu descrii obiectivul, Claude planifică și execută toți pașii, folosind unelte reale (citire fișiere, editare cod, rulare comenzi), până ajunge la rezultat sau are nevoie de clarificări.

Exemplu concret: "Adaugă autentificare JWT la API-ul meu Express" — Claude citește structura proiectului, găsește rutele existente, instalează pachetele necesare, scrie middleware-ul, actualizează rutele, adaugă teste. Zeci de pași, un singur prompt.

Bucla Agentic: Percepe → Planifică → Acționează → Observă

Orice agentic workflow urmează aceeași buclă internă, repetată până la finalizare:

  1. Percepe — Claude citește fișiere, explorează structura proiectului, înțelege contextul curent. Folosește unelte ca Read, Glob, Grep pentru a-și construi o imagine completă.
  2. Planifică — Pe baza contextului, Claude decide ce acțiuni trebuie făcute și în ce ordine. Planul poate fi implicit (în gând) sau explicit (scris ca o listă de pași înainte de execuție).
  3. Acționează — Claude execută un pas: editează un fișier, rulează o comandă, face un request. Fiecare acțiune folosește un tool specific cu parametri precisi.
  4. Observă — Claude citește rezultatul acțiunii: output-ul comenzii, conținutul fișierului editat, eroarea returnată. Dacă rezultatul nu e cel așteptat, ajustează planul și reia bucla.

Această buclă se repetă de câte ori e nevoie — de la 3 iterații pentru un bug fix simplu, la 50+ pentru o funcționalitate complexă.

Uneltele cu care Claude acționează

În fiecare pas de tip "Acționează", Claude folosește una din uneltele disponibile:

  • Read / Glob / Grep — Citire fișiere, căutare pattern-uri, localizare simboluri. Pasul de percepție.
  • Edit / Write — Modificare sau creare fișiere. Acțiunea principală pentru cod.
  • Bash — Execuție comenzi shell: rulare teste, instalare pachete, build, git. Cel mai puternic și cel mai riscant tool.
  • WebFetch / WebSearch — Accesare documentație sau căutare pe web (când e disponibil).

Permisiuni: Uneltele distructive (Bash cu comenzi care modifică sisteme, Edit pe fișiere critice) necesită aprobare explicită dacă nu ai setat permisiuni automate. Claude te va întreba înainte de fiecare acțiune nesigură.

Permission Model — Ce Aprobă Automat, Ce Cere Confirmare

Claude Code are un sistem de permisiuni cu două niveluri:

  • Permisiuni automate (citire) — Read, Glob, Grep rulează fără să ceară confirmare. Cititul e întotdeauna sigur.
  • Permisiuni cu aprobare (scriere/execuție) — Edit, Write, Bash cer confirmare la prima utilizare sau pentru comenzi considerate riscante.

Poți controla granularitatea în .claude/settings.json:

{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(npm run lint)",
      "Edit(*)"
    ],
    "deny": [
      "Bash(rm -rf*)",
      "Bash(git push*)"
    ]
  }
}

allow — rulează automat fără confirmare. deny — blocat complet, Claude nu poate executa. Tot ce nu e în nicio listă → cere confirmare manuală.

Modul autonom complet: Flag-ul --dangerously-skip-permissions dezactivează toate confirmările. Folosește-l doar în medii izolate (containere, CI) unde ai control total și nu există risc de date loss.

Cum să scrii prompts pentru agentic workflows

Un prompt bun pentru un workflow autonom are trei elemente:

  1. Obiectiv clar — Ce trebuie să existe la final, nu cum să ajungă acolo.
  2. Constrângeri explicite — Ce să nu atingă, ce convenții să respecte, ce să evite.
  3. Criterii de succes — Cum știe Claude că a terminat corect.
# Prompt slab — obiectiv vag, fără constrângeri
> Adaugă validare la formulare.

# Prompt bun — obiectiv clar + constrângeri + criterii
> Adaugă validare server-side pentru toate endpoint-urile POST
  din src/api/. Folosește biblioteca Joi deja instalată.
  Nu modifica fișierele de test existente.
  Când termini, rulează npm test — trebuie să treacă toate testele.

Cu al doilea prompt, Claude știe exact ce să facă, ce să evite și cum să verifice că a terminat corect.

Planul explicit înainte de execuție

Pentru task-uri complexe sau riscante, cere lui Claude să prezinte planul înainte să execute. Previi surprize și corectezi direcția fără să fi cheltuit timp pe execuție greșită:

> Înainte să modifici orice fișier, descrie-mi planul complet
  pentru a migra autentificarea de la sesiuni la JWT.
  Listează fișierele afectate și ordinea modificărilor.

# Claude prezintă planul
# Tu revizuiești, corectezi dacă e nevoie
# Apoi:
> Planul arată bine, execută-l.

Alternativ: Folosește skill-ul /plan sau agentul Plan care e specializat exact pentru asta — returnează planul fără să execute nimic.

Întreruperea și corectarea unui workflow în curs

Poți interveni oricând în mijlocul unui workflow autonom:

  • Esc — Oprește acțiunea curentă imediat. Claude se oprește și așteaptă instrucțiuni.
  • Mesaj de corecție — Trimiți un mesaj nou cu direcție corectată. Claude integrează feedback-ul și continuă.
# Claude e la mijlocul unui refactor
# Observi că a ales o abordare greșită
[Esc]
> Oprește-te. Abordarea cu HOCs nu e ce vreau — folosește hooks.
  Reia de la fișierul src/components/Auth.js cu abordarea corectă.

Claude nu "uită" ce a făcut deja — îți va spune ce fișiere a modificat până acum și va integra corecția ta în continuare.

Fă commit înainte de orice workflow lung. Dacă workflow-ul produce rezultate nedorite, git reset --hard HEAD te aduce înapoi la starea curată în câteva secunde. Fără commit, undo-ul e manual și dureros.

Când Claude se oprește și cere input

Chiar și în modul autonom, Claude se oprește când:

  • Întâlnește o decizie ambiguă — două abordări posibile cu trade-off-uri semnificative (ex: SQL vs. NoSQL pentru un nou modul).
  • Detectează un risc de date loss — ștergere de fișiere, drop de tabele, modificări ireversibile.
  • Primește o eroare neașteptată pe care nu o poate rezolva singur după 2-3 iterații.
  • Task-ul necesită informații pe care nu le are — credențiale, URL-uri de API, specificații de business.

Când Claude se oprește și întreabă, răspunsul tău reintră în buclă și workflow-ul continuă.

Verificare la final — nu presupune, confirmă

La finalul unui workflow lung, cere întotdeauna o verificare explicită înainte să considerai task-ul terminat:

> Acum că ai terminat implementarea, rulează toate testele
  și spune-mi dacă există și alte fișiere care ar trebui
  actualizate dar pe care nu le-ai atins încă.

Această întrebare prinde frecvent:

  • Documentație sau tipuri TypeScript care nu au fost actualizate
  • Alte module care importă ce ai modificat și trebuie ajustate
  • Fișiere de configurare (env.example, docker-compose) care reflectă vechea arhitectură

Rețeta completă pentru un workflow reușit

# 1. Checkpoint git înainte de orice
git add -A && git commit -m "checkpoint: before [task name]"

# 2. Cere planul, verifică-l
> Înainte să modifici ceva, prezintă planul pentru [task].

# 3. Aprobă planul și lansează execuția
> Planul e ok. Execută-l.

# 4. Monitorizează — intervine cu Esc dacă ceva e greșit

# 5. Verificare la final
> Rulează testele. Există și altceva de actualizat?

# 6. Review diff și commit
git diff
git add [fișierele relevante]
git commit -m "feat: [ce s-a implementat]"

Acest flux — checkpoint, plan, execuție, verificare, commit — transformă un task complex de ore într-un workflow controlat de minute.
