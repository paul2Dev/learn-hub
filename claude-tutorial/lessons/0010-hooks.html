Claude Hooks — Automatizare la Nivel de Sistem

Comenzi shell care rulează automat înainte sau după acțiunile lui Claude — format, teste și notificări fără nicio intervenție manuală.

Ce sunt Hooks?

Hooks sunt comenzi shell pe care le definești tu și care rulează automat în răspuns la evenimentele Claude Code. Când Claude editează un fișier, când rulează o comandă bash, când termină o sarcină — poți atașa comenzi care se execută automat, independent de el.

Diferența esențială față de skills: skills sunt instrucțiuni pentru Claude (ce să facă el). Hooks sunt comenzi de sistem care rulează în paralel cu Claude, declanșate de acțiunile lui — Claude nu le controlează și nu le poate ocoli.

Analogie: Hooks funcționează ca git hooks (pre-commit, post-merge) — rulează automat la anumite evenimente, indiferent de ce face Claude în acel moment.

De ce să folosești Hooks?

  • Format automat — Prettier sau ESLint rulează după fiecare editare, fără să îi spui lui Claude "și acum formatează". Codul e mereu curat.
  • Teste automate — Testele se declanșează automat după modificări. Claude vede output-ul și poate itera fără să îi ceri explicit să ruleze testele.
  • Notificări — O notificare desktop când Claude termină o sarcină lungă. Nu mai stai cu ochii pe terminal.
  • Audit trail — Loghezi automat ce comenzi bash execută Claude — util pentru debugging și securitate în echipă.
  • Garduri de siguranță — Blochezi comenzi periculoase înainte ca ele să fie executate, indiferent ce a decis Claude.

Tipuri de Hooks

Claude Code oferă 4 tipuri de hooks, corespunzând evenimentelor din ciclul de viață al unei sesiuni:

  • PreToolUse — Rulează înainte ca Claude să folosească un tool. Poți inspecta acțiunea și, dacă returnezi exit code non-zero, blochezi execuția.
  • PostToolUse — Rulează după ce Claude a folosit un tool. Ideal pentru format, teste, logging.
  • Notification — Rulează când Claude trimite o notificare (ex: termină și așteaptă input de la tine).
  • Stop — Rulează când Claude termină complet o sarcină și se oprește din proprie inițiativă.

Configurare — settings.json

Hooks se definesc în .claude/settings.json (project-level, commituit în git) sau ~/.claude/settings.json (global, doar pe mașina ta). Structura de bază:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\""
          }
        ]
      }
    ]
  }
}

Componentele cheie:

  • "matcher" — Ce tool declanșează hook-ul: "Edit", "Bash", "Write", "Read" sau "*" pentru toate
  • "type": "command" — Tipul hook-ului (în prezent doar command e disponibil)
  • "command" — Comanda shell de executat; are acces la variabilele de mediu Claude

Variabile de mediu disponibile

Claude Code injectează automat variabile în mediul hook-ului, pe care le poți folosi în comanda ta:

  • $CLAUDE_TOOL_NAME — Numele tool-ului declanșator (Edit, Bash, Write etc.)
  • $CLAUDE_TOOL_INPUT_FILE_PATH — Calea absolută a fișierului editat (pentru hooks pe Edit/Write)
  • $CLAUDE_TOOL_INPUT_COMMAND — Comanda bash ce urmează să fie executată (pentru hooks pe Bash)

Debug tip: Adaugă un hook temporar care face echo cu variabilele pentru a vedea exact ce primești: "command": "echo \"Tool: $CLAUDE_TOOL_NAME | File: $CLAUDE_TOOL_INPUT_FILE_PATH\""

Exemple concrete

1. Format automat cu Prettier după fiecare editare:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\" 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}

2. Notificare desktop când Claude termină (Windows):

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "powershell -command \"Add-Type -AssemblyName System.Windows.Forms; [System.Windows.Forms.MessageBox]::Show('Claude a terminat!')\""
          }
        ]
      }
    ]
  }
}

3. Audit trail — logging comenzi bash:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[$(date +%Y-%m-%dT%H:%M:%S)] $CLAUDE_TOOL_INPUT_COMMAND\" >> ~/.claude/bash-audit.log"
          }
        ]
      }
    ]
  }
}

4. Rulare automată a testelor pentru fișierele de test modificate:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'if [[ \"$CLAUDE_TOOL_INPUT_FILE_PATH\" == *.test.* ]]; then npm test -- --testPathPattern=\"$(basename $CLAUDE_TOOL_INPUT_FILE_PATH .test.js)\" --passWithNoTests; fi'"
          }
        ]
      }
    ]
  }
}

Hooks care blochează execuția (PreToolUse)

Un hook PreToolUse poate opri acțiunea lui Claude returnând exit code non-zero. Claude primește output-ul hook-ului ca feedback și poate decide să nu continue sau să reformuleze.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'if echo \"$CLAUDE_TOOL_INPUT_COMMAND\" | grep -qE \"rm -rf|DROP TABLE|truncate\"; then echo \"BLOCKED: comandă distructivă detectată. Confirmă manual.\"; exit 1; fi'"
          }
        ]
      }
    ]
  }
}

Atenție: Hooks rulează cu permisiunile tale de utilizator și au acces complet la sistem. Testează-le cu atenție. O comandă greșită într-un PostToolUse rulează după fiecare editare — poate deveni costisitoare rapid.

Mai multe hook-uri pe același eveniment

Poți atașa mai multe hook-uri la același matcher — rulează toate în ordine:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\" 2>/dev/null || true"
          },
          {
            "type": "command",
            "command": "npx eslint --fix \"$CLAUDE_TOOL_INPUT_FILE_PATH\" 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}
