Composables și data fetching

useFetch, useAsyncData, $fetch și cum scrii composable-uri proprii cu reactivitate Vue 3.

Cele trei moduri de data fetching în Nuxt

UF

useFetch(url)

Shorthand convenabil — combină useAsyncData cu $fetch. Cel mai folosit pentru pagini.

recomandat
UAD

useAsyncData(key, fn)

Mai mult control: key explicit pentru deduplicare, funcție arbitrară (nu numai fetch). Ideal pentru logică complexă.

control maxim
$F

$fetch(url)

Fetch simplu fără SSR/caching — pentru event handlers, mutații, operații la click. Nu pentru date inițiale.

mutații

useFetch — pentru pagini și date inițiale

useFetch funcționează pe server (SSR) și pe client (navigare SPA). Pe server face fetch și inserează datele în HTML. Pe client reutilizează datele fără refetch. 

<!-- pages/products/index.vue -->
<script setup lang="ts">
interface Product {
  id: number
  name: string
  price: number
}

// Automat: SSR pe server, hydration pe client, fără refetch
const { data: products, status, error, refresh } = await useFetch<Product[]>('/api/products')

// Cu opțiuni
const { data: user } = await useFetch('/api/me', {
  headers: { 'Authorization': `Bearer ${token}` },
  pick: ['id', 'name', 'email'],    // selectează doar câmpurile necesare
  transform: (data) => data.user,  // transformă răspunsul
  watch: [userId],                  // re-fetch când userId se schimbă
})
</script>

<template>
  <div>
    <p v-if="status === 'pending'">Se încarcă...</p>
    <p v-else-if="error">Eroare: {{ error.message }}</p>
    <ul v-else>
      <li v-for="p in products" :key="p.id">
        {{ p.name }} — {{ p.price }} RON
      </li>
    </ul>
    <button @click="refresh()">Reîncarcă</button>
  </div>
</template>
Față de Next.js
În Next.js (App Router) faci const data = await fetch('/api/...') direct în Server Component. În Nuxt, useFetch gestionează SSR + hydration automat în SFC — nu ai Server Components separate, logica de fetch stă în <script setup> al paginii.

useAsyncData — pentru logică complexă

<!-- pages/dashboard.vue -->
<script setup lang="ts">
const userId = useSupabaseUser()  // presupunând că ai modulul Supabase

// Key explicit previne fetch duplicat dacă componenta se montează de două ori
const { data: stats } = await useAsyncData(
  'dashboard-stats',          // key unic — cache-uit între navigări
  async () => {
    // Funcție arbitrară — poți apela orice, nu numai $fetch
    const [users, orders, revenue] = await Promise.all([
      $fetch('/api/stats/users'),
      $fetch('/api/stats/orders'),
      $fetch('/api/stats/revenue'),
    ])
    return { users, orders, revenue }
  },
  {
    server: true,    // rulează pe server (implicit true)
    lazy: false,     // blochează randarea până termină (implicit false)
  }
)
</script>

$fetch — pentru mutații și event handlers

$fetch este un wrapper peste fetch nativ, cu suport pentru JSON automat, error handling și interceptori. Folosit pentru operații la click (nu pentru date inițiale): 

<script setup lang="ts">
const isLoading = ref(false)
const error = ref<string | null>(null)

async function createPost(title: string, content: string) {
  isLoading.value = true
  error.value = null

  try {
    const newPost = await $fetch('/api/posts', {
      method: 'POST',
      body: { title, content },
    })
    await navigateTo(`/blog/${newPost.slug}`)
  } catch (e: any) {
    error.value = e.data?.message ?? 'Eroare la creare'
  } finally {
    isLoading.value = false
  }
}
</script>

<template>
  <form @submit.prevent="createPost(title, content)">
    <input v-model="title" placeholder="Titlu" />
    <textarea v-model="content" />
    <p v-if="error" style="color:red">{{ error }}</p>
    <button :disabled="isLoading">
      {{ isLoading ? 'Se salvează...' : 'Publică' }}
    </button>
  </form>
</template>

Composable-uri proprii — cum le scrii

Un composable este o funcție care folosește Vue 3 Composition API (ref, computed, watch, etc.) și encapsulează logică reutilizabilă: 

// composables/usePosts.ts
export function usePosts() {
  const posts = ref<Post[]>([])
  const isLoading = ref(false)
  const error = ref<string | null>(null)

  async function fetchPosts(page = 1) {
    isLoading.value = true
    try {
      posts.value = await $fetch(`/api/posts?page=${page}`)
    } catch (e: any) {
      error.value = e.message
    } finally {
      isLoading.value = false
    }
  }

  async function deletePost(id: number) {
    await $fetch(`/api/posts/${id}`, { method: 'DELETE' })
    posts.value = posts.value.filter(p => p.id !== id)
  }

  return { posts, isLoading, error, fetchPosts, deletePost }
}
<!-- Folosit în pagini (auto-importat, fără import explicit): -->
<script setup lang="ts">
const { posts, isLoading, fetchPosts, deletePost } = usePosts()
await fetchPosts()
</script>

<template>
  <div v-if="isLoading">Se încarcă...</div>
  <ul v-else>
    <li v-for="post in posts" :key="post.id">
      {{ post.title }}
      <button @click="deletePost(post.id)">Șterge</button>
    </li>
  </ul>
</template>

useState — state global simplu

useState în Nuxt creează state reactiv shared între componente și persistent între navigări SSR→client: 

// composables/useUser.ts
export const useUser = () => useState<User | null>('user', () => null)

// Componenta A — setează userul
const user = useUser()
user.value = { id: 1, name: 'Paul' }

// Componenta B — citește același user (shared state)
const user = useUser()
console.log(user.value?.name)  // 'Paul'
useState vs ref
ref() creează state local componentei — fiecare componentă are propria valoare. useState(key) creează state global identificat prin key — toate componentele care apelează useState('user') partajează același obiect reactiv. Funcționează și pentru hydration SSR→client.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Care este diferența cheie între useFetch și $fetch?
Corect. useFetch este un composable care gestionează SSR (fetch pe server, date în HTML), deduplicare (nu face fetch de două ori pentru același key) și hydration (clientul preia datele fără refetch). $fetch e echivalentul lui fetch() din browser — simplu, fără magic.
Întrebarea 2 din 5 Când să folosești useAsyncData în loc de useFetch?
Corect. useFetch('/api/users') este shorthand pentru useAsyncData('users', () => $fetch('/api/users')). Folosești useAsyncData când ai nevoie de un key custom, de a combina mai multe fetch-uri (Promise.all), sau de opțiuni avansate de cache.
Întrebarea 3 din 5 Ce se întâmplă cu datele din useFetch când utilizatorul navighează de la pagina A la pagina B și înapoi la A?
Corect. Nuxt cache-uiește payload-ul din useFetch în starea aplicației. Revenirea pe pagina A (prin Back sau NuxtLink) reutilizează datele — zero request-uri suplimentare. Poți forța refetch cu refresh() sau prin opțiunea watch.
Întrebarea 4 din 5 Care este diferența dintre ref() și useState() în Nuxt?
Corect. ref() e local — fiecare componentă are propria instanță. useState('key') creează state global identificat prin key — toate componentele care îl apelează cu același key partajează același obiect reactiv. Funcționează și pentru hydration SSR: starea de pe server e transferată clientului.
Întrebarea 5 din 5 Prin ce convenie trebuie să înceapă numele unui composable pentru a fi auto-importat de Nuxt?
Corect. Nuxt auto-importă funcțiile care încep cu use din directorul composables/. Aceasta e și convenția Vue 3 pentru composable-uri — urmând-o, codul e predictibil și tooling-ul (TypeScript, ESLint) înțelege că e un composable.
