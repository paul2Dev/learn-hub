Server Routes cu Nitro

Cum construiești API-ul backend direct în proiectul Nuxt — endpoints, parametri, validare, middleware de server și utilizarea bazei de date.

Server Routes — echivalentul API Routes din Next.js

În Next.js există app/api/route.ts și Server Actions. În Nuxt, codul de server stă în server/api/ și urmează convenția de denumire a fișierelor pentru metoda HTTP:

MetodăFișier NuxtEchivalent Next.js
GETserver/api/users.get.tsapp/api/users/route.ts export GET
POSTserver/api/users.post.tsapp/api/users/route.ts export POST
DELETEserver/api/users/[id].delete.tsapp/api/users/[id]/route.ts export DELETE
Oriceserver/api/users.ts (fără sufix)Toate metodele exportate în același fișier

defineEventHandler — construirea unui endpoint

Orice fișier din server/api/ exportează un handler cu defineEventHandler: 

// server/api/products.get.ts
export default defineEventHandler(async (event) => {
  // event conține request, response, params, query etc.

  // Returnezi direct un obiect — Nuxt îl serializează ca JSON automat
  return {
    products: [
      { id: 1, name: 'Laptop', price: 3499 },
      { id: 2, name: 'Mouse', price: 89 },
    ]
  }
})

Accesul la baza de date direct din server route:

// server/api/tasks.get.ts
import { serverSupabaseUser, serverSupabaseClient } from '#supabase/server'

export default defineEventHandler(async (event) => {
  const user = await serverSupabaseUser(event)
  if (!user) {
    throw createError({ statusCode: 401, message: 'Neautorizat' })
  }

  const client = await serverSupabaseClient(event)
  const { data: tasks, error } = await client
    .from('tasks')
    .select('*')
    .order('created_at', { ascending: false })

  if (error) throw createError({ statusCode: 500, message: error.message })

  return tasks
})

createError — erori HTTP standardizate

// Nitro știe să convertească aceste erori în răspunsuri HTTP corecte
throw createError({
  statusCode: 404,
  message: 'Task-ul nu a fost găsit'
})

throw createError({
  statusCode: 422,
  data: {
    errors: {
      title: ['Titlul este obligatoriu'],
      priority: ['Valoare invalidă'],
    }
  }
})

// Clientul ($fetch, useFetch) primește eroarea în catch:
try {
  await $fetch('/api/tasks/999')
} catch (error: any) {
  console.log(error.statusCode)   // 404
  console.log(error.data.message) // 'Task-ul nu a fost găsit'
}

Server Middleware — rulat la fiecare request

// server/middleware/auth.ts — rulat ÎNAINTEA oricărui endpoint
export default defineEventHandler(async (event) => {
  const url = getRequestURL(event)

  // Permite accesul la rute publice fără autentificare
  if (url.pathname.startsWith('/api/public/')) {
    return  // continuă fără verificare
  }

  // Verifică token-ul din header
  const authHeader = getHeader(event, 'authorization')
  if (!authHeader?.startsWith('Bearer ')) {
    throw createError({ statusCode: 401, message: 'Token lipsă' })
  }

  // Verifică validitatea token-ului
  const token = authHeader.slice(7)
  const user = await verifyToken(token)
  if (!user) {
    throw createError({ statusCode: 401, message: 'Token invalid' })
  }

  // Salvează userul în contextul event-ului
  event.context.user = user
})
// Accesezi contextul în orice endpoint ulterior:
// server/api/me.get.ts
export default defineEventHandler((event) => {
  return event.context.user  // setat de middleware
})

Util Routes — non-API server routes

server/routes/ funcționează la fel ca server/api/ dar fără prefixul /api. Util pentru webhook-uri, sitemap, RSS feed, robots.txt: 

// server/routes/sitemap.xml.get.ts
export default defineEventHandler(async (event) => {
  setHeader(event, 'Content-Type', 'application/xml')

  const posts = await getPosts()
  const urls = posts.map(p => `
    <url>
      <loc>https://mysite.com/blog/${p.slug}</loc>
      <lastmod>${p.updatedAt}</lastmod>
    </url>
  `).join('')

  return `<?xml version="1.0"?><urlset xmlns="http://www.sitemaps.org/...">${urls}</urlset>`
})

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Ce acceptă fișierul server/api/products.post.ts?
Corect. Nitro aplică routing bazat pe sufix: .get.ts → GET, .post.ts → POST, .delete.ts → DELETE, .patch.ts → PATCH. Un request cu altă metodă primește automat HTTP 405 Method Not Allowed.
Întrebarea 2 din 5 Cum citești body-ul JSON dintr-un POST request în Nitro?
Corect. readBody(event) este utilitarul Nitro care citește și parsează automat body-ul request-ului. Funcționează cu JSON, form-data, text. Similar există readFormData(event) pentru multipart forms.
Întrebarea 3 din 5 Ce face throw createError({ statusCode: 401, message: '...' }) din un event handler?
Corect. createError creează o eroare specială pe care Nitro o prinde și o transformă în răspuns HTTP corespunzător. Clientul (useFetch, $fetch) primește eroarea în catch cu error.statusCode și error.data.message.
Întrebarea 4 din 5 Un fișier în server/middleware/ vs middleware/ — unde rulează fiecare?
Corect. Directoare diferite, contexte diferite. server/middleware/ este middleware Nitro — rulează în Node.js/Edge la fiecare request HTTP, inclusiv direct la API. middleware/ este middleware Vue Router — rulează în browser la navigarea SPA.
Întrebarea 5 din 5 Cum pasezi date între middleware de server și endpoint (ex: userul autentificat)?
Corect. event.context este un obiect gol pe fiecare request, destinat exact pentru a partaja date între middleware și endpoint-uri. Middleware setează event.context.user = user și endpoint-ul citește event.context.user. Este izolat per request — thread-safe.
