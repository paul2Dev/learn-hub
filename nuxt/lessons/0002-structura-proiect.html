Structura unui proiect Nuxt 3

Creare proiect, rolul fiecărui director, convenții de fișiere și cum Nuxt deduce comportamentul din structură.

Creare proiect nou

npx nuxi@latest init my-app
cd my-app
npm install
npm run dev   # pornești pe http://localhost:3000

nuxi este CLI-ul Nuxt — echivalentul lui create-next-app. La primul rulaj, instalează automat pachetele necesare. TypeScript este activ implicit — nu trebuie selectat.

Structura de directoare

my-app/
├── pages/              ← rutele aplicației (file-based routing)
│   └── index.vue       → /
├── components/         ← componente UI (auto-importate)
├── composables/        ← logică reutilizabilă (auto-importate)
├── layouts/            ← layout-uri pentru pagini
│   └── default.vue     ← layout implicit aplicat tuturor paginilor
├── middleware/         ← middleware pentru rute
├── plugins/            ← plugin-uri Vue care rulează la inițializare
├── server/             ← codul de server (Nitro)
│   ├── api/            ← endpoint-uri API
│   └── middleware/     ← middleware de server
├── public/             ← fișiere statice (nu procesate)
├── assets/             ← fișiere procesate de bundler (imagini, CSS global)
├── app.vue             ← root component (înlocuiește <NuxtPage />)
├── nuxt.config.ts      ← configurare Nuxt
└── .env                ← variabile de mediu
Față de Next.js
Next.js: totul în app/, un singur director. Nuxt: directoare specializate pe rol — pages/ pentru rute, components/ pentru componente, composables/ pentru logică, server/ pentru backend. Mai explicit, mai ușor de navigat în proiecte mari.

pages/ — file-based routing

Fiecare fișier .vue din pages/ devine o rută. Structura de directoare definește URL-ul: 

pages/
├── index.vue             → /
├── about.vue             → /about
├── blog/
│   ├── index.vue         → /blog
│   └── [slug].vue        → /blog/orice-slug    (parametru dinamic)
└── dashboard/
    ├── index.vue         → /dashboard
    └── settings.vue      → /dashboard/settings

Observă diferența față de Next.js: în Nuxt, un fișier about.vue creează ruta /about. În Next.js, ai nevoie de un folder about/ cu page.tsx în el. Nuxt e mai compact — o pagină = un fișier.

layouts/ — layouts persistente

Layout-urile definesc structura comună a paginilor. Layout-ul implicit se numește default.vue și se aplică automat tuturor paginilor: 

<!-- layouts/default.vue -->
<template>
  <div>
    <header>
      <nav>
        <NuxtLink to="/">Home</NuxtLink>
        <NuxtLink to="/about">Despre</NuxtLink>
      </nav>
    </header>

    <main>
      <slot />  <!-- conținutul paginii intră aici -->
    </main>

    <footer>© 2025</footer>
  </div>
</template>

Pentru a folosi un alt layout, specifici în pagină: 

<!-- pages/login.vue -->
<script setup lang="ts">
definePageMeta({
  layout: 'auth'   // folosește layouts/auth.vue în loc de default
})
</script>
<!-- layouts/auth.vue — layout fără navbar/footer pentru pagini de autentificare -->
<template>
  <div class="auth-wrapper">
    <slot />
  </div>
</template>

components/ — auto-import

Orice fișier din components/ este disponibil în orice template fără import manual. Structura de directoare se transformă în prefix al numelui componentei: 

components/
├── Button.vue              → <Button />
├── UserCard.vue            → <UserCard />
└── ui/
    ├── Modal.vue           → <UiModal />   (prefixul directorului)
    └── Toast.vue           → <UiToast />
<!-- pages/index.vue — fără niciun import -->
<template>
  <div>
    <Button @click="doSomething">Apasă-mă</Button>
    <UserCard :name="user.name" :email="user.email" />
    <UiModal v-if="showModal" @close="showModal = false" />
  </div>
</template>

composables/ — logică reutilizabilă auto-importată

Echivalentul hook-urilor din React. Funcții care încapsulează logică cu state reactiv: 

// composables/useCounter.ts
export function useCounter(initial = 0) {
  const count = ref(initial)
  const increment = () => count.value++
  const decrement = () => count.value--
  const reset = () => count.value = initial

  return { count, increment, decrement, reset }
}

// Folosit în orice componentă fără import:
// <script setup>
// const { count, increment } = useCounter(10)
// </script>
Convenție de denumire
Composable-urile trebuie să înceapă cu use (ex: useUser, useCart). Nuxt auto-importă numai fișierele din composables/ care urmează această convenție.

server/ — backend cu Nitro

server/
├── api/
│   ├── users.get.ts      → GET  /api/users
│   ├── users.post.ts     → POST /api/users
│   └── users/
│       └── [id].get.ts   → GET  /api/users/:id
└── middleware/
    └── auth.ts           ← middleware de server rulat la fiecare request

Sufixul HTTP (.get.ts, .post.ts, .delete.ts) definește metoda acceptată. Fișierele fără sufix acceptă orice metodă.

nuxt.config.ts — configurare centralizată

// nuxt.config.ts
export default defineNuxtConfig({
  devtools: { enabled: true },       // Nuxt DevTools (recomandat în dev)
  modules: [
    '@nuxtjs/supabase',
    '@pinia/nuxt',
    '@nuxt/ui',
  ],
  runtimeConfig: {
    // Variabile private (server only)
    supabaseServiceKey: process.env.SUPABASE_SERVICE_ROLE_KEY,
    // Variabile publice (disponibile și în browser)
    public: {
      supabaseUrl: process.env.SUPABASE_URL,
      supabaseKey: process.env.SUPABASE_ANON_KEY,
    }
  }
})
Exercițiu — Mapare structură

Unde pui fiecare fișier în structura Nuxt?

  1. Pagina de profil la URL-ul /profile
  2. Un endpoint REST care returnează lista de produse (GET /api/products)
  3. O componentă reutilizabilă ProductCard
  4. Un hook care gestionează starea coșului de cumpărături
  5. Layout-ul pentru paginile de admin (cu sidebar)

Răspuns: 1→pages/profile.vue, 2→server/api/products.get.ts, 3→components/ProductCard.vue, 4→composables/useCart.ts, 5→layouts/admin.vue.

Sursă primară

Nuxt Docs — Directory Structure
Documentația oficială acoperă fiecare director cu exemple detaliate. Citește cel puțin paginile pentru pages/, components/, composables/ și server/.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Un fișier pages/blog/[slug].vue creează ce rută?
Corect. Parantezele pătrate în numele fișierului ([slug]) marchează un parametru dinamic. Accesezi valoarea cu const route = useRoute(); route.params.slug în <script setup>.
Întrebarea 2 din 5 Ce face <slot /> în layout-ul Nuxt?
Corect. <slot /> este echivalentul Vue pentru {children} din React. În context Nuxt layouts, marchează exact unde se inserează conținutul paginii curente în interiorul layout-ului.
Întrebarea 3 din 5 O componentă components/ui/Modal.vue se folosește în template cu ce tag?
Corect. Nuxt transformă structura de directoare în prefix PascalCase. components/ui/Modal.vue → <UiModal />. components/form/Input.vue → <FormInput />. Previne coliziunile de nume în proiecte mari.
Întrebarea 4 din 5 Ce metodă HTTP acceptă fișierul server/api/users.get.ts?
Corect. Sufixul din numele fișierului (.get.ts, .post.ts, .delete.ts) restricționează metoda HTTP acceptată. Nitro returnează automat 405 dacă cineva încearcă altă metodă. Fișierele fără sufix (users.ts) acceptă toate metodele.
Întrebarea 5 din 5 Unde pui variabilele de mediu secrete (server-only) în Nuxt?
Corect. În runtimeConfig: valorile la rădăcină sunt server-only (accesate cu useRuntimeConfig() numai pe server). Valorile sub public sunt disponibile și în browser. Similar cu NEXT_PUBLIC_ prefix din Next.js.
