Deploy pe Vercel, Netlify și Cloudflare

Nitro presets — cum Nuxt se adaptează automat la orice platformă de hosting, și cum faci deploy pe cele mai populare trei platforme.

Nitro Presets — adaptare automată la platformă

Spre deosebire de Next.js care este strâns legat de Vercel, Nuxt prin Nitro generează output diferit în funcție de platforma țintă — fără a schimba nicio linie de cod:

VER

Vercel Preset

Generează funcții Edge și serverless Vercel, fișiere statice optimizate. ISR prin revalidate headers.

auto-detect
NET

Netlify Preset

Generează Netlify Edge Functions și fișiere statice. Netlify.toml configurat automat.

auto-detect
CF

Cloudflare Workers Preset

Generează un singur worker JS fără Node.js — rulează pe edge global (150+ locații). Cel mai rapid.

manual
NOD

Node.js Preset (default)

Server Node.js clasic — pentru VPS, Docker, Railway, Render. node .output/server/index.mjs

self-hosted
// nuxt.config.ts — specifică preset (opțional, auto-detectat pe Vercel/Netlify)
export default defineNuxtConfig({
  nitro: {
    preset: 'vercel'  // sau 'netlify', 'cloudflare-workers', 'node-server'
  }
})

Deploy pe Vercel

Vercel detectează Nuxt automat și setează preset-ul corect:

# 1. Conectează repo GitHub la Vercel (vercel.com → New Project)
# 2. Vercel detectează Nuxt și configurează build command automat:
#    Build Command: nuxt build
#    Output Directory: .output (auto)

# SAU cu Vercel CLI:
npm install -g vercel
vercel login
vercel deploy
# vercel.json (opțional — mai mult control)
{
  "buildCommand": "nuxt build",
  "framework": "nuxtjs",
  "regions": ["fra1", "iad1"]
}

Variabile de mediu pe Vercel:

# Dashboard → Settings → Environment Variables:
SUPABASE_URL          = https://xyz.supabase.co
SUPABASE_KEY          = eyJ... (anon key)
SUPABASE_SERVICE_ROLE_KEY = eyJ... (numai server — bifează "server" only)
Față de Next.js
Același flux ca Next.js pe Vercel. Singura diferență: Next.js folosește .next/, Nuxt folosește .output/. Build command-ul e nuxt build în loc de next build.

Deploy pe Netlify

# netlify.toml — Netlify îl citește automat
[build]
  command   = "nuxt build"
  publish   = ".output/public"  # fișiere statice

[functions]
  directory = ".output/server"  # funcții serverless

[[plugins]]
  package = "@netlify/plugin-nextjs"  # NU pentru Nuxt, folosește @netlify/plugin-nuxt

# SAU cu CLI:
npm install -g netlify-cli
netlify login
netlify deploy --prod
# Variabile de mediu Netlify:
# Dashboard → Site Settings → Environment Variables
SUPABASE_URL=https://xyz.supabase.co
SUPABASE_KEY=eyJ...
NUXT_PUBLIC_APP_NAME=MyApp

Deploy pe Cloudflare Workers

Cloudflare Workers este cel mai diferit față de celelalte platforme — codul rulează la edge global fără Node.js, deci unele module Node.js nu sunt disponibile: 

npm install wrangler --save-dev
// nuxt.config.ts
export default defineNuxtConfig({
  nitro: {
    preset: 'cloudflare-workers'
  }
})
// wrangler.toml
name = "my-nuxt-app"
main = ".output/server/index.mjs"
compatibility_date = "2024-01-01"
compatibility_flags = ["nodejs_compat"]  # activează compat Node.js

[vars]
SUPABASE_URL = "https://xyz.supabase.co"

[[kv_namespaces]]
binding = "CACHE"
id = "xxxx"
# Deploy
npx nuxi build
npx wrangler deploy
Limitări Cloudflare Workers
CPU time limitat (10ms–30ms per request pe planul gratuit). Fără filesystem. Fără binare native npm. Verifică nitro.unjs.io/deploy/providers/cloudflare pentru lista completă de compatibilitate.

Preview Deployments și Git Flow

Toate cele trei platforme suportă preview deployments pe PR-uri — identic cu Next.js pe Vercel: 

# Flux tipic:
git checkout -b feature/new-dashboard
# ... modificări ...
git push origin feature/new-dashboard
# → PR creat pe GitHub
# → Preview deployment automat pe https://my-app-git-feature-new-dashboard-username.vercel.app

# Merge pe main:
# → Production deployment automat pe https://my-app.vercel.app

Build local și testare output

# Build producție:
nuxt build

# Testează output-ul local (simulează server Nitro):
node .output/server/index.mjs

# Build static (SSG — fără server):
nuxt generate
# Output în .output/public/ — servit ca fișiere statice
# Verificare output:
.output/
├── public/          ← fișiere statice (CSS, JS, assets)
│   └── _nuxt/
├── server/
│   ├── index.mjs    ← entry point server Nitro
│   └── chunks/      ← cod server chunked
└── nitro.json       ← metadata preset
Exercițiu — Deploy pe Vercel
  1. Rulează nuxt build local și verifică că .output/ e generat corect
  2. Creează un cont Vercel și conectează repository-ul GitHub
  3. Adaugă variabilele de mediu Supabase în Vercel Dashboard
  4. Fă deploy și verifică că aplicația funcționează cu Supabase în producție
  5. Creează un PR pe un branch nou și verifică preview deployment-ul generat automat

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Ce rol are un Nitro preset în procesul de deploy?
Corect. Un preset Nitro schimbă formatul output-ului nuxt build în funcție de platforma țintă: funcții serverless Vercel, Edge Functions Netlify, un singur Worker pentru Cloudflare, sau un server Node.js standard. Codul aplicației rămâne identic.
Întrebarea 2 din 5 Ce comandă generează un site static complet fără server în Nuxt?
Corect. nuxt generate pre-randează toate paginile ca fișiere HTML statice — echivalentul next export din Next.js (deprecated), sau next build cu output: 'export'. Output-ul merge în .output/public/ și poate fi servit de orice CDN fără server.
Întrebarea 3 din 5 De ce Cloudflare Workers e mai limitat decât Vercel/Netlify pentru Nuxt?
Corect. Cloudflare Workers rulează pe V8 Isolates (nu Node.js) — APIs ca fs, path, crypto nativ Node.js, sau orice modul npm care le folosește intern, nu vor funcționa. Cloudflare oferă compatibility flags pentru a emula unele Node.js APIs, dar nu totul.
Întrebarea 4 din 5 Unde se stochează output-ul nuxt build?
Corect. Nuxt 3 scrie output-ul în .output/ — diferit de Next.js (.next/) sau Vue CLI (dist/). Structure: .output/public/ conține fișierele statice servite direct, .output/server/index.mjs este entry point-ul serverului Nitro.
Întrebarea 5 din 5 Cum testezi local output-ul de producție Nuxt înainte de deploy?
Corect. După nuxt build, rulezi direct output-ul Nitro: node .output/server/index.mjs. Asta pornește serverul de producție local pe portul 3000 (implicit). Alternativ, nuxi preview face build + start automat, dar e mai lent pentru iterații rapide.
