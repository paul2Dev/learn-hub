Module Nuxt — ce sunt și de ce există

Un modul Nuxt este un pachet npm care se injectează în build-ul Nuxt la runtime — adaugă auto-imports, modifică configurația webpack/vite, adaugă server routes, sau configurează runtimeConfig. Echivalentul Next.js ar fi o combinație de plugin Webpack + middleware + config manual.

Adăugare module în nuxt.config.ts

// nuxt.config.ts
export default defineNuxtConfig({
  modules: [
    '@nuxtjs/supabase',
    '@pinia/nuxt',
    '@nuxtjs/tailwindcss',
    '@nuxt/image',
    // Forma alternativă cu opțiuni inline:
    ['@nuxtjs/i18n', { defaultLocale: 'ro' }],
  ],

  // Opțiuni per-modul în secțiunile lor dedicate:
  supabase: {
    redirect: true,
    redirectOptions: { login: '/login', callback: '/confirm', exclude: ['/'] }
  },
  image: {
    quality: 80,
    format: ['webp'],
  }
})

Pinia — state management

Pinia este librăria oficială de state management Vue 3 — echivalentul Redux/Zustand din ecosistemul Next.js/React. Cu modulul @pinia/nuxt, store-urile sunt auto-importate: 

npm install pinia @pinia/nuxt
// stores/tasks.ts
import { defineStore } from 'pinia'

export const useTasksStore = defineStore('tasks', () => {
  // State
  const tasks = ref<Task[]>([])
  const loading = ref(false)
  const error   = ref<string | null>(null)

  // Getters (computed)
  const completedTasks   = computed(() => tasks.value.filter(t => t.done))
  const incompleteTasks  = computed(() => tasks.value.filter(t => !t.done))
  const completionRate   = computed(() =>
    tasks.value.length ? completedTasks.value.length / tasks.value.length : 0
  )

  // Actions
  async function fetchTasks() {
    loading.value = true
    const { data } = await $fetch('/api/tasks')
    tasks.value = data ?? []
    loading.value = false
  }

  async function addTask(title: string) {
    const { data } = await $fetch('/api/tasks', {
      method: 'POST',
      body: { title }
    })
    tasks.value.unshift(data)
  }

  async function toggleTask(id: number) {
    const task = tasks.value.find(t => t.id === id)
    if (!task) return
    await $fetch(`/api/tasks/${id}`, {
      method: 'PATCH',
      body: { done: !task.done }
    })
    task.done = !task.done
  }

  return { tasks, loading, error, completedTasks, incompleteTasks, completionRate, fetchTasks, addTask, toggleTask }
})
<!-- Utilizare în componente — auto-importat cu @pinia/nuxt -->
<script setup lang="ts">
const store = useTasksStore()

onMounted(() => store.fetchTasks())
</script>

<template>
  <div>
    <p>{{ store.incompleteTasks.length }} task-uri rămase</p>
    <ul>
      <li v-for="task in store.tasks" :key="task.id">
        <input type="checkbox" :checked="task.done" @change="store.toggleTask(task.id)" />
        {{ task.title }}
      </li>
    </ul>
  </div>
</template>

nuxt.config.ts — configurare completă

export default defineNuxtConfig({
  // Comportament la development
  devtools: { enabled: true },     // DevTools browser

  // TypeScript strict
  typescript: { strict: true },

  // CSS global
  css: ['~/assets/styles/main.css'],

  // Variabile disponibile pe client și server
  runtimeConfig: {
    // Privat — NUMAI server:
    supabaseServiceRoleKey: process.env.SUPABASE_SERVICE_ROLE_KEY,
    stripeSecretKey: process.env.STRIPE_SECRET_KEY,

    // Public — expus și pe client:
    public: {
      appName: 'MyApp',
      apiBase: process.env.API_BASE ?? '/api',
    }
  },

  // Head HTML global (meta, title default, fonts)
  app: {
    head: {
      title: 'MyApp',
      meta: [
        { name: 'description', content: 'Aplicația mea Nuxt 3' },
        { charset: 'utf-8' },
        { name: 'viewport', content: 'width=device-width, initial-scale=1' },
      ],
      link: [
        { rel: 'icon', href: '/favicon.ico' },
        { rel: 'stylesheet', href: 'https://fonts.googleapis.com/css2?family=Inter:wght@400;600&display=swap' }
      ]
    }
  },

  // Optimizări build
  experimental: {
    payloadExtraction: false,  // dezactivează payload.json pentru SSG static
  },

  // Nitro server config
  nitro: {
    preset: 'vercel',          // override preset la deploy
    storage: {                 // storage pentru Nitro KV
      redis: {
        driver: 'redis',
        url: process.env.REDIS_URL,
      }
    }
  }
})

runtimeConfig — variabile de mediu în Nuxt

// Cum accesezi runtimeConfig:

// În server routes:
const config = useRuntimeConfig()
const key = config.supabaseServiceRoleKey  // privat, numai server

// În componente Vue (public):
const config = useRuntimeConfig()
const apiBase = config.public.apiBase  // disponibil pe client și server
# .env — variabilele sunt citite automat
NUXT_SUPABASE_SERVICE_ROLE_KEY=eyJ...  # overrideaza runtimeConfig.supabaseServiceRoleKey
NUXT_PUBLIC_API_BASE=https://api.myapp.com  # overrideaza runtimeConfig.public.apiBase

# Observă convenția de naming:
# NUXT_ + camelCase → runtimeConfig (server private)
# NUXT_PUBLIC_ + camelCase → runtimeConfig.public (client+server)

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Unde adaugi un modul Nuxt instalat prin npm?
Corect. Modulele Nuxt se înregistrează în nuxt.config.ts în array-ul modules: []. Nuxt le procesează la build time — injectează composable-uri, modifică configurația Vite, adaugă server routes etc. Pluginurile din plugins/ sunt altceva — rulează runtime în browser sau pe server.
Întrebarea 2 din 5 Care dintre aceste variabile din runtimeConfig este disponibilă și pe client?
Corect. runtimeConfig.public.* este serializat și trimis clientului (similar NEXT_PUBLIC_ din Next.js). runtimeConfig.* (la top-level, în afara public) rămâne exclusiv pe server — niciodată nu ajunge în browser.
Întrebarea 3 din 5 Ce face convenția de naming NUXT_PUBLIC_API_BASE în .env?
Corect. Nuxt mapează automat variabile de mediu la runtimeConfig: NUXT_PUBLIC_API_BASE → runtimeConfig.public.apiBase. Snake_case devine camelCase. La fel NUXT_DB_PASSWORD → runtimeConfig.dbPassword. Asta permite override ușor per mediu (dev/staging/prod) fără a edita config.
Întrebarea 4 din 5 Ce diferență există între un store Pinia și useState() din Nuxt?
Corect. useState() din Nuxt e simplu — un ref global cu cheie, accesibil din orice componentă, sincronizat server-client. Pinia adaugă actions structurate, computed getters, suport DevTools, subscriptions, plugins și o arhitectură mai scalabilă. Pentru un ref simplu shared → useState; pentru logică de business complexă → Pinia.
Întrebarea 5 din 5 Cum configurezi meta-tagurile HTML (title, description) global în Nuxt?
Corect. app.head în nuxt.config.ts configurează head-ul global. Per pagină/componentă, folosești useHead({ title: '...', meta: [...] }) sau useSeoMeta() care oferă TypeScript types pentru toate meta-tagurile SEO. Nuxt 3 folosește @unhead/vue internaly.
