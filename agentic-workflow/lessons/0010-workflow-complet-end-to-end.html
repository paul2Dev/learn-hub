Workflow complet — scenariu real

Iată cum arată un feature de la idee la producție cu toate piesele la loc:

# 1. Creezi un branch nou pentru feature
git checkout -b feature/user-notifications

# 2. Ceri lui Claude Code să implementeze feature-ul
"Implementează un sistem de notificări pentru utilizatori noi"

# 3. Claude Code scrie codul și rulează automat /verify
# — dacă găsește probleme, le repară înainte să raporteze

# 4. Rulezi /simplify pentru calitate
/simplify

# 5. Rulezi /code-review înainte de push
/code-review
# → Găsit: lipsă validare email. Aplici fix-ul.

# 6. Rulezi /security-review dacă codul atinge autentificarea
/security-review
# → OK, nicio vulnerabilitate detectată.

# 7. Commit și push
git add .
git commit -m "feat: add user notification system"
git push origin feature/user-notifications

# 8. Claude Code creează PR-ul cu gh CLI
# → PR creat: github.com/paul2Dev/project/pull/42

# 9. GitHub Actions pornește automat CI
# → Build ✓ | Lint ✓ | Tests ✓

# 10. Greptile face review automat (câteva minute)
# → Finding: NotificationService nu gestionează email invalid.

# 11. Rulezi /greploop dacă vrei fix automat
/greploop
# → Iterație 1: aplică fix, push
# → Iterație 2: Greptile re-revizuiește
# → Scor 5/5 — zero comentarii. Greploop se oprește.

# 12. TU deschizi PR-ul în GitHub
gh pr view --web
# → Citești: review Greptile OK, CI passes, diff curat

# 13. TU dai click pe Merge
# → Vercel detectează push pe main → deploy automat → producție

Când să sări peste pași

Nu fiecare schimbare necesită tot workflow-ul. Iată regula simplă:

Tipul schimbării Pași obligatorii Pași opționali
Fix minor de typo sau stilizare Branch + PR + CI /verify, /code-review, Greptile
Bugfix simplu Branch + /verify + /code-review + PR + CI /security-review, /greploop
Feature nou Toți pașii Nimic — rulezi tot
Schimbare autentificare / date sensibile Toți pașii + /security-review obligatoriu Nimic — rulezi tot + security

Checklist de merge — ce verifici înainte de click

Înainte de fiecare merge, verifici rapid:

Checklist pre-merge
  • CI trece — build, lint, teste ✓
  • Review Greptile/CodeRabbit — citit, finding-urile adresate
  • Diff citit — știi ce intră în main
  • Test plan din PR bifat — funcționalitatea verificată manual sau automat

Troubleshooting — situații comune

🔴 CI eșuează la build situație comună

Rulează gh pr checks pentru a vedea log-ul complet. Identifică eroarea, repară local, fă push nou. CI va re-rula automat.

🔴 Greploop nu ajunge la 5/5 după 5 iterații situație avansată

Citește finding-urile Greptile rămase nerezolvate. Dacă sunt false positives, marchează-le ca atare în Greptile. Dacă sunt probleme reale complexe, rezolvă-le manual — poate necesită decizie de arhitectură.

🔴 Branch protection blochează push direct pe master comportament așteptat

Aceasta este funcționarea corectă — nu o eroare. Creează un branch și un PR. Dacă e urgent, poți dezactiva temporar branch protection din GitHub Settings, dar revino la ea imediat după.

Quiz Final — Workflow Complet

Întrebarea 1 din 5 Care este ordinea corectă a pașilor locali ÎNAINTE de push pe branch?
Corect. /verify mai întâi (confirmă că funcționează), /simplify după (curăță codul verificat), /code-review la final (review pe cod curat). Abia după toate trei — commit și push.
Întrebarea 2 din 5 Care este singurul pas din tot workflow-ul care NECESITĂ o decizie umană?
Corect. Toate celelalte pot fi automatizate (Claude Code creează branch-uri, scrie PR-uri, GitHub Actions rulează CI, Greptile/Greploop face review). Merge-ul rămâne la om — ultimul filtru înainte de producție.
Întrebarea 3 din 5 Ce tool nu trebuie instalat dacă Greptile nu este pe repository?
Corect. Greploop nu face nimic fără Greptile — el orchestrează: declanșează review Greptile, citește feedback, aplică fix, repetă. Fără Greptile instalat, nu are cu ce interacționa.
Întrebarea 4 din 5 Care este checklist-ul minim de verificat înainte de a da click pe Merge?
Corect. Cele 4 puncte de verificare: CI trece, review-urile AI adresate, diff-ul citit, test plan verificat. Nu toți pașii trebuie să fie perfect — trebuie să știi ce intră în producție.
Întrebarea 5 din 5 Pentru un fix minor de typo, ce pași din workflow sunt opționali conform tabelului din această lecție?
Corect. Chiar și pentru typos: branch + PR + CI rămân obligatorii — branch protection nu face excepții. /verify, /code-review și review-ul Greptile sunt opționale pentru schimbări triviale.
