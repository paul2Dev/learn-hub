Branch Protection + Git Flow

Fără branch protection, orice automatizare devine un risc. Aceasta este fundația pe care se construiește tot restul.

De ce contează acum
Dacă nu activezi branch protection înainte de orice altceva, un agent AI poate merge direct în master și Vercel deployează în producție fără niciun review. Configureaz-o înainte de a trece la lecțiile 3–10.

Ce este branch protection

Branch protection este o regulă pe GitHub care blochează push-urile directe pe un branch protejat (de obicei master sau main) și impune ca orice schimbare să treacă printr-un Pull Request.

Odată activată, chiar dacă tu sau Claude Code rulați git push origin master, GitHub va refuza push-ul și va returna o eroare. Singura cale de a adăuga cod în main este prin PR aprobat.

Activarea branch protection pe GitHub

1

Mergi la Settings → Branches în repository-ul tău GitHub.

↓
2

Click pe „Add branch protection rule" și scrie master (sau main) în câmpul de pattern.

↓
3

Bifează „Require a pull request before merging". Dezactivează „Require approvals" dacă lucrezi singur.

↓
4

Opțional dar recomandat: bifează „Require status checks to pass before merging" — vei configura check-urile la Lecția 9.

↓
5

Click „Create". Branch protection este activ imediat.

Git flow cu branch protection activ

Odată cu branch protection activat, workflow-ul tău de zi cu zi devine:

# 1. Creează un branch nou pentru fiecare feature sau fix
git checkout -b feature/descriere-scurta

# 2. Lucrează, fă commits pe branch
git add .
git commit -m "feat: descriere clară a schimbării"

# 3. Push pe branch (NU pe master)
git push origin feature/descriere-scurta

# 4. Deschide PR pe GitHub (sau cu gh CLI — Lecția 5)
gh pr create --title "feat: descriere" --body "..."

# 5. Merge în GitHub după review
# (click pe Merge în UI sau: gh pr merge --squash)
Convenție de denumire branch-uri
Folosește prefixe clare: feature/ pentru funcționalități noi, fix/ pentru buguri, refactor/ pentru refactorizări, chore/ pentru actualizări minore. Această convenție ajută GitHub Actions să aplice reguli diferite pe tipuri diferite de branch-uri.

De ce Claude Code are nevoie de această disciplină

Claude Code lucrează mai bine cu branch protection din mai multe motive:

  • Context clar — agentul știe că nu poate merge direct în main, deci va crea branch-uri și PR-uri
  • Feedback loop — PR-ul devine punctul de agregare al tuturor review-urilor (AI + CI + uman)
  • Rollback simplu — dacă ceva merge greșit, revii ușor la commit-ul anterior fără să afectezi main
  • Audit trail — fiecare schimbare are un PR cu descriere, reviewer, și toate check-urile documentate

Configurare .gitconfig pentru identitate corectă

Asigură-te că Git folosește identitatea ta în toate commit-urile:

git config user.name "Paul2Dev"
git config user.email "paul.maxineanu@gmail.com"

Sau global (pentru toate repo-urile):

git config --global user.name "Paul2Dev"
git config --global user.email "paul.maxineanu@gmail.com"

Verificarea branch protection

Testează că branch protection funcționează corect:

# Încearcă un push direct pe master — ar trebui să eșueze
git checkout master
echo "test" >> test.txt
git add test.txt
git commit -m "test: direct push"
git push origin master
# ❌ Expected: remote: error: GH006: Protected branch update failed
Exercițiu — Activează branch protection acum
  1. Mergi la Settings → Branches în repository-ul tău GitHub.
  2. Adaugă o regulă pentru branch-ul master (sau main).
  3. Bifează „Require a pull request before merging".
  4. Testează că push-ul direct pe master este blocat (comanda de mai sus).
  5. Creează un branch test/branch-protection, adaugă un fișier, deschide un PR și merge-l.
Sursă primară

Documentația oficială GitHub: About protected branches — descrie toate opțiunile disponibile și cum interacționează cu GitHub Actions.

Quiz — Verifică înțelegerea

Întrebarea 1 din 5 Ce se întâmplă dacă rulezi `git push origin master` cu branch protection activ?
Corect. Branch protection blochează push-ul la nivel de server — GitHub returnează o eroare și nu acceptă schimbarea. Singura cale este prin PR.
Întrebarea 2 din 5 Care este prefixul corect pentru un branch de bugfix conform convenției din această lecție?
Corect. Convenția recomandată: fix/ pentru buguri, feature/ pentru funcționalități, refactor/ pentru refactorizări, chore/ pentru actualizări minore.
Întrebarea 3 din 5 De ce branch protection ajută Claude Code să lucreze mai disciplinat?
Exact. Constrângerile tehnice forțează un workflow mai bun. Un agent care știe că nu poate merge direct va structura munca în branch-uri și PR-uri — exact ce vrei.
Întrebarea 4 din 5 Ce opțiune din branch protection vei activa la Lecția 9 pentru a face check-urile CI obligatorii?
Corect. Această opțiune face ca merge-ul să fie blocat dacă GitHub Actions nu a trecut build-ul și lint-ul. O vei configura la Lecția 9, după ce workflow-ul de CI este creat.
Întrebarea 5 din 5 Care este primul pas din git flow după ce activezi branch protection?
Corect. Orice nouă schimbare începe cu un branch nou. Niciodată direct pe master. Branch-ul este spațiul tău de lucru izolat — fără risc pentru producție.
