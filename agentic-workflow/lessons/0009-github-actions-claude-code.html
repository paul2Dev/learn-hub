GitHub Actions + Claude Code

Automatizează build, lint și review cu @claude direct din PR-uri și issue-uri.

Ce este GitHub Actions în context agentic

GitHub Actions este stratul de CI al workflow-ului tău. Rulează automat la fiecare PR deschis și verifică că:

  • Build-ul nu este rupt
  • Lint-ul trece fără erori
  • Testele (dacă există) trec

Separat, Claude Code are o integrare nativă cu GitHub Actions care permite invocarea agentului direct din comentarii cu @claude.

Workflow minimal de CI — structura fișierului

Fișierele GitHub Actions stau în .github/workflows/ și sunt YAML:

# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [ master, main ]
  push:
    branches: [ master, main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run lint
        run: npm run lint

      - name: Run build
        run: npm run build

      - name: Run tests
        run: npm test
        if: hashFiles('**/*.test.*') != ''  # Skip dacă nu există teste
Adaptare pentru PHP/Laravel
Dacă proiectul tău este PHP/Laravel, înlocuiește Setup Node.js cu shivammathur/setup-php@v2 și comenzile cu composer install, php artisan test, etc. Claude Code poate genera workflow-ul adaptat dacă îi spui tipul proiectului.

Integrarea Claude Code în GitHub Actions

Claude Code are un action oficial care permite invocarea agentului din PR-uri și issue-uri cu @claude:

# .github/workflows/claude.yml
name: Claude Code AI Assistant

on:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]

jobs:
  claude:
    if: contains(github.event.comment.body, '@claude')
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@beta
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          claude_args: "--max-turns 10"

Odată configurat, în orice comentariu de PR sau issue poți scrie:

@claude Fix the failing test in UserService.php
@claude Explain what this function does
@claude Implement the feature described in this issue

Setup: ANTHROPIC_API_KEY ca GitHub Secret

1

Du-te la console.anthropic.com → API Keys → generează o cheie nouă.

↓
2

În repository GitHub: Settings → Secrets and variables → Actions → New repository secret

↓
3

Name: ANTHROPIC_API_KEY, Value: cheia generată. Secret nu este vizibil după salvare.

↓
4

Workflow-ul claude.yml referențiază secret-ul cu ${{ secrets.ANTHROPIC_API_KEY }}.

HTTP Hooks — automatizare la nivel local

Pe lângă GitHub Actions, Claude Code suportă HTTP Hooks care rulează la evenimente locale (nu în cloud). Cele mai utile:

# În .claude/settings.json (sau CLAUDE.md)
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{
          "type": "command",
          "command": "npm run lint --fix"
        }]
      }
    ],
    "Stop": [
      {
        "hooks": [{
          "type": "command",
          "command": "echo 'Session ended — verify changes before push'"
        }]
      }
    ]
  }
}
PostToolUse vs Stop
PostToolUse rulează după fiecare tool call (ex: după fiecare Edit sau Write). Stop rulează când Claude Code termină sesiunea. Combinate, pot rula lint după fiecare modificare și pot posta un raport la final.

Activarea „Require status checks" după setup

Odată ce CI-ul funcționează, revino la branch protection (Lecția 2) și activează opțiunea de status checks:

  1. Settings → Branches → Edit regula pentru master
  2. Bifează „Require status checks to pass before merging"
  3. Adaugă job-urile de CI: build (și lint, test dacă există)

Acum merge-ul este blocat dacă build-ul eșuează — altă garanție automată.

Exercițiu — Adaugă CI la proiectul tău
  1. Creează .github/workflows/ci.yml adaptat la tipul proiectului tău (Node, PHP, etc.).
  2. Roagă Claude Code să genereze workflow-ul: "Generează un GitHub Actions workflow pentru proiectul meu [tip] cu build, lint și teste".
  3. Fă push pe un branch și deschide un PR — observă că workflow-ul rulează automat.
  4. Adaugă și .github/workflows/claude.yml dacă vrei să poți invoca @claude din PR-uri.
  5. Revino la branch protection și activează „Require status checks".

Quiz — Verifică înțelegerea

Întrebarea 1 din 5 În ce director din repository stau fișierele de configurare GitHub Actions?
Corect. Toate fișierele YAML de GitHub Actions stau în .github/workflows/. GitHub detectează automat orice fișier .yml sau .yaml din acest director și îl tratează ca un workflow.
Întrebarea 2 din 5 Cum invoci Claude Code din un comentariu de PR după ce ai configurat claude.yml?
Corect. Sintaxa este @claude instrucțiune în orice comentariu de PR sau issue. GitHub Actions detectează mention-ul și declanșează workflow-ul cu instrucțiunea ca prompt pentru agent.
Întrebarea 3 din 5 Unde se salvează ANTHROPIC_API_KEY pentru a fi folosit de GitHub Actions?
Corect. GitHub Secrets este locul securizat pentru credențiale sensibile. Secretele nu sunt vizibile în log-urile de Actions și nu apar în diff-ul repository-ului — spre deosebire de un fișier .env commis din greșeală.
Întrebarea 4 din 5 Ce face hook-ul PostToolUse configurat în exemplul din această lecție?
Corect. PostToolUse cu matcher „Write|Edit" rulează comanda configurată după fiecare tool call de tip scriere sau editare. Rezultat: lint se rulează automat la fiecare fișier modificat de agent.
Întrebarea 5 din 5 De ce activezi „Require status checks" în branch protection DUPĂ ce CI-ul funcționează?
Corect. GitHub poate lista job-urile de CI în opțiunile de status check doar după ce au rulat cel puțin o dată. Activezi CI-ul mai întâi, faci un PR de test, și abia apoi configurezi status checks în branch protection.
