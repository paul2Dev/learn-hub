Schema Postgres și Row Level Security

Cele trei tabele principale, trigger-ul care creează automat profilul la primul login, și de ce RLS-ul contează chiar dacă serverul deja verifică autentificarea.

Migrațiile ca istoric versionat

supabase/migrations/ conține schema completă, în ordine cronologică — fiecare fișier e un pas incremental, niciodată modificat retroactiv: 

supabase/migrations/
├── 001_initial_schema.sql          ← profiles, watchlist, watched + RLS
├── 002_semantic_search.sql         ← pgvector, media_embeddings, search_usage
├── 003_fix_media_embeddings_rls.sql
├── 004_watchlist_watched_media_type.sql
├── 005_drop_legacy_search_movies_rpc.sql
├── 006_document_profiles_insert_policy.sql
└── 007_document_ensure_rls_event_trigger.sql
Lecție reală din migrația 004
Schema inițială (001) definea unique (user_id, movie_id) pe watchlist/watched — dar coloana media_type a fost adăugată direct pe remote, fără migrație, mai târziu. Rezultat: un film și un serial cu același TMDB ID nu puteau coexista în watchlist-ul aceluiași user. Migrația 004 documentează coloana și repară constraint-ul la unique (user_id, movie_id, media_type). E un exemplu real de „schema drift" — diferență între ce spune codul și ce există de fapt în DB — și cum se repară cu o migrație explicită.

Tabela profiles — extensie peste auth.users

supabase/migrations/001_initial_schema.sql
create table public.profiles (
  id                uuid primary key references auth.users (id) on delete cascade,
  preferred_genres  integer[] default '{}',   -- TMDB genre IDs
  onboarding_done   boolean default false,
  created_at        timestamptz default now(),
  updated_at        timestamptz default now()
);

-- Creat automat la primul login
create or replace function public.handle_new_user()
returns trigger
language plpgsql security definer set search_path = public
as $$
begin
  insert into public.profiles (id) values (new.id);
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();

Supabase gestionează auth.users automat (email, provider OAuth, etc). UpNext nu duplică acele date — profiles ține doar ce e specific aplicației (preferred_genres, onboarding_done), legat prin același id. Trigger-ul on_auth_user_created garantează că fiecare user nou primește instant o linie în profiles, fără ca serverul aplicației să trebuiască să o creeze manual.

Tabelele watchlist și watched

create table public.watchlist (
  id          uuid primary key default gen_random_uuid(),
  user_id     uuid not null references auth.users (id) on delete cascade,
  movie_id    integer not null,           -- TMDB movie/tv ID
  media_type  text default 'movie',       -- 'movie' sau 'tv'
  tmdb_data   jsonb not null,             -- răspunsul TMDB cache-uit
  added_at    timestamptz default now(),
  unique (user_id, movie_id, media_type)
);

create table public.watched (
  id          uuid primary key default gen_random_uuid(),
  user_id     uuid not null references auth.users (id) on delete cascade,
  movie_id    integer not null,
  media_type  text default 'movie',
  tmdb_data   jsonb not null,
  rating      smallint check (rating >= 1 and rating <= 5),
  watched_at  timestamptz default now(),
  unique (user_id, movie_id, media_type)
);
De ce tmdb_data jsonb?
Watchlist-ul salvează o copie a datelor TMDB (titlu, poster, dată) în loc să facă un nou fetch la fiecare afișare. Compromis conștient: datele pot deveni ușor stale (ex: rating-ul TMDB se schimbă), dar afișarea watchlist-ului nu depinde de disponibilitatea TMDB și e mult mai rapidă.

Row Level Security — a doua linie de apărare

alter table public.watchlist enable row level security;

create policy "Users can view own watchlist"
  on public.watchlist for select
  using (auth.uid() = user_id);

create policy "Users can insert to own watchlist"
  on public.watchlist for insert
  with check (auth.uid() = user_id);

create policy "Users can delete from own watchlist"
  on public.watchlist for delete
  using (auth.uid() = user_id);

Ai văzut în Lecția 5 că watchlist.post.ts verifică deja serverSupabaseUser(event) înainte de orice insert. De ce mai e nevoie și de RLS?

Apărare pe două niveluri
Verificarea din server ("user există?") previne request-uri complet neautentificate. RLS-ul din Postgres previne un scenariu diferit: un bug în cod care ar încerca să insereze/citească date pentru alt user_id decât cel autentificat. Chiar dacă handler-ul ar avea o eroare (ex: uită să filtreze după user.id), Postgres respinge oricum orice query care ar încălca auth.uid() = user_id. E principiul „defense in depth" — niciodată nu te bazezi pe un singur strat de securitate.

Extensie: pgvector pentru căutare semantică

Migrația 002 adaugă o tabelă complet diferită de structură, pentru feature-ul opțional de căutare semantică (detalii complete în Lecția 11): 

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE public.media_embeddings (
  movie_id   integer NOT NULL,
  title      text    NOT NULL,
  overview   text,
  embedding  vector(1536) NOT NULL,        -- vector OpenAI text-embedding-3-small
  media_type text    NOT NULL DEFAULT 'movie',
  PRIMARY KEY (movie_id, media_type)
);

CREATE INDEX ON public.media_embeddings USING hnsw (embedding vector_cosine_ops);

vector(1536) vine din extensia pgvector — Postgres tratează un vector numeric ca tip de date de prim rang, cu index specializat (HNSW) pentru căutare rapidă de similaritate.

Exercițiu — Scrie o policy

Adaugi un tabel nou custom_lists (liste de filme create de utilizator), cu coloane id, user_id, name. Ce policy RLS lipsește mai jos? 

alter table public.custom_lists enable row level security;

create policy "Users can view own lists"
  on public.custom_lists for select
  using (auth.uid() = user_id);

-- ??? utilizatorii nu pot crea liste noi fără asta

Răspuns: create policy "Users can insert own lists" on public.custom_lists for insert with check (auth.uid() = user_id); — fără policy de insert, RLS blochează implicit orice inserare, chiar dacă utilizatorul e autentificat.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 De ce profiles nu duplică email-ul și numele din auth.users?
Corect. profiles.id referențiază auth.users.id — orice cod care are nevoie de email/nume poate lua acele date din sesiunea de auth, fără duplicare.
Întrebarea 2 din 5 Ce s-ar întâmpla dacă ai uita policy-ul de insert pentru un tabel nou cu RLS activ?
Corect. RLS respinge implicit orice operație pentru care nu există o policy explicită — „deny by default". Fără policy de insert, inserările eșuează pentru toți, indiferent ce spune codul serverului.
Întrebarea 3 din 5 De ce mai există RLS dacă watchlist.post.ts deja verifică serverSupabaseUser(event)?
Corect. RLS e a doua linie de apărare — chiar dacă serverul ar avea un bug (ex: ar omite un filtru pe user_id), Postgres respinge oricum orice query care încalcă policy-ul.
Întrebarea 4 din 5 Ce problemă reală a rezolvat migrația 004_watchlist_watched_media_type.sql?
Corect. Constraint-ul original unique(user_id, movie_id) bloca un film și un serial cu același TMDB ID să coexiste în watchlist-ul aceluiași user — 004 îl repară la unique(user_id, movie_id, media_type).
Întrebarea 5 din 5 Ce face coloana embedding vector(1536) din media_embeddings, la nivel conceptual?
Corect. Fiecare film/serial are un vector numeric de 1536 dimensiuni, generat de OpenAI din titlu+overview — folosit pentru a găsi filme „similare ca sens", nu doar similare ca text exact. Detalii în Lecția 11.
