Codul relevant, în ordinea fluxului

app/composables/useWatchlist.ts — funcția toggle()
async function toggle(movieId: number, mediaType: 'movie' | 'tv', tmdbData: object) {
  const key = watchlistKey(movieId, mediaType)
  if (ids.value.has(key)) {
    await $fetch('/api/user/watchlist', { method: 'DELETE', body: { movie_id: movieId, media_type: mediaType } })
    const next = new Set(ids.value); next.delete(key); ids.value = next
  } else {
    await $fetch('/api/user/watchlist', { method: 'POST', body: { movie_id: movieId, media_type: mediaType, tmdb_data: tmdbData } })
    const next = new Set(ids.value); next.add(key); ids.value = next
  }
}
Observație importantă
Codul nu reface un GET după POST ca să confirme starea — actualizează ids.value local, optimist, imediat ce request-ul reușește. Dacă request-ul aruncă (network error, 401, etc), await propagă excepția, iar ids.value nu se modifică — starea rămâne consistentă cu ce s-a întâmplat cu adevărat pe server.
server/api/user/watchlist.post.ts — validare + insert
const user = await serverSupabaseUser(event)
if (!user) throw createError({ statusCode: 401, message: 'Unauthorized' })

const parsed = schema.safeParse(await readBody(event))
if (!parsed.success) throw createError({ statusCode: 400, message: parsed.error.issues[0]?.message })

const client = await serverSupabaseClient(event)
const { data, error } = await client
  .from('watchlist')
  .insert({ user_id: user.id, movie_id: parsed.data.movie_id, media_type: parsed.data.media_type, tmdb_data: parsed.data.tmdb_data })
  .select()
  .single()

Trei bariere înainte ca un rând să ajungă în baza de date: autentificare (401), validare Zod (400), și — invizibil în acest cod, dar prezent — RLS-ul din Postgres, care ar respinge insert-ul chiar dacă user.id ar fi greșit calculat undeva mai sus.

Ce se schimbă dacă butonul e apăsat din navbar în loc de MediaCard?

Nimic din pașii 4-11. Diferența e doar în pasul 1-3: app/app.vue apelează useWatchlist() pentru badge-ul cu numărul total, exact ca MediaCard.vue. Amândouă vorbesc cu același useState('watchlist-ids', ...) — de asta badge-ul din navbar se actualizează instant când adaugi un film dintr-un card, fără niciun cod suplimentar de sincronizare.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Ce se întâmplă cu ids.value dacă $fetch aruncă o eroare de rețea în timpul toggle()?
Corect. Codul de după await $fetch(...) nu rulează dacă $fetch aruncă — deci ids.value rămâne cum era, evitând un state local dezincronizat de server.
Întrebarea 2 din 5 Câte bariere de validare/autorizare trece un request POST înainte să scrie în Postgres?
Corect. 401 (fără user), 400 (body invalid), și — chiar dacă primele două trec — RLS-ul respinge orice insert care nu respectă auth.uid() = user_id.
Întrebarea 3 din 5 De ce se actualizează badge-ul din navbar automat când adaugi un film din MediaCard, fără cod de sincronizare explicit?
Corect. Fără useState partajat, ar fi nevoie de props/evenimente între componente fără nicio relație părinte-copil directă — mult mai complicat decât un singur state global.
Întrebarea 4 din 5 Ce fișier rulează primul dintre toate, la orice request către server?
Corect. Middleware-ul de server rulează pe orice request, înainte de orice handler server/api/ specific — vezi Lecția 5.
Întrebarea 5 din 5 Care e diferența structurală între fluxul de watchlist și fluxul de rating (5 stele)?
Corect. Structura fluxului e identică pas cu pas — diferă doar tabela Postgres țintă și un câmp suplimentar în payload. Odată înțeles un flux CRUD din acest cod, celelalte sunt variații previzibile.
