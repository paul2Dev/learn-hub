Pasul 1-2: Login

app/pages/login.vue
<script setup lang="ts">
definePageMeta({ layout: false })

const supabase = useSupabaseClient()
const loading = ref(false)

async function signInWithGoogle() {
  loading.value = true
  await supabase.auth.signInWithOAuth({
    provider: 'google',
    options: {
      redirectTo: `${window.location.origin}/confirm`
    }
  })
  loading.value = false
}
</script>

useSupabaseClient() e injectat automat de modulul @nuxtjs/supabase — niciun setup manual de client SDK. redirectTo spune Google-ului unde să trimită utilizatorul înapoi după ce confirmă contul.

Pasul 3-4: Callback

app/pages/confirm.vue
<script setup lang="ts">
definePageMeta({ layout: false })

const supabase = useSupabaseClient()
const router = useRouter()

onMounted(async () => {
  const { data: { session } } = await supabase.auth.getSession()
  if (session) {
    await router.replace('/')
  } else {
    await router.replace('/login')
  }
})
</script>

Pagina nu face altceva decât să afișeze „Signing in..." și să verifice dacă sesiunea a fost procesată cu succes. Modulul Supabase se ocupă de parsarea token-ului din URL automat, în fundal — codul de aici doar așteaptă și redirecționează.

Pasul 5: Protejarea rutelor cu middleware

Există două tipuri de middleware de rută în această aplicație:

FișierTipCând rulează
auth.tsPer-pagină (opt-in)Doar pe paginile care declară middleware: 'auth'
onboarding.global.tsGlobalPe orice navigare, în orice pagină
app/middleware/auth.ts
export default defineNuxtRouteMiddleware(() => {
  const user = useSupabaseUser()
  if (!user.value) {
    return navigateTo('/login')
  }
})

Simplu: dacă nu există user, redirect la login. Aplicat pe watchlist.vue, profile.vue și onboarding.vue via definePageMeta({ middleware: 'auth' }).

app/middleware/onboarding.global.ts
const SKIP_PATHS = ['/login', '/confirm', '/onboarding']

export default defineNuxtRouteMiddleware(async (to) => {
  if (SKIP_PATHS.includes(to.path)) return

  const user = useSupabaseUser()
  if (!user.value) return

  const profileCache = useState<{ onboarding_done: boolean, preferred_genres?: number[] } | null>(
    'profile-cache', () => null
  )

  if (!profileCache.value) {
    try {
      const supabase = useSupabaseClient()
      const { data } = await supabase
        .from('profiles')
        .select('onboarding_done, preferred_genres')
        .eq('id', user.value.id)
        .single()
      if (data) profileCache.value = data
    } catch {
      return
    }
  }

  if (profileCache.value && !profileCache.value.onboarding_done) {
    return navigateTo('/onboarding')
  }
})
Sufixul .global.ts
Un fișier din app/middleware/ cu sufixul .global.ts rulează automat pe fiecare navigare, fără să fie nevoie să-l declari în definePageMeta. E diferit de auth.ts (fără .global), care rulează doar unde e cerut explicit.

Observă și profileCache — același pattern useState văzut în Lecția 4: profilul e citit o singură dată din Supabase și cache-uit global, ca să nu repete query-ul la fiecare navigare.

Configurarea din nuxt.config.ts

supabase: {
  redirectOptions: {
    login: '/login',
    callback: '/confirm',
    exclude: ['/', '/login', '/confirm', '/discover', '/movie/*', '/tv/*', '/person/*', '/people', '/collection/*']
  },
  cookieOptions: {
    maxAge: 60 * 60 * 24 * 10,   // 10 zile
    sameSite: 'lax',
    secure: true
  }
}

Modulul @nuxtjs/supabase are propriul middleware global implicit, care ar redirecționa orice pagină neautentificată la login. exclude spune explicit ce rute rămân publice — toate paginile de conținut (discover, detalii film/serial/actor) sunt vizibile fără cont, doar acțiunile personale (watchlist, profil) cer autentificare.

Exercițiu — Trasează fluxul

Un utilizator nou dă click pe „Continue with Google" pentru prima oară. Ordonează pașii:

  1. Trigger-ul on_auth_user_created creează automat o linie în profiles (vezi Lecția 7)
  2. onboarding.global.ts vede onboarding_done: false și redirecționează la /onboarding
  3. confirm.vue confirmă sesiunea și redirecționează la /
  4. Google validează contul și redirecționează la /confirm

Ordine corectă: 4 (Google validează) → 1 (trigger DB creează profilul) → 3 (confirm.vue redirect la /) → 2 (middleware global vede onboarding_done: false, redirect la /onboarding).

Întrebarea 1 din 5 Ce diferență e între app/middleware/auth.ts și app/middleware/onboarding.global.ts?
Corect. Sufixul .global.ts face middleware-ul să ruleze automat pe fiecare navigare din aplicație, fără să fie nevoie de definePageMeta pe fiecare pagină.
Întrebarea 2 din 5 De ce discover.vue și movie/[id].vue nu cer login, deși aplicația are autentificare?
Corect. Fără exclude, middleware-ul implicit al modulului Supabase ar redirecționa orice pagină neautentificată — exclude marchează explicit ce rute rămân publice.
Întrebarea 3 din 5 Ce face confirm.vue după ce Google redirecționează utilizatorul acolo?
Corect. Modulul Supabase procesează automat token-ul OAuth din URL — confirm.vue doar așteaptă acest proces (via onMounted) și verifică rezultatul cu getSession().
Întrebarea 4 din 5 Vrei să adaugi o nouă pagină /settings care cere login. Ce faci?
Corect. Exact pattern-ul din watchlist.vue/profile.vue/onboarding.vue — middleware: 'auth' declanșează app/middleware/auth.ts.
Întrebarea 5 din 5 De ce onboarding.global.ts folosește useState pentru profileCache?
Corect. Fără cache, fiecare navigare ar declanșa un nou supabase.from('profiles').select(...). Cu useState('profile-cache', ...), valoarea persistă între navigări și e citită o singură dată.
