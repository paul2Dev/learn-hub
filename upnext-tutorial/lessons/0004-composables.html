Cel mai simplu composable: useTmdbImage

app/composables/useTmdbImage.ts
const IMAGE_BASE = 'https://image.tmdb.org/t/p'

export function useTmdbImage() {
  const poster = (path: string | null, size: 'w185' | 'w342' | 'w500' | 'original' = 'w342') =>
    path ? `${IMAGE_BASE}/${size}${path}` : null

  const backdrop = (path: string | null, size: 'w780' | 'w1280' | 'original' = 'w1280') =>
    path ? `${IMAGE_BASE}/${size}${path}` : null

  return { poster, backdrop }
}

Niciun ref, niciun useState — doar două funcții helper grupate sub un nume use* ca să beneficieze de auto-import. Un composable nu trebuie neapărat să gestioneze state reactiv; uneori e doar un namespace convenabil pentru logică reutilizabilă.

State global: useWatchlist

Navbar-ul (badge cu numărul de filme din watchlist) și fiecare MediaCard de pe ecran trebuie să știe instant dacă un film e în watchlist — fără să facă fiecare propriul request. Soluția: un singur Set global, ținut cu useState.

app/composables/useWatchlist.ts
export function watchlistKey(movieId: number, mediaType: string) {
  return `${mediaType}:${movieId}`
}

export function useWatchlist() {
  const user = useSupabaseUser()
  const ids = useState<Set<string>>('watchlist-ids', () => new Set())

  async function refresh() {
    if (!user.value) {
      ids.value = new Set()
      return
    }
    const data = await $fetch<WatchlistRow[]>('/api/user/watchlist').catch(() => [])
    ids.value = new Set(data.map(r => watchlistKey(r.movie_id, r.media_type)))
  }

  function has(movieId: number, mediaType: string) {
    return ids.value.has(watchlistKey(movieId, mediaType))
  }

  async function toggle(movieId: number, mediaType: 'movie' | 'tv', tmdbData: object) {
    const key = watchlistKey(movieId, mediaType)
    if (ids.value.has(key)) {
      await $fetch('/api/user/watchlist', { method: 'DELETE', body: { movie_id: movieId, media_type: mediaType } })
      const next = new Set(ids.value); next.delete(key); ids.value = next
    } else {
      await $fetch('/api/user/watchlist', { method: 'POST', body: { movie_id: movieId, media_type: mediaType, tmdb_data: tmdbData } })
      const next = new Set(ids.value); next.add(key); ids.value = next
    }
  }

  return { ids, count: computed(() => ids.value.size), refresh, has, toggle }
}
De ce useState și nu un simplu ref
Dacă useWatchlist() ar face const ids = ref(new Set()), fiecare componentă care apelează composable-ul ar primi propria copie izolată — navbar-ul n-ar ști niciodată că ai adăugat un film din MediaCard. useState('watchlist-ids', ...) garantează că toate apelurile la useWatchlist() din întreaga aplicație citesc și scriu același ref, identificat prin cheia 'watchlist-ids'.

De asta app/app.vue apelează useWatchlist() pentru badge-ul din navbar, iar MediaCard.vue apelează același composable pentru butonul de bookmark — și cele două rămân mereu sincronizate, fără props sau evenimente între ele.

State local: useMediaActions

Pe pagina de detalii a unui film, ai nevoie de mai mult decât „e în watchlist da/nu" — vrei și rating-ul curent, stări de loading separate, și toast-uri de eroare. Acest composable ia mediaId ca parametru și ține state local, nu global:

app/composables/useMediaActions.ts (extras)
export function useMediaActions(mediaId: number, mediaType: 'movie' | 'tv', tmdbData: TmdbData) {
  const user = useSupabaseUser()
  const toast = useToast()

  const inWatchlist = ref(false)
  const isWatched = ref(false)
  const userRating = ref<number | null>(null)
  const loadingWatchlist = ref(false)
  const loadingWatched = ref(false)

  async function fetchStatus() {
    if (!user.value) return
    const [watchlistRes, ratingsRes] = await Promise.all([
      $fetch<WatchlistRow[]>('/api/user/watchlist').catch(() => []),
      $fetch<WatchedRow[]>('/api/user/ratings').catch(() => [])
    ])
    inWatchlist.value = watchlistRes.some(r => r.movie_id === mediaId && r.media_type === mediaType)
    const watched = ratingsRes.find(r => r.movie_id === mediaId && r.media_type === mediaType)
    isWatched.value = !!watched
    userRating.value = watched?.rating ?? null
  }

  onMounted(fetchStatus)

  return { user, inWatchlist, isWatched, userRating, loadingWatchlist, loadingWatched, toggleWatchlist, setRating }
}

Fiecare pagină movie/[id].vue care cheamă useMediaActions(id, 'movie', data) primește propriul set de ref-uri — nu există partajare între pagini diferite, pentru că fiecare pagină vorbește despre un film diferit. Contrast direct cu useWatchlist, unde partajarea globală era exact scopul.

Exercițiu — Prezici comportamentul

Ai două componente MediaCard pe ecran, pentru același film. Dai click pe butonul de watchlist din prima. Ce se întâmplă cu butonul din a doua, fără nicio reîncărcare de pagină?

Răspuns: Se actualizează instant. Ambele carduri citesc has() din același useState('watchlist-ids', ...) — modificarea lui ids.value dintr-un card declanșează reactivitate în toate locurile care citesc acel state, indiferent de componentă.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 De ce useWatchlist folosește useState în loc de un simplu ref()?
Corect. Cheia 'watchlist-ids' face ca toate apelurile useState('watchlist-ids', ...) din întreaga aplicație să acceseze exact același ref — asta permite sincronizarea automată între navbar și carduri.
Întrebarea 2 din 5 useMediaActions primește mediaId ca parametru al funcției. Ce implică asta pentru state-ul returnat?
Corect. Spre deosebire de useWatchlist, useMediaActions folosește ref() simplu — fiecare apel al funcției creează propriile variabile reactive, complet izolate.
Întrebarea 3 din 5 useTmdbImage nu conține niciun ref sau useState. Mai e considerat composable valid?
Corect. Un composable e orice funcție cu prefix use care beneficiază de auto-import din app/composables/. Gestionarea de state reactiv e comună, dar nu obligatorie.
Întrebarea 4 din 5 Ce se întâmplă în useWatchlist dacă user.value devine null (utilizatorul se deloghează)?
Corect. refresh() verifică explicit if (!user.value) { ids.value = new Set(); return } — dar asta se întâmplă doar când refresh() e apelat (de obicei prin watch(user, ...) în app.vue), nu automat la fiecare schimbare de user.
Întrebarea 5 din 5 Ai construi o funcționalitate „recent văzute" (ultimele 5 filme vizitate, vizibile din orice pagină). Ce ai folosi?
Corect. Exact pattern-ul useWatchlist — date care trebuie văzute din orice colț al aplicației (aici: navbar și orice pagină) cer useState cu o cheie stabilă.
