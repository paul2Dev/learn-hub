Toate rutele reale ale aplicației, cum funcționează rutele dinamice [id].vue, și de ce login/confirm arată altfel decât restul aplicației.

Fiecare rând din tabel corespunde 1-la-1 unui fișier în app/pages/ — nu există configurare de router separată. Nuxt scanează folderul la build time și generează rutele automat.

Rute dinamice: [id].vue

Patru pagini folosesc același pattern — un segment variabil în paranteze pătrate. Iată cum app/pages/movie/[id].vue citește parametrul din URL:

app/pages/movie/[id].vue
const route = useRoute()
const id = route.params.id

const [{ data: movie, error }, { data: similarData }] = await Promise.all([
  useFetch<Movie>(`/api/movies/${id}`),
  useFetch<{ results: MovieItem[] }>(`/api/movies/${id}/similar`)
])

if (error.value) {
  throw createError({ statusCode: 404, message: 'Movie not found' })
}

route.params.id vine direct din numele fișierului. Pentru /movie/27205, id este "27205" (string, chiar dacă TMDB ID e numeric). Observă și Promise.all — cele două useFetch pornesc în paralel, nu secvențial.

Pattern repetat
tv/[id].vue, person/[id].vue și collection/[id].vue urmează exact aceeași structură — doar endpoint-ul server și forma datelor diferă. Vezi Lecția 10 pentru cum se extinde acest pattern la o entitate nouă.

Pagini fără chrome: definePageMeta({ layout: false })

login.vue și confirm.vue nu vor navbar/footer — sunt pagini de tranziție. Ambele setează:

app/pages/login.vue
<script setup lang="ts">
definePageMeta({ layout: false })

const supabase = useSupabaseClient()
const loading = ref(false)

async function signInWithGoogle() {
  loading.value = true
  await supabase.auth.signInWithOAuth({
    provider: 'google',
    options: { redirectTo: `${window.location.origin}/confirm` }
  })
  loading.value = false
}
</script>

<template>
  <UApp>
    <div class="fixed inset-0 flex items-center justify-center bg-background px-4">
      ...
    </div>
  </UApp>
</template>
Gotcha real din acest proiect
UpNext nu are folder layouts/ — app/app.vue conține navbar-ul și footer-ul direct în template, în jurul unui simplu <NuxtPage />. Asta înseamnă layout: false aici nu comută între două layout-uri (nu există un al doilea) — pagina își desenează propriul <UApp> de la zero ca să evite complet chrome-ul din app.vue. E o soluție validă, mai simplă, pentru aplicații cu o singură coajă vizuală — dar diferă de pattern-ul „layouts/default.vue + layouts/auth.vue" din cursul general nuxt/.

Protejarea rutelor cu middleware pe pagină

Trei pagini cer autentificare — fiecare o declară explicit în definePageMeta: 

// app/pages/watchlist.vue
definePageMeta({ middleware: 'auth' })

// app/pages/profile.vue
definePageMeta({ middleware: 'auth' })

// app/pages/onboarding.vue
definePageMeta({ middleware: 'auth' })

Numele 'auth' se leagă automat de fișierul app/middleware/auth.ts — Nuxt face maparea după nume, fără import manual. Detalii complete despre ce face acest middleware în Lecția 6.

Exercițiu — Adaugă o pagină nouă

Vrei să adaugi o pagină /genres care listează toate genurile TMDB și le linkuiește la discover. Ce faci?

  1. Creezi app/pages/genres.vue
  2. În <script setup>, apelezi useFetch('/api/movies/genres') (endpoint deja existent)
  3. Nu ai nevoie de nicio configurare de router — ruta /genres apare automat
  4. Dacă vrei ca pagina să ceară login, adaugi definePageMeta({ middleware: 'auth' })

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Pentru ruta /movie/27205, ce valoare are route.params.id în movie/[id].vue?
Corect. Parametrii de rută sunt întotdeauna string-uri. De asta codul folosește Number(id) acolo unde are nevoie de tip numeric (ex: în useMediaActions).
Întrebarea 2 din 5 De ce nu are UpNext un folder layouts/, spre deosebire de multe proiecte Nuxt?
Corect. Layouts multiple au sens când ai mai multe „forme" de shell (ex: admin vs public). UpNext are un singur shell, deci l-a scris direct în app.vue — mai puțină indirecție pentru un caz simplu.
Întrebarea 3 din 5 Cum se protejează /watchlist să nu fie accesat fără login?
Corect. Fiecare pagină protejată declară explicit middleware-ul în definePageMeta. Nuxt leagă string-ul 'auth' de fișierul app/middleware/auth.ts automat.
Întrebarea 4 din 5 Ce fac în paralel cele două useFetch din movie/[id].vue?
Corect. Promise.all([...]) pornește ambele cereri (detalii film + filme similare) simultan — timpul total de așteptare e cât cel mai lent request, nu suma lor.
Întrebarea 5 din 5 Vrei o pagină nouă /genres. Ce trebuie să configurezi în plus față de a crea fișierul?
Corect. File-based routing înseamnă exact asta — niciun pas de configurare manuală. Crearea fișierului e suficientă.
