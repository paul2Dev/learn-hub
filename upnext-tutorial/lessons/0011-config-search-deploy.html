nuxt.config.ts, căutare semantică și deploy

Ultima lecție: fișierul central de configurare linie cu linie, cum funcționează feature flags, arhitectura căutării semantice cu embeddings, și cum ajunge codul pe Vercel.

nuxt.config.ts complet, explicat

export default defineNuxtConfig({
  modules: [
    '@nuxt/eslint',        // linting integrat în build
    '@nuxt/ui',            // componente Nuxt UI v4
    '@nuxtjs/supabase',    // client Supabase + auth (Lecția 6)
    '@vite-pwa/nuxt'       // suport PWA (instalabil pe telefon)
  ],

  ssr: false,              // SPA mode (Lecția 1)

  css: ['~/assets/css/main.css'],   // Tailwind CSS v4

  runtimeConfig: {
    tmdbApiKey: process.env.TMDB_API_KEY,          // privat — doar server (Lecția 5)
    openaiApiKey: process.env.OPENAI_API_KEY,      // privat — pentru embeddings
    seedSecret: process.env.SEED_SECRET,           // privat — protejează /api/admin
    searchDailyLimit: parseInt(process.env.SEARCH_DAILY_LIMIT ?? '15', 10),
    public: {
      tmdbBaseUrl: 'https://api.themoviedb.org/3',
      tmdbImageBaseUrl: 'https://image.tmdb.org/t/p',
      featureSemanticSearch: process.env.FEATURE_SEMANTIC_SEARCH === 'true'
    }
  },

  supabase: {
    redirectOptions: {
      login: '/login',
      callback: '/confirm',
      exclude: ['/', '/login', '/confirm', '/discover', '/movie/*', '/tv/*', '/person/*', '/people', '/collection/*']
    },
    cookieOptions: { maxAge: 60 * 60 * 24 * 10, sameSite: 'lax', secure: true }
  }
})

Trei tipuri de config coexistă aici: modules (ce librării înregistrate global), runtimeConfig (variabile de mediu, server vs public — Lecția 5), și supabase (opțiuni specifice modulului de auth — Lecția 6).

Feature flag: featureSemanticSearch

Căutarea semantică e opțională — controlată de o singură variabilă de mediu, FEATURE_SEMANTIC_SEARCH. Trei locuri verifică acest flag: 

// app/app.vue — arată sau nu link-ul "Smart Search" în navbar
<UButton v-if="user && config.public.featureSemanticSearch" to="/search" ... />

// server/api/movies/semantic-search.get.ts — refuză cererea dacă flag-ul e oprit
if (!config.public.featureSemanticSearch) {
  throw createError({ statusCode: 404, message: 'Feature not enabled.' })
}
De ce feature flag și nu doar „adaugi codul"
Căutarea semantică cere infrastructură suplimentară (embeddings populate în DB, cheie OpenAI activă). Flag-ul permite ca aceeași bază de cod să ruleze fără acea infrastructură (dezactivat), fără să fie nevoie de un branch de cod separat sau de eliminarea feature-ului complet.

De ce RPC și nu query direct din client?
search_media_by_embedding și increment_search_usage sunt funcții Postgres (SECURITY DEFINER), apelate prin supabase.rpc(...) — nu simple select/update. Logica de search (operatorul <=> pentru distanță cosine) și incrementarea atomică a contorului (evitând race conditions la cereri simultane) trăiesc în baza de date, nu în server Nitro, exact pentru că ambele operații trebuie să fie atomice.

Deploy pe Vercel

Nitro detectează automat platforma Vercel la deploy și generează output-ul potrivit (funcții serverless pentru server/api/, plus asset-urile statice). Pași reali:

  1. Conectezi repo-ul GitHub la un proiect Vercel nou
  2. Adaugi variabilele de mediu în dashboard Vercel: SUPABASE_URL, SUPABASE_KEY, TMDB_API_KEY, opțional OPENAI_API_KEY + FEATURE_SEMANTIC_SEARCH=true
  3. Vercel rulează pnpm build automat la fiecare push pe branch-ul principal
  4. Migrațiile SQL din supabase/migrations/ se rulează manual în Supabase Dashboard → SQL Editor, nu automat la deploy
Un detaliu ușor de ratat
Migrațiile SQL nu fac parte din pipeline-ul de build Vercel — sunt independente. Dacă adaugi o migrație nouă local și dai push, aplicația se deployază, dar schema DB pe producție rămâne neschimbată până rulezi manual fișierul SQL în Supabase. E o sursă comună de bug-uri „merge pe localhost, nu merge pe producție".

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 De ce featureSemanticSearch e citit din runtimeConfig.public și nu din runtimeConfig direct?
Corect. Doar valorile din public ajung și în bundle-ul client. tmdbApiKey și openaiApiKey rămân private pentru că sunt secrete — flag-ul e doar o valoare boolean, sigură de expus.
Întrebarea 2 din 5 De ce increment_search_usage e o funcție RPC Postgres, nu un simplu update din server Nitro?
Corect. Dacă doi request-uri simultane ar citi contorul, l-ar incrementa în cod, și l-ar scrie înapoi, ai putea pierde un increment (race condition). Funcția SQL INSERT ... ON CONFLICT DO UPDATE ... RETURNING e atomică la nivel de bază de date.
Întrebarea 3 din 5 Dai push la o migrație SQL nouă pe branch-ul principal. Ce se întâmplă automat pe Vercel?
Corect. Vercel doar rulează pnpm build pentru codul aplicației. Migrațiile din supabase/migrations/ trebuie rulate manual în Supabase Dashboard — un pas separat, ușor de uitat.
Întrebarea 4 din 5 Ce variabile de mediu minime ai avea nevoie ca aplicația să pornească fără căutare semantică?
Corect. OPENAI_API_KEY e necesară doar dacă FEATURE_SEMANTIC_SEARCH=true — fără flag activat, acel cod nu rulează niciodată, deci lipsa cheii nu blochează restul aplicației.
Întrebarea 5 din 5 Ce curs din acest repo ar trebui să parcurgi pentru a înțelege conceptul de "embedding" în profunzime?
Corect. Această lecție arată unde și cum UpNext folosește embeddings, dar cursul embedings/ explică ce sunt și de ce funcționează — cunoștințe transferabile la orice proiect cu căutare semantică.
