Server API cu Nitro

Cum ajunge un click din browser să cheme TMDB fără ca cheia API să părăsească serverul, și de ce toate endpoint-urile arată la fel de simplu.

Fiecare fișier *.get.ts, *.post.ts, *.delete.ts devine automat o rută /api/... — Nitro scanează folderul la build time, exact ca pages/ pentru rute de UI. Sufixul din nume restricționează metoda HTTP acceptată.

Cel mai simplu endpoint: proxy TMDB

server/api/movies/trending.get.ts
export default defineEventHandler(async (event) => {
  const { page = 1 } = getQuery(event)

  return tmdbFetch('/trending/movie/week', { page: Number(page) })
})

Trei linii. Tot lucrul greu (construirea URL-ului, cheia API, parametrii) e mutat în server/utils/tmdb.ts — un utilitar de server, auto-importat exact ca un composable, dar disponibil doar în codul din server/:

server/utils/tmdb.ts
export async function tmdbFetch<T>(
  path: string,
  params: Record<string, string | number | boolean | undefined> = {}
): Promise<T> {
  const config = useRuntimeConfig()

  return $fetch<T>(`https://api.themoviedb.org/3${path}`, {
    params: {
      api_key: config.tmdbApiKey,
      language: 'en-US',
      ...params
    }
  }) as unknown as Promise<T>
}
De ce e sigur
config.tmdbApiKey vine din useRuntimeConfig() apelat aici, în cod care rulează doar pe server (Nitro). Cheia nu ajunge niciodată în bundle-ul JS trimis browserului. Orice pagină care are nevoie de date TMDB cheamă propriul /api/movies/* — niciodată api.themoviedb.org direct din client.

Endpoint protejat: CRUD Supabase

Endpoint-urile din server/api/user/ sunt diferite — nu proxy-ază TMDB, ci citesc/scriu în Postgres via Supabase, și cer autentificare:

server/api/user/watchlist.post.ts
import { serverSupabaseClient, serverSupabaseUser } from '#supabase/server'
import { z } from 'zod'

const schema = z.object({
  movie_id: z.number().int().positive().max(999_999_999),
  media_type: z.enum(['movie', 'tv']),
  tmdb_data: z.record(z.string(), z.unknown()).optional().default({})
    .refine(v => JSON.stringify(v).length <= 50_000, 'tmdb_data too large')
})

export default defineEventHandler(async (event) => {
  const user = await serverSupabaseUser(event)
  if (!user) throw createError({ statusCode: 401, message: 'Unauthorized' })

  const raw = await readBody(event)
  const parsed = schema.safeParse(raw)
  if (!parsed.success) {
    throw createError({ statusCode: 400, message: parsed.error.issues[0]?.message ?? 'Invalid input' })
  }
  const body = parsed.data

  const client = await serverSupabaseClient(event)
  const { data, error } = await client
    .from('watchlist')
    .insert({ user_id: user.id, movie_id: body.movie_id, media_type: body.media_type, tmdb_data: body.tmdb_data })
    .select()
    .single()

  if (error) throw createError({ statusCode: 500, message: 'Internal server error' })
  return data
})

Patru pași fixați, în ordine, în aproape fiecare endpoint din server/api/user/:

  1. serverSupabaseUser(event) — verifică sesiunea; null → 401
  2. readBody(event) + schema.safeParse() — validare Zod; eșec → 400
  3. serverSupabaseClient(event) — client Supabase cu sesiunea userului atașată
  4. Query Postgres — RLS-ul din bază decide, în plus, ce poate atinge acel user (vezi Lecția 7)

Middleware de server: headers de securitate

server/middleware/ rulează pe fiecare request, înainte de orice handler din server/api/. UpNext îl folosește ca să seteze headere HTTP de securitate global:

server/middleware/security.ts
export default defineEventHandler((event) => {
  setResponseHeaders(event, {
    'X-Content-Type-Options': 'nosniff',
    'X-Frame-Options': 'DENY',
    'Referrer-Policy': 'strict-origin-when-cross-origin',
    'Strict-Transport-Security': 'max-age=31536000; includeSubDomains',
    'Content-Security-Policy': [
      "default-src 'self'",
      "img-src 'self' data: blob: https://image.tmdb.org https://*.googleusercontent.com",
      "connect-src 'self' https://*.supabase.co wss://*.supabase.co https://api.themoviedb.org",
      "object-src 'none'"
      // ...
    ].join('; ')
  })
})
server/api/ vs server/middleware/
server/api/*.get.ts — răspunde la o rută specifică, cu o metodă HTTP specifică.
server/middleware/*.ts — rulează pe orice request către server, indiferent de rută; nu returnează un răspuns direct, ci pregătește contextul (aici: headere) pentru handler-ul care urmează.

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 De ce niciun endpoint din server/api/movies/ nu importă cheia TMDB direct?
Corect. Centralizarea în tmdbFetch() înseamnă că orice endpoint nou o poate refolosi fără să repete logica de citire a cheii — un singur loc de adevăr.
Întrebarea 2 din 5 Ce se întâmplă dacă cineva trimite un request POST către /api/movies/trending?
Corect. Sufixul .get.ts restricționează strict metoda acceptată — Nitro respinge automat orice altă metodă cu 405, fără cod suplimentar din partea ta.
Întrebarea 3 din 5 În watchlist.post.ts, ce se întâmplă dacă body-ul trimis nu respectă schema Zod?
Corect. parsed.success === false declanșează createError({ statusCode: 400, ... }) înainte ca vreun query Supabase să ruleze — datele proaste nu ajung niciodată la baza de date.
Întrebarea 4 din 5 Ce diferență e între server/api/ și server/middleware/?
Corect. security.ts din server/middleware/ rulează pentru absolut orice request către server, indiferent de rută — spre deosebire de un handler din server/api/, care e legat de o rută anume.
Întrebarea 5 din 5 Vrei un nou endpoint GET /api/movies/now-playing. Ce fișier creezi și ce funcție reutilizezi?
Corect. Sufixul .get.ts pentru metoda HTTP, locația în server/api/movies/ pentru convenție, și tmdbFetch() pentru a reutiliza logica existentă de proxy TMDB.
