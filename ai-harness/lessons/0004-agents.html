Agents & Fluxuri Multi-Agent

Lansează sub-agenți specializați pentru lucru în paralel, protecția contextului sau aport de expertiză de domeniu.

Ce este un subagent?

Un subagent este un agent independent lansat de agentul curent prin tool-ul Agent. Pornește cu o fereastră de context proaspătă — nu are nicio memorie a sesiunii părinte, niciun istoric de conversație și niciun acces la tool-uri pe care agentul părinte nu le-a descris explicit în promptul său.

Când subagentul termină, returnează un singur mesaj înapoi la părinte. Părintele poate folosi acel rezultat pentru a-și continua propria activitate.

De ce să lansezi un subagent?
Două motive: paralelism (rularea simultană a lucrului independent) și protecția contextului (menținerea rezultatelor mari de căutare în afara ferestrei de context a părintelui, care ar încetini inferența și ar crește costul).

Tool-ul Agent

Tool-ul Agent acceptă câțiva parametri cheie: 

Agent({
  description: "Găsește toate comentariile TODO din src/",   // etichetă scurtă
  subagent_type: "Explore",                                   // specializare opțională
  prompt: "Caută în src/ comentariile TODO. Raportează fiecare cu calea fișierului și numărul liniei.",
  run_in_background: false,                                   // foreground implicit
  isolation: "worktree"                                       // izolare git opțională
})

Parametrii explicați

  • description — o etichetă de 3–5 cuvinte afișată în UI cât timp agentul rulează. Scrie-o ca o acțiune: "Caută comentarii TODO".
  • subagent_type — selectează o variantă de agent specializat. Dacă este omis, se folosește agentul de uz general.
  • prompt — briefing-ul complet pentru subagent. Scrie-l ca și cum ai da instrucțiuni unui coleg care nu a văzut niciodată conversația ta. Include căi de fișiere, obiective și context.
  • run_in_background — true pentru a rula în paralel cât timp părintele continuă. Părintele este notificat când subagentul termină.
  • isolation: "worktree" — creează un worktree git temporar pentru ca subagentul să poată face modificări fără a afecta arborele tău de lucru.

Tipuri de agent specializat

1

claude (implicit)

Uz general. Acces complet la tool-uri. Folosit când niciun alt tip nu se potrivește sau când sarcina acoperă mai multe domenii.

toate tool-urile
2

Explore

Agent de căutare rapid, read-only. Optimizat pentru "găsește fișiere după pattern" și "grep după simbol." Folosit pentru căutări țintite, nu analiză amplă.

read-only
3

Plan

Arhitect software. Returnează planuri de implementare pas cu pas, identifică fișierele critice, ia în considerare compromisurile. Nu scrie cod.

doar design
4

code-reviewer

Revizuiește codul pentru corectitudine, bug-uri și oportunități de simplificare. Util ca a doua opinie independentă pe un diff.

doar review

Agenții specializați au un set mai restrâns de tool-uri și instrucțiuni mai stricte, ceea ce îi face mai rapizi și mai ieftini de rulat decât agentul general pentru sarcina lor specifică.

Foreground vs. background

Implicit este foreground: părintele așteaptă ca subagentul să termine înainte de a continua. Folosește foreground când ai nevoie de rezultatul subagentului pentru a decide ce să faci mai departe.

Cu run_in_background: true, părintele continuă imediat. Subagentul rulează în concurență și părintele este notificat când se termină. Folosește background când ai muncă cu adevărat independentă de făcut în paralel. 

// Foreground — așteaptă rezultatul
const result = Agent({
  description: "Auditează dependențele",
  prompt: "Citește package.json și listează toate dependențele cu probleme de securitate cunoscute."
});
// părintele așteaptă aici, apoi folosește rezultatul

// Background — continuă în paralel
Agent({
  description: "Generează docs în background",
  prompt: "Generează documentația API pentru src/api/. Salvează în docs/api.md.",
  run_in_background: true
});
// părintele continuă imediat; notificat când agentul termină

Scrierea prompturilor eficiente pentru subagent

Deoarece subagentul pornește rece — fără memorie a conversației tale — promptul trebuie să fie complet autonom. Un prompt slab produce rezultate vagi și nefolositoare.

Anti-pattern
"Pe baza descoperirilor tale, repară bug-ul." — Subagentul nu are descoperiri. Nu delega niciodată sinteza informațiilor pe care agentul nu le deține.

Un prompt puternic pentru subagent include:

  • Obiectiv — ce trebuie realizat și de ce
  • Context — căi de fișiere relevante, nume de variabile, constrângeri pe care agentul trebuie să le cunoască
  • Format output — cum ar trebui să arate răspunsul ("raport în sub 200 de cuvinte", "returnează o listă JSON")
  • Ce să omiți — ce NU ar trebui să facă agentul, pentru a menține scopul strict
// Slab
prompt: "Găsește bug-ul din modulul de autentificare"

// Puternic
prompt: "Endpoint-ul de login din src/auth/login.ts:47 returnează 401 pentru tokeni valizi.
  Citește acel fișier și logica de validare a tokenilor din src/auth/validate.ts.
  Identifică linia specifică care cauzează respingerea incorectă.
  Raportează: cauza rădăcină, numărul liniei și o sugestie de corectare pe o singură linie.
  NU modifica niciun fișier — doar cercetare."

Practic: rulează agenți de cercetare în paralel

Exercițiu — Lansează doi agenți în background

Folosește doi agenți Explore în background pentru a cerceta două lucruri în același timp, apoi combină rezultatele.

  1. În Claude Code, cere: 
    Lansează doi agenți în background:
1. Găsește toate fișierele din acest proiect care importă din 'react'
2. Găsește toate fișierele care au comentarii TODO
Raportează ambele rezultate când sunt gata.
  2. Claude ar trebui să lanseze doi sub-agenți Explore în paralel. Urmărește cum etichetele agenților apar în UI cât timp rulează.
  3. Când ambii termină, vei primi două mesaje de rezultat separate. Apoi întreabă: "Rezumă ambele descoperiri într-un singur tabel."
  4. Observă că contextul sesiunii părintelui nu a fost poluat cu outputul brut de căutare al fiecărui agent — au sosit doar rezumatele.
Exercițiu — Izolare worktree

Lansează un agent cu isolation: "worktree" pentru a-l lăsa să facă modificări în siguranță.

  1. Cere lui Claude: "Lansează un agent într-un worktree pentru a adăuga comentarii JSDoc la fiecare funcție din src/utils/. Raportează numele branch-ului când termină."
  2. Agentul lucrează pe o copie izolată. Arborele tău de lucru rămâne neatins.
  3. Dacă îți plac modificările, mergi branch-ul worktree-ului; altfel, renunță la el fără nicio curățare necesară.

Verifică-ți înțelegerea

Întrebarea 1 din 5 Un subagent este lansat. Are acces la istoricul conversației sesiunii părintelui?
Corect. Subagentii pornesc întotdeauna rece. Acesta este o caracteristică, nu un bug — protejează contextul părintelui și forțează promptul subagentului să fie autonom. Tot ce are nevoie subagentul trebuie scris în promptul său.
Întrebarea 2 din 5 Trebuie să cauți o funcție specifică în întreaga bază de cod. Ce tip de agent este cel mai bun?
Corect. Explore este optimizat pentru căutări țintite de fișiere/simboluri. Este mai rapid și mai ieftin decât agentul general pentru această clasă de sarcini. Rezervă claude pentru sarcini care necesită scrierea de cod sau analiză transversală.
Întrebarea 3 din 5 Când ar trebui să folosești run_in_background: true?
Corect. Background-ul este util doar când există muncă paralelă reală. Dacă părintele are nevoie de rezultatul subagentului pentru a decide pasul următor, folosește foreground. Lansarea unui agent în background și așteptarea imediată a lui este overhead irosit.
Întrebarea 4 din 5 Ce face isolation: "worktree"?
Corect. Un worktree este un checkout separat al aceluiași repo pe un branch nou. Subagentul poate scrie fișiere liber; branch-ul tău principal rămâne neatins. Dacă agentul nu face nicio modificare, worktree-ul este curățat automat.
Întrebarea 5 din 5 Câte mesaje trimite un subagent terminat înapoi la părinte?
Corect. Un subagent returnează exact un mesaj — rezumatul sau răspunsul său final. Părintele nu vede niciodată apelurile interne de tool ale subagentului sau raționamentul intermediar. Aceasta este ceea ce face fluxurile multi-agent gestionabile: fiecare agent este o cutie neagră care returnează un singur rezultat.
