Ce este AI Harness?

Concept, arhitectură și primul exercițiu practic — trasarea unui request prin harness.

Conceptul

Un LLM brut primește text și produce text. Atât. Nu poate citi un fișier, nu poate rula o comandă și nu reține nimic între conversații. Este puternic, dar inert — un creier fără mâini și fără memorie de scurtă durată.

Un AI harness este stratul care îi dă creierului mâini. Înfășoară LLM-ul într-un runtime structurat care oferă patru lucruri:

  • Tool-uri — funcții pe care modelul le poate apela (Read, Bash, WebSearch, …)
  • Context — istoricul conversației, system prompt-ul și rezultatele tool-urilor pe care modelul le poate vedea
  • Permissions — reguli despre ce tool-uri sunt permise și care sunt blocate
  • Hooks — comenzi shell care se declanșează la evenimente (înainte de rularea unui tool, după finalizare, la încheierea sesiunii)

Combinate, aceste patru elemente formează un agent: un LLM care poate percepe mediul înconjurător, poate lua acțiuni și poate fi constrâns la un domeniu definit de comportament. Harness-ul este cel care transformă modelul dintr-un chatbot într-un colaborator de inginerie software.

Idee cheie
Harness-ul rulează în afara modelului. Claude nu vede niciodată verificarea de permisiuni sau script-ul de hook — vede doar rezultatul apelului său de tool returnat în context. Modelul gândește în tokeni; harness-ul gândește în evenimente.

Cele patru straturi

Harness-ul Claude Code are patru straturi suprapuse. Fiecare apel de tool trece prin toate patru, de sus în jos și înapoi.

1

Modelul (Claude)

Citește fereastra de context, decide ce să facă în continuare și emite fie un răspuns text, fie un apel de tool. Acesta este singurul strat care folosește inferență AI.

claude-sonnet-4-x
2

Tool Registry

Setul de funcții disponibile pentru model. Built-in-uri: Read, Write, Edit, Bash, Grep, Glob, Agent, WebSearch. Extins prin servere MCP.

tools + MCP
3

Permission Layer

Verifică fiecare apel de tool față de listele allow/deny din settings.json înainte de execuție. Poate bloca o comandă Bash specifică în timp ce le permite pe celelalte.

settings.json
4

Hook Layer

Comenzi shell declanșate de evenimente ale harness-ului: PreToolUse, PostToolUse, Stop, Notification. Shell pur — fără implicarea modelului.

hooks

Aceste straturi sunt independente. Poți configura permission layer-ul fără să scrii vreun hook. Poți adăuga un tool MCP fără să schimbi permisiunile. Fiecare strat are un singur rol și se compun curat împreună.

Trasarea unui request

Imaginează-ți că îi ceri lui Claude: "Citește package.json și spune-mi versiunea." Iată exact ce face harness-ul, pas cu pas:

1

7

Rezultatul tool-ului (conținutul fișierului) este injectat înapoi în fereastra de context. Modelul îl citește și generează răspunsul text: "Versiunea este 1.4.2."

Stratul 1 — Model
Ce să observi
Modelul este implicat doar la pașii 1 și 7. Tot ce se întâmplă între ele — verificarea permisiunilor, hooks-urile, I/O-ul efectiv al fișierului — este logică pură de harness. Modelul nu "știe" de permission layer; pur și simplu primește un rezultat sau o eroare.

Practic: inspectează-ți harness-ul

Ai deja un harness care rulează — ești în interiorul lui chiar acum. Acest exercițiu face vizibile cele patru straturi.

Exercițiul 1 — Găsește permission layer-ul
  1. Deschide un terminal în directorul rădăcină al proiectului și rulează:
    cat .claude/settings.json
    (Sau verifică %USERPROFILE%\.claude\settings.json pentru configurarea la nivel de utilizator pe Windows.)
  2. Caută un bloc "permissions". Vei vedea arrayuri allow și deny — acestea sunt permission layer-ul tău.
  3. Caută un bloc "hooks". Gol? Acesta este hook layer-ul fără nimic înregistrat încă — exact ce vom remedia în Lecția 2.
  4. Caută "mcpServers". Orice intrări de acolo sunt extensii ale tool registry-ului tău.
Exercițiul 2 — Cere lui Claude să-și descrie propriul harness
  1. În Claude Code, întreabă: "Ce tool-uri ai disponibile în momentul acesta?"
  2. Lista pe care ți-o oferă este viziunea sa asupra tool registry-ului (Stratul 2).
  3. Apoi întreabă: "Ce permisiuni sunt configurate pentru această sesiune?" — va încerca să citească settings.json, oferindu-ți o trasare live prin harness.

Cum arată un hook

Înainte de quiz, iată o previzualizare concretă a ceea ce vei scrie în Lecția 2. Aceasta este o înregistrare completă și funcțională a unui hook în settings.json: 

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"Bash tool finished\" >> /tmp/harness.log"
          }
        ]
      }
    ]
  }
}

Citește asta ca: "După fiecare apel al tool-ului Bash, rulează această comandă shell." Acea comandă shell este pură OS — fără model, fără AI, doar un script. Harness-ul o execută și continuă.

Notă Windows
Pe Windows cu Claude Code, comenzile de hook rulează implicit în PowerShell. Înlocuiește /tmp/harness.log cu $env:TEMP\harness.log sau o cale relativă la proiect.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Îi ceri lui Claude să ruleze npm install. Înainte ca comanda să se execute, ce strat verifică dacă aceasta este permisă?
Corect. Permission layer-ul citește settings.json și verifică listele allow/deny înainte de execuția oricărui tool. Modelul nu vede niciodată această verificare — vede doar rezultatul sau o eroare.
Întrebarea 2 din 5 Un hook înregistrat pe PostToolUse pentru tool-ul Write se declanșează în ce moment?
Corect. PostToolUse se declanșează după finalizarea tool-ului. Dacă ai nevoie să interceptezi înainte de execuție (de exemplu, pentru a valida inputul), folosește în schimb PreToolUse.
Întrebarea 3 din 5 Conectezi un tool personalizat de interogare a bazei de date printr-un server MCP. În ce strat intră acest tool?
Corect. Serverele MCP extind tool registry-ul. Odată înregistrat, modelul poate apela tool-ul tău personalizat exact ca un built-in — și tot trece prin permission layer înainte de execuție.
Întrebarea 4 din 5 Care dintre cele patru straturi ale harness-ului este singurul care folosește inferență AI?
Corect. Celelalte trei straturi sunt logică pură: căutarea tool-ului, verificarea fișierului de configurare și execuția script-ului shell. Numai stratul model apelează API-ul LLM. Această separare este motivul pentru care harness-urile pot fi rapide, deterministe și auditabile.
Întrebarea 5 din 5 Un script de hook al harness-ului primește inputul tool-ului ca JSON pe stdin. În ce limbaj rulează?
Corect. Un hook este doar un șir "command" — orice comandă shell validă. Poate apela un script Python, un fișier Node.js, un one-liner PowerShell sau un binar compilat. Harness-ul nu se preocupă de asta; rulează comanda și continuă.
