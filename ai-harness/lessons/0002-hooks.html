Hooks

Comenzi shell pe care harness-ul le rulează în numele tău — înainte de tool-uri, după tool-uri și la încheierea sesiunilor.

Ce este un hook?

Un hook este o comandă shell pe care o înregistrezi în settings.json. Harness-ul o rulează automat când se declanșează un eveniment specific — fără prompt, fără implicarea modelului. Hooks-urile sunt shell pur: un script Python, un one-liner PowerShell, un binar compilat, orice poate executa shell-ul tău.

Gândește-te la hooks-uri ca la ascultători de evenimente atașați pe exteriorul harness-ului. Modelul nu le vede niciodată — pur și simplu vede rezultatele tool-urilor sosind în context, fără să știe că un script de logging a rulat între timp.

De ce contează hooks-urile
Fără hooks-uri, harness-ul este o cutie neagră — nu poți intercepta, loga, bloca sau reacționa la ce face. Hooks-urile îți oferă o îmbinare programabilă între "modelul decide" și "tool-ul se execută."

Cele patru evenimente de hook

1

PreToolUse

Se declanșează înainte de execuția unui tool. Primește numele tool-ului și argumentele de input ca JSON pe stdin. Dacă hook-ul iese cu cod non-zero, tool-ul este blocat.

poate bloca
2

PostToolUse

Se declanșează după finalizarea unui tool. Primește numele tool-ului, inputul și outputul său. Codul de ieșire este ignorat — tool-ul a rulat deja.

doar observare
3

Stop

Se declanșează când agentul își termină tura (produce un răspuns text final). Util pentru notificări, curățare sau rezumate post-sesiune.

sfârșitul sesiunii
4

Notification

Se declanșează când harness-ul trimite o notificare utilizatorului (de ex. "așteptând input"). Folosit pentru a declanșa notificări desktop sau alerte.

evenimente UI

Structura configurării

Hooks-urile se află în settings.json sub o cheie de nivel superior "hooks". Fiecare nume de eveniment se mapează la un array de obiecte matcher. Fiecare matcher țintește unul sau mai multe tool-uri și listează comenzile de rulat. 

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node scripts/validate-bash.js"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "python scripts/audit-log.py"
          }
        ]
      }
    ]
  }
}

Puncte cheie din această configurare:

  • "matcher": "Bash" — se declanșează doar pentru tool-ul Bash
  • "matcher": "*" — se declanșează pentru orice apel de tool
  • "type": "command" — singurul tip suportat în prezent
  • Mai multe comenzi de hook sub un singur matcher rulează în ordine

Citirea inputului de hook

Harness-ul trimite prin pipe un obiect JSON la stdin-ul script-ului tău de hook. Pentru PreToolUse și PostToolUse arată astfel: 

// stdin PreToolUse
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf dist/"
  }
}

// stdin PostToolUse (adaugă tool_response)
{
  "tool_name": "Bash",
  "tool_input": { "command": "ls src/" },
  "tool_response": {
    "output": "index.ts\nutils.ts\ntypes.ts\n",
    "exit_code": 0
  }
}

Script-ul tău citește din stdin, face ce are de făcut, apoi iese. Pentru PreToolUse: ieși cu 0 pentru a permite, ieși cu cod non-zero pentru a bloca.

Practic: scrie primele tale două hooks-uri

Exercițiul 1 — Jurnal de audit (PostToolUse)

Loghează fiecare apel de tool într-un fișier. Acesta este cel mai simplu hook util — append pur, fără logică.

  1. Creează .claude/hooks/audit.sh în proiectul tău: 
    #!/bin/sh
# Citește stdin și adaugă cu timestamp
echo "[$(date -u +%H:%M:%S)] $(cat -)" >> .claude/audit.log
    Pe Windows, folosește un script PowerShell (audit.ps1): 
    $input = $input | Out-String
$ts = (Get-Date -Format "HH:mm:ss")
Add-Content .claude/audit.log "[$ts] $input"
  2. Înregistrează-l în .claude/settings.json: 
    {
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "sh .claude/hooks/audit.sh"
          }
        ]
      }
    ]
  }
}
  3. Rulează Claude Code și cere-i să citească un fișier. Apoi verifică .claude/audit.log — ar trebui să vezi apelul de tool logat.
Exercițiul 2 — Blochează comenzi periculoase (PreToolUse)

Interceptează apelurile Bash și blochează orice comandă care conține rm -rf.

  1. Creează .claude/hooks/guard.sh: 
    #!/bin/sh
INPUT=$(cat -)
echo "$INPUT" | python3 -c "
import sys, json
data = json.load(sys.stdin)
cmd = data.get('tool_input', {}).get('command', '')
if 'rm -rf' in cmd:
    print('BLOCAT: rm -rf nu este permis de hook guard', file=sys.stderr)
    sys.exit(1)
"
  2. Adaugă o intrare PreToolUse alături de hook-ul existent în settings.json.
  3. Cere lui Claude să ruleze rm -rf /tmp/test. Hook-ul ar trebui să blocheze comanda. Urmărește cum harness-ul raportează "tool blocat de hook" în loc să execute.
Sfat
Poți de asemenea să returnezi un obiect JSON dintr-un hook PreToolUse cu {"decision": "block", "reason": "mesajul tău"} pentru a trimite utilizatorului un mesaj de eroare clar în loc de un simplu cod de ieșire non-zero.

Verifică-ți înțelegerea

Întrebarea 1 din 5 Un hook PreToolUse iese cu codul 2. Ce se întâmplă cu apelul de tool?
Corect. Orice ieșire non-zero dintr-un hook PreToolUse blochează tool-ul. Modelul primește o eroare, nu rezultatul tool-ului. Codurile de ieșire ale PostToolUse sunt ignorate deoarece tool-ul a rulat deja.
Întrebarea 2 din 5 Cum transmite harness-ul detaliile apelului de tool la script-ul tău de hook?
Corect. Harness-ul trimite JSON prin pipe la stdin-ul script-ului tău. Script-ul tău citește stdin, parsează JSON-ul și acționează pe câmpurile tool_name și tool_input. Asta face hooks-urile independente de limbaj — orice runtime care poate citi stdin funcționează.
Întrebarea 3 din 5 Ce valoare de matcher declanșează hook-ul pentru orice apel de tool, indiferent de tool?
Corect. Wildcardul "*" se potrivește cu toate numele de tool-uri. Este util pentru jurnale de audit sau limitatoare de rată care ar trebui să ruleze la fiecare acțiune. Matchere specifice precum "Bash" vizează un singur tool.
Întrebarea 4 din 5 Vrei să trimiți o notificare desktop când Claude își termină tura. La ce eveniment te abonezi?
Corect. Stop se declanșează când agentul își termină tura — ideal pentru notificări de tip "gata". Notification se declanșează la notificările harness-ului din interiorul sesiunii (cum ar fi "așteptând utilizatorul"), nu la terminarea turei.
Întrebarea 5 din 5 Știe modelul (Claude) că a rulat un hook între apelul său de tool și rezultat?
Corect. Hooks-urile rulează în afara ferestrei de context a modelului. Modelul vede rezultatul tool-ului (sau o eroare de blocare) — niciodată hook-ul în sine. Acesta este un design intenționat: hooks-urile sunt logică controlată de operator, nu parte din bucla de raționament AI.
Întreabă-ți profesorul

Vrei să blochezi un pattern bash specific? Nu știi cum să parsezi stdin în limbajul tău preferat? Întreabă-mă — îți pot arăta scripturi de hook în Python, PowerShell, Node.js sau shell.
