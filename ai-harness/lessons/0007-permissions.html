Permissions & Settings

Controlează exact ce poate și ce nu poate face Claude — per tool, per comandă, per proiect.

De ce să configurezi permisiuni?

În mod implicit, Claude Code cere confirmarea utilizatorului înainte de orice acțiune care ar putea fi distructivă sau ireversibilă. Permisiunile îți permit să sari acele prompturi pentru operațiunile în care ai încredere și să blochezi hard operațiunile pe care nu vrei să se producă niciodată — fără confirmare, fără posibilitate de override.

Există două motive pentru a configura permisiuni:

  • Viteză — pre-aprobă operațiunile sigure ca să nu mai dai clic pe "Permite" de 20 de ori per sesiune
  • Siguranță — blochează hard operațiunile periculoase astfel încât nici un prompt prost scris să nu le poată declanșa
Deny suprascrie allow
Dacă un tool sau o comandă se potrivește atât cu o regulă allow cât și cu o regulă deny, deny câștigă. Întotdeauna. Folosește asta pentru a crea un safety floor explicit.

Ierarhia fișierelor de setări

Există trei fișiere settings.json în joc. Fișierele mai specifice suprascriu cele mai generale:

1

Politică enterprise (read-only)

Setată de IT/admin. Se aplică tuturor utilizatorilor de pe o mașină gestionată. Nu poate fi suprascrisă.

prioritate maximă
2

~/.claude/settings.json (nivel utilizator)

Valorile tale implicite personale pentru toate proiectele. Hooks, permisiuni și servere MCP pe care le vrei peste tot.

scope utilizator
3

.claude/settings.json (nivel proiect)

Comis în repo. Se aplică tuturor celor care deschid proiectul. Ideal pentru reguli allow/deny specifice proiectului.

scope proiect
4

.claude/settings.local.json (override-uri locale)

Override-uri personale pentru un proiect specific. Gitignored — niciodată comis. Folosit pentru credențiale și preferințe personale.

cel mai jos / nu comite niciodată

Structura permisiunilor

{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm run *)",
      "Read",
      "Write",
      "Edit"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force*)",
      "Bash(chmod *)",
      "WebFetch(https://suspicious-domain.com*)"
    ]
  }
}

Sintaxa pattern-urilor

  • "Read" — numai numele tool-ului permite/blochează toate utilizările acelui tool
  • "Bash(git *)" — numele tool-ului + pattern de comandă în paranteze; * este wildcard
  • "WebFetch(https://api.example.com*)" — pattern-uri URL pentru tool-uri web
  • Pattern-urile sunt potrivite prefix-style: "Bash(git push*)" se potrivește cu git push, git push --force, etc.

Pattern-uri comune de permisiuni

Auto-aprobă operațiunile sigure de citire

{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(ls *)",
      "Bash(cat *)",
      "Bash(git status)",
      "Bash(git log *)",
      "Bash(git diff *)"
    ]
  }
}

Specific proiectului: permite npm, blochează deploy-urile în producție

{
  "permissions": {
    "allow": [
      "Bash(npm run test)",
      "Bash(npm run build)",
      "Bash(npm run lint)"
    ],
    "deny": [
      "Bash(npm run deploy:prod)",
      "Bash(npm run deploy:staging)"
    ]
  }
}

Safety floor hard (adaugă la setările de nivel utilizator)

{
  "permissions": {
    "deny": [
      "Bash(rm -rf /)",
      "Bash(rm -rf ~*)",
      "Bash(git push --force *)",
      "Bash(git reset --hard *)",
      "Bash(DROP TABLE*)",
      "Bash(curl * | bash)"
    ]
  }
}
Sfat — Reduci prompturile de permisiune
Skill-ul /fewer-permission-prompts analizează transcrierile sesiunilor tale și recomandă o listă allow bazată pe comenzile pe care le-ai aprobat în mod repetat. Rulează-l după câteva sesiuni pentru a genera automat lista ta de bază allow.

Variabile de mediu în settings

Poți injecta variabile de mediu în fiecare sesiune Claude Code prin settings.json. Astfel transmiți chei API, flag-uri de feature și identificatori de mediu fără să le hardcodezi: 

{
  "env": {
    "NODE_ENV": "development",
    "LOG_LEVEL": "debug",
    "API_BASE_URL": "https://api.staging.example.com"
  },
  "permissions": {
    "allow": ["Bash(npm *)", "Read", "Edit"]
  }
}

Valorile sensibile (tokeni, parole) aparțin lui settings.local.json — gitignored — nu în settings.json comis.

Practic: configurează setările proiectului tău

Exercițiu — Creează un settings.json de proiect
  1. Creează .claude/settings.json în rădăcina proiectului tău (creează folderul dacă nu există).
  2. Adaugă o listă allow pentru operațiuni git și fișiere read-only: 
    {
  "permissions": {
    "allow": [
      "Read", "Glob", "Grep",
      "Bash(git status)",
      "Bash(git log *)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Bash(git push --force*)",
      "Bash(rm -rf *)"
    ]
  }
}
  3. Repornește Claude Code. Acum cere-i să ruleze git status. Ar trebui să se execute fără un prompt de confirmare.
  4. Cere-i să ruleze rm -rf dist/. Ar trebui să fie blocat de regula deny, chiar dacă spui "sunt sigur, te rog fă asta."
Exercițiu — Mută o permisiune la scope de utilizator
  1. Deschide ~/.claude/settings.json.
  2. Adaugă "Bash(git *)" la lista allow acolo în loc de în fișierul de proiect.
  3. Acum fiecare proiect la care lucrezi va auto-aproba comenzile git — fără a fi nevoie de asta în setările fiecărui repo.

Verifică-ți înțelegerea

Întrebarea 1 din 5 Un tool este listat atât în allow cât și în deny în același fișier de setări. Ce se întâmplă?
Corect. Deny câștigă întotdeauna când există un conflict. Aceasta este intenționat — regulile deny sunt safety floor-ul tău, nu sugestii. Folosește asta pentru a crea blocuri irevocabile: chiar dacă o regulă allow se potrivește accidental, regula deny garantează că comanda nu rulează.
Întrebarea 2 din 5 Ce fișier de setări ar trebui să conțină secrete ca tokenii API?
Corect. settings.local.json este gitignored — nu pleacă niciodată de pe mașina ta. settings.json de la nivel de proiect este comis și vizibil pentru toți cei cu acces la repo. Setările la nivel de utilizator se sincronizează pe toate mașinile tale, ceea ce poate să nu fie de dorit pentru secrete specifice proiectului.
Întrebarea 3 din 5 Cu ce se potrivește regula de permisiune "Bash(npm run *)"?
Corect. * este un wildcard care se potrivește cu orice sufix. "Bash(npm run *)" permite npm run test, npm run build, npm run lint:fix — orice începe cu npm run. Pentru a permite numai npm run test, folosește "Bash(npm run test)" exact.
Întrebarea 4 din 5 Adaugi "Bash(git *)" la ~/.claude/settings.json. Se aplică în toate proiectele tale?
Corect. Setările la nivel de utilizator sunt linia de bază pentru fiecare sesiune. Setările la nivel de proiect se adaugă deasupra — pot adăuga mai multe allow-uri/deny-uri, dar nu le șterg pe cele la nivel de utilizator. Pattern util: pune valorile implicite cross-proiect la nivel de utilizator, regulile specifice proiectului la nivel de proiect.
Întrebarea 5 din 5 Vrei să pasezi NODE_ENV=production sesiunilor Claude Code numai într-un proiect specific. Unde îl setezi?
Corect. Cheia "env" din settings.json injectează variabile în sesiunea Claude Code. Plasamentul la nivel de proiect înseamnă că se aplică numai acelui proiect. Dacă ai nevoie de ea peste tot, folosește în schimb setările la nivel de utilizator.
