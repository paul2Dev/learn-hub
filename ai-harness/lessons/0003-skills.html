Skills

Împachetează instrucțiuni reutilizabile în slash commands pe care le poți invoca în orice proiect.

Ce este un skill?

Un skill este un set de instrucțiuni cu nume, reutilizabil, pe care îl invoci cu un slash command — de exemplu, /teach, /code-review sau /deploy. Când tastezi /skill-name, harness-ul încarcă conținutul skill-ului și îl injectează în sesiune ca un prompt structurat, opțional cu resurse bundled, fișiere de format și sub-prompturi.

Skills-urile sunt răspunsul harness-ului la întrebarea: "Cum să nu mai scriu același prompt lung de fiecare dată?" Fac fluxurile de lucru experților repetabile și partajabile între proiecte.

Skills vs. hooks
Hooks-urile se declanșează automat la evenimente ale harness-ului. Skills-urile sunt invocate explicit de utilizator. Un hook este mereu activ; un skill este mereu la cerere.

Structura fișierelor

Skills-urile se află în directorul ~/.claude/skills/ la nivel de utilizator, sau în .claude/skills/ în interiorul unui proiect pentru skills-uri cu scope de proiect. Fiecare skill este un folder numit după comandă: 

~/.claude/skills/
├── deploy/
│   ├── deploy.md          ← entry point (același nume cu folderul)
│   ├── checklist.html     ← resursă bundled
│   └── schema.json        ← altă resursă
├── code-review/
│   └── code-review.md
└── teach/
    ├── teach.md
    ├── MISSION-FORMAT.md
    └── assets/
        └── style.css

Entry point-ul este fișierul markdown cu același nume ca folderul. Harness-ul îl încarcă când skill-ul este invocat. Toate celelalte fișiere din folder sunt disponibile ca resurse bundled la care entry point-ul poate face referire.

Entry point-ul

Entry point-ul este un fișier markdown cu un bloc de frontmatter YAML urmat de instrucțiunile skill-ului. 

---
name: greet
description: Salută utilizatorul cald și întreabă cu ce are nevoie de ajutor astăzi.
---

Ai fost invocat cu /greet. Rolul tău este să:

1. Îl saluți pe utilizator pe nume dacă îl știi, altfel generic.
2. Pui o singură întrebare clară: care este cel mai important lucru pe care vor să îl realizeze astăzi?
3. Asculți răspunsul, apoi sugerezi un prim pas clar.

Menține răspunsul scurt — sub 100 de cuvinte. Fii cald dar direct.

ARGUMENTS: {{args}}

Placeholder-ul {{args}} este înlocuit cu tot ceea ce utilizatorul a tastat după comandă, de ex. /greet bună dimineața pasează "bună dimineața" ca args.

Câmpuri cheie frontmatter

  • name — numele slash command-ului (fără slash)
  • description — afișat în lista de skills; folosit de harness pentru a decide relevanța

Progressive disclosure

Skills-urile complexe nu ar trebui să reverse toată logica lor în prima interacțiune a utilizatorului. Un design bun de skill folosește progressive disclosure: începe cu instrucțiunea cea mai importantă, apoi dezvăluie detaliile doar când este nevoie.

Practic, asta înseamnă structurarea unui skill ca o serie de etape. Un skill /plan ar putea funcționa astfel: 

---
name: plan
description: Ajută utilizatorul să planifice o implementare înainte de a scrie orice cod.
---

# Etapa 1: Înțelege obiectivul
Întreabă utilizatorul: ce încerci să construiești și de ce?
NU scrie cod sau plan încă. Ascultă și pune întrebări clarificatoare.

# Etapa 2: Identifică constrângerile
Odată ce înțelegi obiectivul, întreabă despre constrângeri:
timp, stack tehnologic, cod existent, dimensiunea echipei.

# Etapa 3: Produce planul
Numai după ce etapele 1 și 2 sunt complete, produce un plan de implementare numerotat.
Fiecare pas trebuie să fie suficient de concret pentru a acționa fără clarificări suplimentare.

Modelul citește toate etapele deodată, dar conversația le desfășoară natural în ordine. Utilizatorul experimentează adâncime progresivă în loc de un zid de instrucțiuni.

Practic: construiește un skill /standup

Exercițiu — Creează și invocă /standup

Construiește un skill care îți pune trei întrebări de standup și formatează răspunsurile ca un mesaj gata pentru Slack.

  1. Creează directorul: 
    mkdir -p ~/.claude/skills/standup
    Pe Windows: New-Item -ItemType Directory ~/.claude/skills/standup
  2. Creează ~/.claude/skills/standup/standup.md: 
    ---
name: standup
description: Rulează standup-ul zilnic și formatează outputul ca mesaj Slack.
---

Rulezi standup-ul zilnic al utilizatorului. Pune aceste trei întrebări una câte una,
așteptând fiecare răspuns înainte de a pune următoarea:

1. Ce ai terminat ieri?
2. La ce lucrezi astăzi?
3. Ai blocanți?

După colectarea tuturor celor trei răspunsuri, formatează-le ca mesaj Slack:

*Daily Standup — {{today's date}}*
✅ Ieri: [răspuns 1]
🔨 Astăzi: [răspuns 2]
🚧 Blocanți: [răspuns 3 — sau "Niciunul" dacă nu sunt]
  3. În Claude Code, tastează /standup. Skill-ul ar trebui să se încarce și să înceapă fluxul cu trei întrebări.
  4. Verifică lista de skills cu /help — noul tău skill ar trebui să apară acolo.
Sfat — Bundling assets
Dacă skill-ul tău are nevoie de fișiere de referință (o schemă JSON, un șablon HTML, o listă de verificare), pune-le în același folder. Referențiază-le în entry point cu numele lor relativ: Consultă [checklist.html](./checklist.html) pentru lista de verificare completă a deployment-ului. Harness-ul rezolvă calea relativ la folderul skill-ului.

Verifică-ți înțelegerea

Întrebarea 1 din 5 Unde trebuie să se afle fișierul entry point și cum trebuie denumit?
Corect. Un folder de skill denumit deploy trebuie să conțină un fișier numit deploy.md. Astfel harness-ul identifică entry point-ul când utilizatorul tastează /deploy.
Întrebarea 2 din 5 Ce primește placeholder-ul {{args}} dintr-un entry point de skill?
Corect. Dacă utilizatorul tastează /deploy production, atunci {{args}} este înlocuit cu "production". Dacă nu se dau argumente, este gol. Folosește-l pentru a face skills-urile context-aware.
Întrebarea 3 din 5 Un skill în .claude/skills/ (la nivel de proiect) față de unul în ~/.claude/skills/ (la nivel de utilizator) — care este diferența cheie?
Corect. Skills-urile la nivel de utilizator te urmăresc în fiecare proiect. Skills-urile la nivel de proiect sunt limitate la acel repo — utile pentru fluxuri specifice proiectului (cum ar fi o listă de verificare pentru deploy) pe care vrei să le comizi alături de cod.
Întrebarea 4 din 5 Progressive disclosure într-un skill înseamnă…
Corect. Progressive disclosure menține conversația gestionabilă. În loc să copleșești utilizatorul cu totul deodată, un skill bine conceput pune întrebări, ascultă și dezvăluie următorul strat de comportament pe baza răspunsurilor.
Întrebarea 5 din 5 Skills-urile sunt invocate automat de harness când sunt relevante. Adevărat sau fals?
Corect — fals. Skills-urile sunt întotdeauna inițiate de utilizator. Harness-ul poate sugera skills-uri când se potrivesc cu sarcina, dar utilizatorul trebuie să tasteze slash command-ul pentru a invoca unul. Aceasta este diferența față de hooks-uri, care se declanșează automat.
