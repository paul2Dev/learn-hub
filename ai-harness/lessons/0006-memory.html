Sistemul de memorie

Oferă lui Claude context persistent între sesiuni — fără să te explici de fiecare dată de la zero.

Problema: fiecare sesiune pornește rece

În mod implicit, fiecare sesiune Claude Code începe fără nicio memorie a conversațiilor anterioare. Îți explici rolul, preferințele și contextul proiectului — iar la sesiunea următoare, o iei de la capăt.

Harness-ul rezolvă asta cu un sistem de memorie bazat pe fișiere: un folder de fișiere markdown pe care harness-ul le încarcă în fereastra de context la începutul fiecărei sesiuni. Modelul le citește și se comportă ca și cum și-ar aminti.

Distincție cheie
Memoria nu este o rechemare magică — este context injectat. Harness-ul citește fișierele tale de memorie și le adaugă la începutul conversației. Cu cât fișierele tale de memorie sunt mai bune, cu atât mai bună este continuitatea lui Claude între sesiuni.

Unde se află memoria

Fișierele de memorie se află într-un folder specific proiectului: 

~/.claude/projects/<project-hash>/memory/
├── MEMORY.md          ← indexul (întotdeauna încărcat primul)
├── user_role.md
├── feedback_testing.md
├── project_auth_rewrite.md
└── reference_linear.md

Fișierul MEMORY.md este special — este un index, întotdeauna încărcat în context. Ar trebui să conțină pointeri de o linie către fiecare alt fișier de memorie. Harness-ul îl folosește pentru a decide ce fișiere complete de memorie să încarce pentru o anumită conversație.

Fișierele individuale de memorie sunt încărcate când conținutul lor este relevant. Menține MEMORY.md sub 200 de linii — liniile de dincolo de aceasta sunt trunchiate.

Cele patru tipuri de memorie

1

user

Cine este utilizatorul: rol, expertiză, preferințe. Folosit pentru personalizarea explicațiilor și recomandărilor. Exemplu: "Inginer Go senior, nou pe frontend-ul React al acestui proiect."

despre tine
2

feedback

Cum ar trebui să se comporte Claude: ce să evite, ce a funcționat bine. Atât corecții CÂT ȘI abordări confirmate. Exemplu: "Nu folosi mock-uri pentru baza de date în teste — divergența față de prod ne-a ars înainte."

reguli de comportament
3

project

Muncă în desfășurare, decizii, termene limită. Se perimează rapid — include De ce și Cum se aplică. Exemplu: "Rescrierea autentificării este motivată de cerințe legale, nu de datorie tehnică. Prioritizează conformitatea față de ergonomie."

context curent
4

reference

Pointeri către sisteme externe. Exemplu: "Bug-uri urmărite în proiectul Linear INGEST."

unde să cauți

Formatul fișierului de memorie

Fiecare fișier de memorie are două părți: frontmatter YAML și un corp markdown.

---
name: feedback-no-trailing-summaries
description: Utilizatorul nu îi plac rezumatele finale la sfârșitul răspunsurilor.
metadata:
  type: feedback
---

Nu rezuma ce ai făcut la sfârșitul răspunsurilor.

**De ce:** Utilizatorul a spus "Pot citi diff-ul" — rezumatele finale adaugă zgomot fără valoare.

**Cum se aplică:** Termină răspunsurile cu ultima modificare sau descoperire semnificativă.
Nu adăuga niciodată un paragraf "Rezumat:" sau "Pe scurt:".

[[user-role]]

Elemente cheie:

  • name — un slug kebab-case, unic în rândul fișierelor tale de memorie
  • description — o linie; folosit pentru a decide relevanța la încărcare
  • metadata.type — unul din: user, feedback, project, reference
  • [[alt-name]] — linkuri către fișiere de memorie conexe după slug-ul lor

Pentru tipurile feedback și project, include întotdeauna De ce: (motivul din spatele regulii) și Cum se aplică: (când se aplică regula). Fără motiv, sesiunile viitoare nu pot judeca cazurile limită.

Indexul MEMORY.md

MEMORY.md este o listă plată — o linie per fișier de memorie. Nu are frontmatter și nicio secțiune dincolo de titlu. 

# Indice de memorie

- [Fără rezumate finale](feedback-no-trailing-summaries.md) — utilizatorul nu îi plac rezumatele de la sfârșitul răspunsurilor
- [Rol utilizator](user-role.md) — inginer Go senior, nou pe partea React a acestui repo
- [Context rescriere autentificare](project-auth-rewrite.md) — motivată de cerințe legale, conformitate peste ergonomie
- [Referință Linear](reference-linear.md) — bug-uri urmărite în proiectul Linear INGEST

Menține intrările sub ~150 de caractere fiecare. Indexul este pentru navigare — conținutul complet este în fișierele individuale.

Ce să NU salvezi

Fișierele de memorie ar trebui să captureze numai lucruri care nu sunt evidente și care nu pot fi derivate din baza de cod:

  • Nu salva: tipare de cod, convenții, structura fișierelor — citește codul
  • Nu salva: istoricul git — folosește git log
  • Nu salva: soluții de debugging — corecția este în cod, contextul este în mesajul de commit
  • Nu salva: detalii ale sarcinii în desfășurare — folosește în schimb o listă de sarcini sau un plan
  • Nu salva: nimic ce există deja în CLAUDE.md

Practic: scrie primele tale fișiere de memorie

Exercițiu — Memorie utilizator
  1. Găsește directorul tău de memorie: 
    ls ~/.claude/projects/
    Alege directorul care corespunde proiectului tău curent (verifică după hash sau caută activitatea recentă).
  2. Creează un fișier user-role.md în memory/: 
    ---
name: user-role
description: Utilizatorul este un developer care învață harness-ul Claude Code.
metadata:
  type: user
---

Utilizatorul este un developer care se familiarizează cu harness-ul Claude Code —
hooks, skills, agents, scheduling, memorie și servere MCP.
Preferă exemple practice în locul teoriei.
Lucrează pe Windows 11 cu Claude Code în VS Code.
  3. Adaugă o linie în MEMORY.md: 
    - [Rol utilizator](user-role.md) — developer care învață harness-ul, preferă exemple practice
  4. Pornește o sesiune nouă și întreabă: "Ce știi despre mine?" Claude ar trebui să descrie rolul tău din fișierul de memorie fără să fie spus.
Exercițiu — Memorie feedback
  1. Gândește-te la un lucru pe care Claude îl face și pe care îl găsești enervant sau nefolositor în sesiuni. Scrie o memorie de feedback pentru asta.
  2. Include linia De ce: — chiar dacă motivul este pur și simplu "preferință personală."
  3. Testează-l în sesiunea următoare.

Verifică-ți înțelegerea

Întrebarea 1 din 5 Ce fișier este întotdeauna încărcat în context la începutul fiecărei sesiuni?
Corect. MEMORY.md este indexul — întotdeauna încărcat. Fișierele individuale de memorie sunt încărcate selectiv când conținutul lor este relevant. De aceea MEMORY.md trebuie să rămână concis: este mereu în bugetul de context.
Întrebarea 2 din 5 Vrei ca Claude să rețină "nu folosi mock-uri în teste." Ce tip de memorie este corect?
Corect. feedback captează cum ar trebui să se comporte Claude — atât corecții ("nu faci X") cât și abordări confirmate ("continuă să faci Y"). project ar fi corect dacă regula era context specific proiectului (de ex., "baza de date de test a acestui proiect este populată cu X"), nu o constrângere comportamentală.
Întrebarea 3 din 5 Ar trebui să salvezi un rezumat al commit-urilor git recente în memorie?
Corect. Fișierele de memorie sunt pentru lucruri care nu pot fi derivate din baza de cod sau istoricul git. Un rezumat de commit în memorie se va perima imediat. Claude poate rula întotdeauna git log pentru a obține istoricul curent. Salvează non-evidentul: decizii, preferințe, constrângeri, pointeri externi.
Întrebarea 4 din 5 O memorie de feedback spune "nu rezuma." De ce ar trebui să incluzi o linie De ce:?
Corect. O regulă fără motiv devine fragilă. Dacă știi de ce ("rezumatele adaugă zgomot, citesc diff-ul"), Claude poate decide corect dacă regula se aplică atunci când o situație este ambiguă — de exemplu, un output lung pe care utilizatorul cu siguranță nu l-a văzut încă.
Întrebarea 5 din 5 Memoria îi oferă lui Claude o rechemare reală între sesiuni. Adevărat sau fals?
Corect — fals. Modelul nu "își amintește" în niciun sens profund. Harness-ul citește fișierele tale de memorie și le injectează în fereastra de context la începutul fiecărei sesiuni. Modelul acționează pe acel context. Este gestionare de stare bazată pe fișiere, nu fine-tuning al modelului. Dacă ștergi un fișier de memorie, cunoașterea dispare.
