Glosar

Termenii cheie folosiți în cadrul cursului Principii de Inginerie. Fiecare termen trimite la lecția în care este introdus în profunzime.

A

Criterii de acceptanță cerințe
Afirmații specifice și testabile care definesc când o funcționalitate este completă. Fiecare criteriu trebuie să fie binar: sistemul fie face asta, fie nu. Instrumentul principal pentru a face cerințele evaluabile.
→ Lecția 6: Ingineria Cerințelor
Control de acces securitate
Mecanismul care impune cine poate face ce. Necesită atât autentificare (identitate) cât și autorizare (permisiune). Categoria #1 OWASP — cel mai frecvent încălcată prin verificarea autentificării dar nu a ownership-ului.
→ Lecția 9: Securitate prin Design
Autentificare securitate
Confirmarea identității — „ești cine spui că ești?" De obicei implementată cu token-uri, sesiuni sau OAuth. Distinctă de autorizare.
→ Lecția 9: Securitate prin Design
Autorizare securitate
Confirmarea permisiunii — „ai voie să faci asta?" Verificată întotdeauna după autentificare. Include verificări de ownership (deține acest utilizator această resursă?).
→ Lecția 9: Securitate prin Design

C

Coeziune sisteme
Cât de strâns legate sunt responsabilitățile dintr-o singură componentă. Coeziunea ridicată înseamnă că componenta are un scop clar, focalizat — tot ce este înăuntru aparține împreună. Pereche cu cuplaj scăzut.
→ Lecția 4: Gândire Sistemică
Compatibilitate inversă api
Garanția că clienții existenți ai unui API continuă să funcționeze după o schimbare. Schimbările aditive (câmpuri opționale noi, endpoint-uri noi) sunt compatibile cu versiunile anterioare. Redenumirea, eliminarea sau schimbarea tipurilor sunt schimbări care sparg compatibilitatea.
→ Lecția 5: Design API
Cuplaj sisteme
Cât de mult trebuie să știe o componentă despre alta. Cuplajul scăzut înseamnă că componentele se pot schimba independent. Cuplajul ridicat înseamnă că o schimbare în A forțează o schimbare în B, care poate forța o schimbare în C.
→ Lecția 4: Gândire Sistemică

D

Apărare în profunzime securitate
Existența mai multor straturi de securitate independente (rețea, aplicație, date, infrastructură) astfel încât un eșec într-unul să nu compromită întregul sistem.
→ Lecția 9: Securitate prin Design
Principiul Inversiunii Dependențelor (DIP) solid
Modulele de nivel înalt și modulele de nivel scăzut ar trebui să depindă amândouă de abstracțiuni (interfețe), nu unele de altele. Permite testabilitatea permițând înlocuirea dependențelor concrete cu test double-uri.
→ Lecția 2: Principiile SOLID
Datorie tehnică mentenanță
Costul implicat al alegerii unei soluții rapide acum în loc de o abordare mai bună care ar dura mai mult. Ca datoria financiară, se compune: plătești „dobândă" — dezvoltare mai lentă, risc mai mare de bug-uri — până când refactorizezi. Datoria deliberată prudentă (documentată, rambursare planificată) este singurul tip acceptabil.
→ Lecția 7: Datorie Tehnică
DRY (Don't Repeat Yourself) simplitate
Fiecare bucată de cunoaștere trebuie să aibă o singură reprezentare neambiguă și autoritativă într-un sistem. Vizează duplicarea cunoașterii, nu duplicarea liniilor. Două funcții cu aspect similar pot reprezenta cunoaștere diferită și ar trebui să rămână separate.
→ Lecția 3: YAGNI, KISS, DRY

E

Enumerare utilizatori securitate
O vulnerabilitate de securitate în care un atacator poate determina dacă un e-mail sau un nume de utilizator dat este înregistrat prin observarea unor răspunsuri diferite la input-uri valide vs. invalide. Prevenită prin returnarea unor răspunsuri identice indiferent dacă input-ul este înregistrat sau nu.
→ Lecția 6: Ingineria Cerințelor

I

IDOR (Insecure Direct Object Reference) securitate
O vulnerabilitate de control al accesului: un endpoint folosește un ID furnizat de utilizator pentru a prelua o resursă fără a verifica că utilizatorul curent deține sau are permisiunea de a o accesa. Utilizatorul autentificat A poate accesa datele utilizatorului B furnizând ID-ul lui B.
→ Lecția 9: Securitate prin Design
Principiul Segregării Interfețelor (ISP) solid
Clienții nu ar trebui să fie forțați să depindă de interfețe pe care nu le folosesc. Împarte interfețele grase în interfețe de rol focalizate. Semnal de încălcare: clase care implementează o interfață dar stub-uiesc mai multe metode ca „not implemented".
→ Lecția 2: Principiile SOLID

K

KISS (Keep It Simple, Stupid) simplitate
Preferă cea mai simplă soluție care rezolvă corect problema. Complexitatea trebuie să se justifice. Întrebarea KISS: „care este cel mai simplu lucru care ar putea funcționa?" Pornește de acolo; adaugă complexitate numai când versiunea simplă nu poate demonstrabil să îndeplinească cerința.
→ Lecția 3: YAGNI, KISS, DRY
Cusătură sisteme
Un loc din sistem unde două componente se ating și informația traversează o limită. Inventat de Michael Feathers. Cusăturile sunt punctele naturale de design: unde definești contracte și unde inserezi test double-uri.
→ Lecția 4: Gândire Sistemică

L

Limită sisteme
Interfața dintre două componente — ce este expus vs. ascuns. O limită bine trasată înseamnă că schimbările în interiorul unei componente nu forțează schimbări ale apelantilor săi, atâta timp cât interfața rămâne stabilă.
→ Lecția 4: Gândire Sistemică
Principiul Substituției Liskov (LSP) solid
Subtipurile trebuie să fie substituibile pentru tipurile lor de bază: oriunde folosești părintele, un copil ar trebui să funcționeze corect. Încălcare: o subclasă care aruncă excepție acolo unde părintele returna, sau returnează un tip incompatibil.
→ Lecția 2: Principiile SOLID

N

Cerință non-funcțională cerințe
O cerință despre cât de bine face un sistem ceva, nu doar ce face. Include performanță, securitate, scalabilitate, fiabilitate. AI omite aproape întotdeauna acestea dacă nu sunt solicitate explicit.
→ Lecția 6: Ingineria Cerințelor

O

Principiul Deschis/Închis (OCP) solid
O clasă ar trebui să fie deschisă pentru extensie dar închisă pentru modificare. Odată ce o clasă este testată și funcțională, comportamentul nou ar trebui să poată fi adăugat fără a edita codul existent. Realizat prin interfețe și injecție de dependențe.
→ Lecția 2: Principiile SOLID
OWASP Top 10 securitate
Cea mai autoritativă listă de riscuri critice de securitate a aplicațiilor web, menținută de Open Web Application Security Project. Categoria #1 din 2021 este Controlul de Acces Defect.
→ Lecția 9: Securitate prin Design

P

Piramida de testare testare
Un model pentru alocarea efortului de testare: multe teste unitare (rapide, focalizate) la bază; teste de integrare moderate la mijloc; puține teste end-to-end la vârf (lente, costisitoare). Forma reflectă unde să investești cel mai mult efort.
→ Lecția 8: Strategia de Testare

R

REST api
Representational State Transfer. Un stil arhitectural pentru API-urile HTTP. Constrângeri cheie: URL-urile denumesc resurse (nu acțiuni); metodele HTTP poartă semantică (GET/POST/PUT/PATCH/DELETE); codurile de stare sunt semantice; fiecare request este stateless.
→ Lecția 5: Design API
Testare bazată pe risc testare
Prioritizarea efortului de testare proporțional cu costul eșecului. Autentificarea, plățile și gestionarea PII sunt critice; textul UI și CRUD-ul simplu sunt prioritate scăzută. Întrebarea este „ce se strică dacă acest cod este greșit?" nu „ce procent din linii sunt acoperite?"
→ Lecția 8: Strategia de Testare

S

Principiul Responsabilității Unice (SRP) solid
O clasă ar trebui să aibă un singur motiv de schimbare — adică să servească un singur grup de stakeholderi. Semnal de încălcare: o clasă sau funcție al cărei nume conține „și" sau „plus".
→ Lecția 2: Principiile SOLID
SOLID design
Cinci principii de design orientat pe obiecte: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion. Împreună ghidează design-ul componentelor spre cod ușor de schimbat fără a-l strica.
→ Lecția 2: Principiile SOLID

Y

YAGNI (You Aren't Gonna Need It) simplitate
Nu implementa ceva până când nu ai cu adevărat nevoie de el. Șterge funcționalitățile speculative. Deosebit de important cu AI, care generează complexitate fără costuri aparente — dar plătești costul complet de mentenanță indiferent dacă speculația se dovedește corectă sau nu.
→ Lecția 3: YAGNI, KISS, DRY
