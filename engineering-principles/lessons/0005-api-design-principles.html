Principii de Design API

Cum să proiectezi API-uri HTTP care să-și onoreze contractul, să supraviețuiască schimbărilor și să fie ușor de folosit corect de clienți.

Un API este o promisiune

Un API nu este doar un set de endpoint-uri — este un contract între sistemul tău și fiecare client care depinde de el. Odată publicat, modificările care sparg compatibilitatea strica clienții. De aceea design-ul API este o disciplină inginerească, nu doar un exercițiu de denumire: design-ul prost te blochează; design-ul bun îți oferă spațiu să evoluezi.

Constrângerea de bază
Orice client care funcționează în prezent cu API-ul tău trebuie să continue să funcționeze după orice actualizare pe care o trimiți. Această constrângere ar trebui să ghideze fiecare decizie de design.

Fundamentele REST (ce contează cu adevărat)

REST este larg înțeles greșit. Aspectele care contează pentru design-ul practic al API:

R

Resurse, nu acțiuni

URL-urile identifică lucruri, nu operații. /orders, nu /createOrder sau /getOrders.

design URL
V

Verbele poartă semantică

Metodele HTTP nu sunt decorative. GET este sigur și idempotent. POST creează. PUT înlocuiește. PATCH actualizează. DELETE elimină.

semantica metodelor
S

Codurile de stare sunt semantice

200 înseamnă succes. 201 înseamnă creat. 400 înseamnă eroare client. 404 înseamnă negăsit. 500 înseamnă eroare server. Nu returna niciodată 200 cu {"error": "not found"} în body.

coduri de stare
S

Lipsa stării (Stateless)

Fiecare request trebuie să conțină toate informațiile necesare pentru a fi procesat. Serverul nu stochează starea sesiunii clientului între request-uri.

stateless

Structura URL

# Colecțiile și elementele urmează un tipar consistent
GET    /orders          → listează toate comenzile
POST   /orders          → creează o comandă nouă
GET    /orders/{id}     → obține o comandă specifică
PUT    /orders/{id}     → înlocuiește o comandă specifică
PATCH  /orders/{id}     → actualizează parțial o comandă specifică
DELETE /orders/{id}     → șterge o comandă specifică

# Resurse imbricate — doar un nivel adânc
GET    /orders/{id}/items    → elementele aparținând unei comenzi
POST   /orders/{id}/items    → adaugă un element la o comandă

# Acțiuni care nu se potrivesc CRUD — folosește sub-resurse
POST   /orders/{id}/cancel   → anulează (nu DELETE — anularea este o tranziție de stare)
POST   /orders/{id}/refund   → inițiază o rambursare

Compatibilitatea inversă: regula non-negociabilă

Adăugarea la un API este sigură. Eliminarea sau modificarea este o schimbare care sparge compatibilitatea. Aceasta este regula pe care fiecare designer de API trebuie să o interiorizeze:

Tip de schimbareSigur?De ce
Adaugă un câmp opțional nou în response ✓ Sigur Clienții existenți ignoră câmpurile necunoscute
Adaugă un parametru de request opțional nou ✓ Sigur Clienții existenți nu îl trimit; este ignorat
Adaugă un endpoint nou ✓ Sigur Clienții existenți nu îl apelează
Redenumește un câmp în response ✗ Sparge compatibilitatea Clienții existenți caută numele vechi al câmpului
Schimbă tipul de date al unui câmp ✗ Sparge compatibilitatea Clienții existenți așteaptă tipul original
Elimină un câmp din response ✗ Sparge compatibilitatea Clienții existenți depind de prezența lui
Face un câmp anterior opțional să devină obligatoriu ✗ Sparge compatibilitatea Clienții existenți nu îl trimit

Versionare

Când trebuie să faci o schimbare care sparge compatibilitatea, versionează API-ul. Convenția practică pentru API-urile web:

# Versionare prin URL — explicită, cacheable, ușor de înțeles
/api/v1/orders
/api/v2/orders

# Menține v1 funcțional în timp ce clienții migrează la v2
# Depreciează v1 cu o dată de sunset în documentație și headerele response
Deprecation: Sat, 01 Jan 2027 00:00:00 GMT
Sunset: Sat, 01 Jan 2027 00:00:00 GMT

Response-uri de eroare

Response-urile de eroare fac parte din contractul API-ului tău. Clienții scriu cod împotriva formatului tău de eroare. Fă-l consistent și ușor de procesat de mașini:

// Format de eroare consistent — RFC 7807 "Problem Details" este un standard bun
{
  "type":    "https://api.yourapp.com/errors/validation-error",
  "title":   "Validare eșuată",
  "status":  422,
  "detail":  "Request-ul nu a putut fi procesat.",
  "errors": {
    "email": ["Câmpul email este obligatoriu."],
    "amount": ["Suma trebuie să fie mai mare decât 0."]
  }
}
// Niciodată: {"success": false, "msg": "input incorect"}

Design API generat de AI: ce să verifici

AI produce frecvent design-uri API cu aceste probleme:

  • URL-uri bazate pe acțiuni (/getUserById în loc de /users/{id})
  • Utilizare inconsistentă a metodelor HTTP (request-uri GET care modifică starea)
  • Response-uri 200 cu body-uri de eroare
  • Nicio strategie de versionare — presupune că API-ul nu se va schimba niciodată
  • Denumire inconsistentă a câmpurilor (userId la un endpoint, user_id la altul)
Exercițiu — Revizuiește un API proiectat de AI
  1. Cere unui AI să proiecteze un REST API pentru o aplicație de blog (postări, comentarii, autori).
  2. Verifică URL-urile: sunt bazate pe resurse sau pe acțiuni?
  3. Verifică metodele HTTP: GET modifică ceva? POST returnează resursa creată?
  4. Verifică codurile de stare: are fiecare response de eroare un cod non-200 corespunzător?
  5. Verifică consistența: convenția de denumire este uniformă în toate endpoint-urile?
  6. Rescrie orice încălcări găsești.
Sursă primară

Microsoft REST API Guidelines. Convenții testate în bătălie de la un producător major de API-uri publice. Acoperă versionarea, denumirea, formatele de eroare, paginarea și mai mult. Folosește-l ca referință, nu ca biblie — aplică ce se potrivește contextului tău.

Quiz

Întrebarea 1 din 5 Ce design de URL urmează convențiile de resurse REST?
URL-urile REST denumesc resurse (substantive), nu acțiuni (verbe). POST /orders spune „creează un lucru nou în colecția de comenzi" folosind verbul HTTP pentru a exprima acțiunea. /createOrder este stilul RPC, nu REST.
Întrebarea 2 din 5 Care dintre aceste schimbări ale API sparge compatibilitatea?
Redenumirea unui câmp sparge compatibilitatea deoarece clienții existenți citesc response.user_id — după redenumire, acesta returnează undefined. Adăugarea câmpurilor opționale, a noilor endpoint-uri sau a noilor parametri opționali sunt toate aditive și compatibile cu versiunile anterioare.
Întrebarea 3 din 5 Ce este greșit la returnarea HTTP 200 cu {"error": "not found"} în body?
Infrastructura HTTP — load balancere, instrumente de monitorizare, logică de retry, logging — citește codurile de stare pentru a distinge succesul de eșec. Returnarea 200 pentru o eroare înseamnă că toată această infrastructură o tratează ca succes, ascunzând eșecul de fiecare strat, cu excepția clientului care parsează body-ul.
Întrebarea 4 din 5 De ce este POST /orders/{id}/cancel mai bun decât DELETE /orders/{id} pentru anularea unei comenzi?
DELETE înseamnă „elimină această resursă." Anularea unei comenzi îi schimbă statusul dar o păstrează în baza de date pentru istoric și conformitate. Modelarea ca POST /orders/{id}/cancel face intenția explicită și păstrează înregistrarea.
Întrebarea 5 din 5 Care este abordarea corectă când o schimbare a API care sparge compatibilitatea este inevitabilă?
Versionarea este soluția standard pentru schimbările inevitabile care sparg compatibilitatea. Rulezi v1 și v2 în paralel, oferi clienților o fereastră de migrare cu o dată de sunset documentată și dezafectezi v1 numai după închiderea ferestrei.
