Code Review în Era AI

Cum să evaluezi sistematic codul generat de AI — și de ce abilitatea de code review este acum una dintre cele mai importante pe care le poate avea un dezvoltator.

Revizuirea este ultima linie de apărare

Într-un flux de lucru tradițional de dezvoltare, autorul codului l-a și proiectat, s-a luptat cu el și îi înțelege compromisurile. În dezvoltarea asistată de AI, autorul este un model de limbaj care nu are nicio înțelegere a sistemului tău, nicio conștientizare a cerințelor tale de securitate și niciun interes în consecințele că a greșit.

Asta face code review cea mai critică activitate inginerească din fluxul tău de lucru — nu o formalitate, ci principala poartă de calitate între output-ul AI și producție.

Ce să cauți: o revizuire stratificată

Code review-ul eficient al codului AI funcționează în straturi — de la imaginea de ansamblu în jos la detalii. Revizuirea detaliilor mai întâi este o capcană: ai putea aproba o funcție bine scrisă care rezolvă problema greșită.

1

Rezolvă problema corectă? Verifică output-ul față de specificația ta. Implementează ce s-a cerut sau o interpretare plauzibilă dar incorectă?

conformitate specificație
2

Design-ul este corect? Verifică limitele, cuplajul, SRP. Introduce o clasă Dumnezeu? Cuplează lucruri care ar trebui să fie independente?

arhitectură
3

Este sigur? Rulează checklist-ul de securitate din Lecția 9. Există verificări de ownership? Validare la limită? Fără risc de SQL injection?

securitate
4

Este YAGNI/KISS/DRY? Identifică complexitatea speculativă, supra-abstracția și duplicarea cunoașterii.

simplitate
5

Sunt gestionate cazurile limită? Ce se întâmplă pe input gol, zero, null, stare invalidă, acces concurent?

cazuri limită
6

Este testabil și testat? Poți injecta dependențe? Testele sunt bazate pe contract sau pe implementare?

Checklist-ul de revizuire AI

Păstrează această listă în fluxul tău de revizuire. Comprimă lecțiile din acest curs într-un checklist practic:

CategorieVerificare
Specificație Implementează exact criteriile de acceptanță?
Control acces Fiecare acces la resursă verifică atât autentificarea cât și ownership-ul?
Validare input Tot input-ul extern validat la limita sistemului?
SRP Fiecare clasă/funcție are un singur motiv de schimbare?
DIP Fără new ConcreteClass() în interiorul logicii de business?
YAGNI Fără abstracțiuni speculative sau configurare nefolosită?
Gestionarea erorilor Ce se întâmplă când dependențele eșuează? Eșecurile gestionate elegant?
Contract API Metode HTTP corecte, coduri de stare, denumire consistentă a câmpurilor?
Teste Teste există pentru cazul fericit și cazurile limită critice?
Datorie Orice scurtături documentate cu un TODO și un plan?

Cum dai feedback AI-ului

Când output-ul AI eșuează o verificare de revizuire, cel mai eficient răspuns nu este să îl corectezi singur — ci să îi spui AI-ului ce principiu a încălcat și să ceri o revizuire. Asta te menține în rolul arhitectural și produce output mai bun decât patch-uirea manuală:

// În loc să îl corectezi singur în tăcere:
// „Controllerul generat are o vulnerabilitate IDOR —
//  preia $order = Order::findOrFail($id) fără a verifica
//  că utilizatorul autentificat deține acea comandă.
//  Te rog revizuiește pentru a adăuga o verificare de ownership sau folosește o policy Laravel."

// În loc să accepți complexitate speculativă:
// „Tiparului factory pe care l-ai adăugat este YAGNI — specificația necesită
//  doar un gateway de plată. Te rog înlocuiește-l cu o clasă directă StripeGateway
//  până când avem cu adevărat nevoie de mai multe gateway-uri."
Numește principiul, nu doar simptomul
Când îi spui AI-ului „asta încalcă Principiul Responsabilității Unice deoarece X," revizia este de obicei mai bună decât când spui „împarte asta în două clase." Numind principiul îi dai AI-ului raționamentul de care are nevoie pentru a face compromisurile corecte în output-ul revizuit.

Construirea intuiției tale de revizuire

Principiile de inginerie devin intuiție revizuibilă în timp. Un inginer senior nu parcurge conștient un checklist la fiecare PR — observă mirosul. Acea intuiție se construiește prin:

  • Practică deliberată: revizuirea codului cu un checklist până când verificările devin automate
  • Post-mortem-uri: când un bug ajunge în producție, urmărește-l înapoi la ce principiu l-a încălcat codul
  • Citirea revizuirilor altora: comentariile bune de code review de la ingineri experimentați sunt o lecție magistrală de principii aplicate
Exercițiu — Simulare completă de revizuire
  1. Alege o funcționalitate generată recent de AI în codebase-ul tău, sau cere AI-ului să genereze una nouă: „Construiește un endpoint Laravel API pentru crearea unei comenzi noi pentru un utilizator autentificat."
  2. Aplică revizuirea stratificată în ordine: specificație → design → securitate → simplitate → cazuri limită → testabilitate.
  3. Pentru fiecare încălcare găsită, scrie feedback-ul pe care l-ai da AI-ului (numește principiul, explică de ce).
  4. Dă feedback-ul AI-ului și compară revizia cu originalul.
  5. Numără: câte verificări au eșuat în prima versiune? Câte în versiunea revizuită?
Sursă primară

Google Engineering Practices: Code Review. Ghidul intern de code review al Google, făcut public. Acoperă ce să cauți, cum să oferi feedback acționabil și cum să echilibrezi temeinicia cu viteza. Scris inițial pentru revizuire umană — aplică aceeași rigoare output-ului AI.

Quiz

Întrebarea 1 din 5 În revizuirea stratificată, ce ar trebui să verifici MAI ÎNTÂI?
Revizuirea detaliilor mai întâi este cea mai frecventă capcană de revizuire. Dacă output-ul rezolvă problema greșită, calitatea implementării sale este irelevantă. Pornește cu conformitatea specificației — implementează asta ce s-a cerut cu adevărat? — înainte de a verifica cât de bine o implementează.
Întrebarea 2 din 5 De ce rolul de code review este mai critic în dezvoltarea asistată de AI decât în dezvoltarea tradițională?
Un autor uman a proiectat codul și îi înțelege compromisurile. AI a produs cod care corespunde unui tipar fără a înțelege cerințele tale specifice de securitate, modelul de date sau constrângerile arhitecturale. Revizuirea este mecanismul principal care acoperă acest decalaj.
Întrebarea 3 din 5 Care este cel mai eficient mod de a gestiona un output AI care are o încălcare a controlului de acces?
Numind principiul („aceasta este o încălcare IDOR — ai nevoie de o verificare de ownership aici deoarece…") produce revizii AI mai bune decât feedback-ul la nivel de simptom („adaugă un where clause pentru ID-ul utilizatorului"). De asemenea te menține în rolul de arhitect în loc să recurgi la patch-uire manuală.
Întrebarea 4 din 5 Care este cel mai bun mod de a construi intuiția de revizuire inginerească în timp?
Intuiția se construiește prin practică deliberată — aplicând principiile conștient până devin automate — și prin post-mortem-uri care conectează bug-urile din producție înapoi la principiul ingineresc pe care l-au încălcat. Acestea creează recunoașterea de tipare pe care inginerii experimentați o numesc „miros".
Întrebarea 5 din 5 Revizuiești cod generat de AI și găsești o clasă factory pentru gateway-uri de plată când este necesar doar un singur gateway. Ce element din checklist prinde asta?
Un factory pentru un singur gateway este o încălcare YAGNI — complexitate speculativă construită pentru o cerință viitoare care nu a fost specificată. Soluția este să înlocuiești factory-ul cu o clasă directă până când ai cu adevărat două gateway-uri care justifică abstracția.
Întreabă profesorul

Ai finalizat toate cele 10 lecții. Întrebări bune de pus în continuare: „Pe ce principiu ar trebui să mă concentrez să îl practic mai întâi?" sau „Poți să-mi dai un PR real de revizuit folosind acest checklist?"
