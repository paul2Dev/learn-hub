YAGNI, KISS, DRY

Trei principii ale simplității care devin și mai importante când AI poate genera complexitate fără costuri aparente.

Capcana supraingineriei cu AI

AI vrea să fie pe plac. Cere-i să construiască o funcționalitate de înregistrare a utilizatorilor și va returna adesea un factory, un repository, un strat de servicii, un event dispatcher și un DTO — chiar dacă aplicația ta este un instrument intern de două pagini. Codul este arhitectural coerent. Este și masiv supraingineriat.

YAGNI, KISS și DRY sunt cele trei întrebări pe care le pui înainte de a accepta orice cod generat — sau orice idee proprie:

YAGNI — You Aren't Gonna Need It

YAGNI spune: nu implementa ceva bazat pe o cerință ipotetică viitoare. Construiește ce cere specificația curentă, și nimic altceva. Funcționalitățile speculative costă de două ori: o dată la construire și din nou la mentenanță (sau la eliminare, când ipoteza se dovedește greșită).

Semnale YAGNI în output-ul AI
Caută: clase de bază abstracte cu o singură subclasă; arhitecturi de plugin pentru o funcționalitate cu o singură implementare; flag-uri de configurare care nu sunt niciodată setate; metode utilitare generice apelate o singură dată. 
// Încălcare YAGNI — sistem de plugin pe care specificația nu l-a cerut
class PaymentProcessorFactory {
    private array $drivers = [];

    public function register(string $name, callable $driver): void {
        $this->drivers[$name] = $driver;
    }
    public function make(string $name): PaymentGateway {
        return ($this->drivers[$name])();
    }
}
// Specificația spune: "suportă Stripe". Atât.
// O clasă directă StripeGateway costă cu 80% mai puțin de citit și întreținut.

Ironia dureroasă: AI construiește sistemul de plugin în câteva secunde, deci pare gratuit. Dar plătești pentru el de fiecare dată când integrezi un nou dezvoltator, depanezi o problemă în producție sau înțelegi ce face sistemul.

KISS — Keep It Simple, Stupid

KISS nu înseamnă „scrie cod banal." Înseamnă: când ai două soluții care rezolvă aceeași problemă la fel de bine, preferă-o pe cea pe care un nou dezvoltator o poate înțelege în cinci minute. Complexitatea este un cost, iar acel cost se compune în timp.

Întrebarea KISS este: care este cel mai simplu lucru care ar putea funcționa? Pornește de acolo. Adaugă complexitate numai când versiunea simplă nu poate demonstrabil să îndeplinească cerința.

DRY — Don't Repeat Yourself

DRY este cel mai înțeles greșit dintre cele trei. Nu este despre eliminarea liniilor de cod duplicate. Este despre eliminarea cunoașterii duplicate. Pragmatic Programmers l-au formulat precis: „Fiecare bucată de cunoaștere trebuie să aibă o singură reprezentare neambiguă și autoritativă într-un sistem."

Duplicarea codului este adesea un simptom. Boala este duplicarea cunoașterii: două locuri care „știu" amândouă aceeași regulă de business, constrângere de validare sau schemă de date. Când regula se schimbă, trebuie să actualizezi ambele — și dacă ratezi una, ai un bug.

// Încălcare DRY — regula de validare duplicată în două locuri
// OrderController.php
$request->validate(['amount' => 'required|numeric|min:0.01|max:10000']);

// RefundController.php
$request->validate(['amount' => 'required|numeric|min:0.01|max:10000']);

// Rezolvare DRY — o singură locație autoritativă pentru regulă
// AmountRule.php
class AmountRule {
    public static function rules(): array {
        return ['required', 'numeric', 'min:0.01', 'max:10000'];
    }
}
// Ambele controllere apelează AmountRule::rules()
Anti-pattern DRY: abstracție prematură
Nu aplica DRY pe cod doar pentru că arată similar. Două lucruri care seamănă dar reprezintă concepte de business diferite ar trebui să rămână separate — cuplarea lor deoarece împart trei linii azi cauzează durere când vor diverge mâine. Așteaptă până ai trei sau mai multe cazuri înainte de a abstrage.

Cum interacționează cele trei principii

Aceste trei principii trag în direcții ușor diferite, și această tensiune este productivă:

  • YAGNI te oprește să construiești lucruri de care nu ai nevoie.
  • KISS te oprește să faci lucrurile necesare mai complexe decât trebuie să fie.
  • DRY te oprește să duplici cunoașterea odată ce ai ales ce să construiești.

Aplicate împreună, produc cel mai mic și mai clar sistem care îndeplinește cerințele actuale — care este și cel mai ușor sistem de extins când cerințele se schimbă cu adevărat.

Exercițiu — Audit al simplității
  1. Deschide orice clasă de serviciu generată recent de AI în proiectul tău.
  2. Evidențiază fiecare linie care există pentru a suporta o cerință viitoare (verificare YAGNI).
  3. Identifică cel mai complex tipar din clasă (lucrul care ar dura cel mai mult să-l explici unui nou angajat). Este complexitatea aceea câștigată? (verificare KISS)
  4. Găsește orice regulă de business duplicată — aceeași validare, formulă sau constantă apărând în două locuri. (verificare DRY)
  5. Șterge sau consolidează ce găsești. Măsoară: câte linii ai eliminat?

Quiz

Întrebarea 1 din 5 Un dezvoltator adaugă un sistem de înregistrare a plugin-urilor „în caz că vom avea nevoie de alte gateway-uri de plată mai târziu." Ce principiu încalcă?
YAGNI este principiul împotriva generalității speculative. Construirea infrastructurii pentru a suporta cerințe viitoare care nu au fost specificate — și care poate nu vor fi niciodată — este o încălcare YAGNI. Plătești costul de mentenanță acum; beneficiul poate să nu ajungă niciodată.
Întrebarea 2 din 5 KISS NU înseamnă care dintre următoarele?
KISS înseamnă selectarea celei mai simple soluții care rezolvă corect problema — nu simplificarea excesivă a codului sau evitarea tuturor tiparelor avansate. Complexitatea este permisă când este cu adevărat necesară și își justifică costul.
Întrebarea 3 din 5 DRY vizează în principal eliminarea duplicării a ce?
DRY vizează duplicarea cunoașterii, nu duplicarea liniilor. Două funcții pot arăta identic dar reprezintă concepte de business diferite și ar trebui să rămână separate. Invers, aceeași regulă de business aplicată în două fișiere este o încălcare DRY chiar dacă codul arată diferit.
Întrebarea 4 din 5 Când este potrivit să abstractizezi două bucăți de cod cu aspect similar?
„Regula celor trei" este o euristică practică DRY: tolerezi duplicarea până la a treia apariție, apoi abstractizezi. Două apariții ar putea fi coincidență; trei sugerează un concept comun genuini care merită extras.
Întrebarea 5 din 5 De ce este YAGNI deosebit de important când lucrezi cu cod generat de AI?
AI generează un sistem de evenimente sau un factory de plugin-uri în câteva secunde — deci complexitatea speculativă pare că nu costă nimic. Dar plătești în continuare costul complet de mentenanță: să-l citești, să-l explici, să-l depanezi și să-l elimini când cerințele se dovedesc diferite față de anticipat.
