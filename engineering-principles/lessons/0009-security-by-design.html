Securitate prin Design

Cum să construiești securitatea de la început mai degrabă decât să o adaugi la final — și de ce codul generat de AI necesită prompting explicit de securitate.

Securitatea nu este o funcționalitate

Cea mai costisitoare greșeală de securitate este tratarea securității ca o funcționalitate de adăugat la final: „vom face sprintul de întărire a securității înainte de lansare." Până atunci design-ul este fixat, suprafețele de atac sunt inglobate și „întărirea" înseamnă acoperirea găurilor dintr-un design care nu ar fi trebuit să le aibă.

Securitatea prin design înseamnă că la fiecare punct de decizie — modelul de date, contractul API, fluxul de autentificare, validarea input-ului — întrebi: cine ar putea abuza de asta și ce s-ar întâmpla dacă ar face-o?

AI și securitatea
AI generează cod care funcționează pentru cazul fericit. Nu modelează utilizatori adversariali. Dacă nu soliciti explicit constrângeri de securitate, AI va omite în tăcere validarea input-ului, verificările de autorizare și limitarea ratei.

Control de Acces Defect: cel mai critic

Controlul de Acces Defect este riscul #1 OWASP și cea mai frecventă omisiune de securitate din codul generat de AI. Când AI generează un controller, autorizează adesea pe baza autentificării („este utilizatorul logat?") dar nu a ownership-ului („deține acest utilizator această resursă?").

// Control de Acces Defect — verifică autentificarea dar nu ownership-ul
class OrderController {
    public function show(Request $request, int $id): JsonResponse {
        $order = Order::findOrFail($id);  // orice utilizator autentificat poate vedea orice comandă!
        return response()->json($order);
    }
}

// Corect — verifică ownership-ul
class OrderController {
    public function show(Request $request, int $id): JsonResponse {
        $order = Order::where('id', $id)
                      ->where('user_id', $request->user()->id)  // verificare ownership
                      ->firstOrFail();
        return response()->json($order);
    }
}

// Și mai bine — folosește o policy
class OrderController {
    public function show(Request $request, Order $order): JsonResponse {
        $this->authorize('view', $order);  // policy impune ownership-ul
        return response()->json($order);
    }
}

Validarea input-ului: nu ai încredere în nimic din exterior

Limita sistemului este locul unde se termină încrederea. Tot ce este în afara aplicației tale — request-uri HTTP, upload-uri de fișiere, payload-uri webhook, răspunsuri API externe — este input neîncrezut. Validează la limită, nu adânc în logica de business.

Tip de inputCe să valideziGreșeală frecventă
Câmpuri de text Tip, lungime, caractere permise, format Bazarea pe validarea frontend în loc de server-side
ID-uri numerice Numeric, pozitiv, există în DB, deținut de apelant Necontrolarea ownership-ului (IDOR — referință directă nesigură la obiecte)
Upload-uri de fișiere Tip MIME (prin conținut, nu extensie), dimensiune, scanare malware Bazarea pe headerul Content-Type de la client
Payload-uri JSON Câmpuri obligatorii, tipuri, intervale de valori, fără chei extra Atribuire în masă (acceptarea tuturor câmpurilor de input orbește)

Apărare în profunzime

Niciun singur strat de securitate nu este suficient. Apărarea în profunzime înseamnă mai multe straturi independente, astfel încât un eșec într-unul să nu compromită sistemul:

1

Stratul de rețea — HTTPS, firewall, protecție DDoS (Cloudflare etc.)

2

Stratul de aplicație — autentificare, autorizare, validare input, limitare rată

3

Stratul de date — query-uri parametrizate, criptare la repaus, utilizator de baze de date cu privilegii minime

4

Stratul de infrastructură — gestionarea secretelor, fără credențiale în cod, principiul privilegiului minim

Checklist de securitate pentru codul generat de AI

Înainte de a accepta orice funcționalitate generată de AI în codebase-ul tău:

  • Verifică fiecare endpoint care accesează o resursă atât autentificarea cât și ownership-ul?
  • Este tot input-ul utilizatorului validat server-side (nu doar client-side)?
  • Sunt toate query-urile de baze de date parametrizate (fără concatenare de șiruri)?
  • Este output-ul sensibil (parole, token-uri, numere complete de card) exclus din response-urile API și log-uri?
  • Există limitare de rată pe endpoint-urile de autentificare și pe orice endpoint care trimite e-mail?
  • Sunt upload-urile de fișiere validate prin conținut, nu prin extensie?
Exercițiu — Audit IDOR

IDOR (Insecure Direct Object Reference) este cel mai frecvent bug de control al accesului.

  1. Listează fiecare endpoint din API-ul tău care ia un ID de resursă ca parametru de cale sau interogare.
  2. Pentru fiecare: codul verifică că utilizatorul autentificat deține sau are permisiunea de a accesa acea resursă?
  3. Testează cele pe care nu ești sigur: autentifică-te ca utilizatorul A, apoi încearcă să accesezi ID-ul resursei utilizatorului B.
  4. Corectează orice lacune adăugând o verificare de ownership sau o policy Laravel.

Quiz

Întrebarea 1 din 5 Care este diferența dintre autentificare și autorizare?
Autentificare = identitate („ești cine spui că ești?"). Autorizare = permisiune („ai voie să faci asta?"). Ambele sunt necesare. Un utilizator logat (autentificat) nu ar trebui să poată citi comenzile altui utilizator dacă nu este și autorizat să facă asta.
Întrebarea 2 din 5 Ce este o vulnerabilitate IDOR?
IDOR (Insecure Direct Object Reference) apare când un endpoint folosește un ID furnizat de utilizator pentru a prelua o resursă fără a verifica că utilizatorul curent deține sau are permisiunea de a accesa acea resursă. Utilizatorul autentificat A poate accesa datele utilizatorului B doar schimbând ID-ul.
Întrebarea 3 din 5 De ce nu ar trebui niciodată să validezi upload-urile de fișiere verificând extensia fișierului?
Extensiile sunt metadate pe care le controlează cel care face upload-ul. Un atacator poate numi un shell PHP shell.jpg. Validează prin citirea conținutului real al fișierului (magic bytes / MIME sniffing) — nu numele furnizat de atacator.
Întrebarea 4 din 5 Ce categorie OWASP Top 10 este clasată #1 ca cel mai critic risc de securitate web?
Controlul de Acces Defect a urcat la #1 în OWASP Top 10 din 2021. Este cel mai răspândit deoarece autentificarea este de obicei gestionată de framework-uri, dar verificările de ownership/permisiune sunt logică de business personalizată care trebuie scrisă — și este ușor de uitat în codul generat de AI.
Întrebarea 5 din 5 Ce înseamnă „apărare în profunzime" în practică?
Apărarea în profunzime înseamnă controale suprapuse: rețea (HTTPS, firewall), aplicație (autentificare, validare input, limitare rată), date (query-uri parametrizate, criptare), infrastructură (gestionarea secretelor, privilegiu minim). Ocolirea unui strat nu dă unui atacator acces la tot.
