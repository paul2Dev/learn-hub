Gândire Sistemică

Cum să descompui un sistem în componente, să controlezi cuplajul și să proiectezi limite care permit fiecărei părți să se schimbe independent.

Ce este un sistem

Un sistem este un set de componente care interacționează pentru a produce comportamente pe care niciuna nu le-ar putea produce singură. Într-o aplicație web, un sistem nu este doar codul tău de aplicație — este baza de date, stratul de cache, workerii de coadă, API-urile externe și oamenii care îl operează. Gândirea sistemică este disciplina de a raționa despre întreg, nu doar despre partea pe care o editezi în prezent.

Vocabularul sistemelor

C

Componentă

O unitate a sistemului cu o limită definită. Poate fi o clasă, un modul, un microserviciu sau un tabel de bază de date.

unitate de design
L

Limită

Interfața dintre componente. Ce este expus vs. ascuns. Decizia cheie de design în orice sistem.

interfață
C

Cuplaj

Cât de mult trebuie să știe o componentă despre alta pentru a-și face treaba. Cuplaj scăzut = mai ușor de schimbat fiecare componentă independent.

forța dependenței
C

Coeziune

Cât de strâns legate sunt responsabilitățile dintr-o singură componentă. Coeziune ridicată = componenta are un scop clar și focalizat.

focalizare internă

Obiectivul în design-ul sistemelor este constant: coeziune ridicată, cuplaj scăzut. Fiecare componentă ar trebui să facă un singur lucru bine (coeziune ridicată) și să știe cât mai puțin posibil despre celelalte componente (cuplaj scăzut).

Cuplajul: dușmanul schimbării

Cuplajul este cel care face ca o schimbare „simplă" să strice ceva neașteptat. Când componenta A trebuie să știe despre interioarele componentei B, o schimbare în B forțează o schimbare în A. Dacă C depinde de A, efectul de undă continuă. Cuplajul ridicat transformă un sistem într-o „minge mare de noroi" în care atingerea oricărei părți este riscantă.

Tipuri de cuplaj, de la cel mai rău la acceptabil:

TipCe înseamnăExemplu
Cuplaj de conținut Componenta A modifică direct interioarele componentei B Scriere directă în proprietatea privată a altei clase prin reflection
Cuplaj comun Mai multe componente împart stare globală mutabilă $_SESSION global sau singleton-uri statice accesate peste tot
Cuplaj de control A transmite un flag lui B indicându-i cum să se comporte intern processOrder($order, $sendEmail = true)
Cuplaj de date Componentele împart doar datele de care au nevoie, prin parametri Transmiterea unui DTO tipat între două servicii — acceptabil

Limite: unde se produce design-ul

O limită este linia de demarcație între ce expune o componentă și ce ascunde. Limitele bune sunt cea mai importantă decizie de design pe care o iei — ele determină câtă parte din sistem trebuie să înțelegi pentru a schimba oricare parte.

Regula limitelor
O limită bine trasată înseamnă: pentru a schimba ce este înăuntru, trebuie să înțelegi doar ce este înăuntru. Schimbările interne nu forțează schimbări în exterior, atâta timp cât limita (interfața) rămâne aceeași.

În aplicațiile web, limitele apar de obicei la:

  • Stratul HTTP — forma request/response este contractul dintre client și server
  • Repository — contractul dintre logica de business și stocare
  • Evenimente de domeniu — contractul dintre acțiunea care are loc și tot ce reacționează la ea
  • Adaptori pentru servicii externe — contractul dintre sistemul tău și API-urile terțe

Descompunerea: cum să desfaci un sistem

Descompunerea este abilitatea de a împărți un sistem în componente la granularitatea potrivită — nu prea grosier (o clasă Dumnezeu), nu prea fin (o clasă per getter). Descompunerea corectă grupează lucrurile care se schimbă împreună și separă lucrurile care se schimbă independent.

1

Identifică vectorii de schimbare. Care părți din acest sistem sunt cel mai probabil să se schimbe? Un gateway de plată se poate schimba; conceptul de bază de „comandă" este stabil. Ține lucrurile volatile izolate.

2

Grupează după stakeholder. Care părți ale sistemului sunt deținute de aceeași echipă sau conduse de aceeași funcție de business? Acelea aparțin aceluiași modul.

3

Denumește cusăturile. O cusătură este un loc unde două părți ale sistemului se ating. La fiecare cusătură, decide: care este informația minimă care trebuie să traverseze această limită?

4

Definește contractul la fiecare cusătură. Scrie interfața înainte de implementare. Interfața este design-ul; implementarea este execuția.

Un exemplu concret: checkout e-commerce

// Descompunere proastă — toată lumea știe despre toată lumea
class CheckoutController {
    public function handle(Request $request): Response {
        // validează input-ul direct aici
        // interoghează baza de date pentru stocul produselor direct aici
        // percepe cardul folosind Stripe SDK direct aici
        // trimite e-mail prin SMTP direct aici
        // scrie logul de audit direct aici
    }
}

// Descompunere bună — fiecare limită este explicită
class CheckoutController {
    public function __construct(
        private CheckoutService $checkout,   // orchestrează cazul de utilizare
        private CartValidator   $validator,  // limita de validare
    ) {}

    public function handle(Request $request): Response {
        $cart = $this->validator->validate($request->all());
        $this->checkout->process($cart);
        return redirect()->route('checkout.confirmed');
    }
}

// CheckoutService cunoaște doar abstracțiunile la limita sa:
class CheckoutService {
    public function __construct(
        private InventoryRepository $inventory,  // limita de stocare
        private PaymentGateway      $payment,    // limita serviciului extern
        private OrderEventDispatcher $events,    // limita evenimentelor
    ) {}
}
Exercițiu — Desenează limitele sistemului tău
  1. Alege o funcționalitate din proiectul tău actual care acoperă mai mult de un fișier.
  2. Listează fiecare componentă implicată (controller, serviciu, repository, API extern etc.).
  3. Pentru fiecare pereche de componente adiacente, notează: ce date traversează limita dintre ele?
  4. Identifică perechea cel mai strâns cuplată: care două componente știu cel mai mult despre interioarele celeilalte?
  5. Schițează cum ar arăta o limită mai curată între ele — ce interfață ar reduce cuplajul?
Sursă primară

BoundedContext — Martin Fowler. Intrarea din bliki-ul lui Fowler despre conceptul Bounded Context din Domain-Driven Design — numele formal pentru ideea „limita între lucrurile care se schimbă împreună" din această lecție. Scurtă, precisă, merită citită de două ori.

Quiz

Întrebarea 1 din 5 Care este obiectivul de design pentru cuplaj și coeziune?
Coeziunea ridicată înseamnă că fiecare componentă are un scop clar, focalizat (tot ce e înăuntru aparține împreună). Cuplajul scăzut înseamnă că componentele știu cât mai puțin unele despre altele. Împreună produc sisteme în care fiecare componentă se poate schimba independent.
Întrebarea 2 din 5 Care tip de cuplaj este cel mai periculos?
Cuplajul de conținut este cel mai periculos deoarece elimină complet limita — o componentă ajunge în interioarele private ale alteia. Orice schimbare internă în a doua componentă o strică imediat pe prima, fără nicio interfață care să absoarbă schimbarea.
Întrebarea 3 din 5 Ce este o „cusătură" în contextul descompunerii sistemelor?
O cusătură este locul unde se întâlnesc două părți ale sistemului. Michael Feathers a inventat termenul în „Working Effectively with Legacy Code." Cusăturile sunt locurile naturale de design în sistem — unde definești contracte și inserezi test double-uri.
Întrebarea 4 din 5 Ce principiu ar trebui să ghideze ce lucruri sunt grupate în aceeași componentă?
„Principiul închiderii comune" (de la Robert Martin) spune: grupează lucrurile care se schimbă împreună. Dacă integrarea gateway-ului de plată și verificarea stocului se schimbă întotdeauna în același timp, ele aparțin împreună. Dacă se schimbă independent, separă-le.
Întrebarea 5 din 5 În exemplul de checkout, de ce este injectarea unui PaymentGateway ca interfață un design mai bun decât apelarea directă a Stripe SDK?
Injectarea interfeței izolează controllerul de știrea ce furnizor de plăți este în uz. Trecerea de la Stripe la alt furnizor, sau substituirea unui fake în teste, necesită zero modificări ale controllerului — limita absoarbe schimbarea.
