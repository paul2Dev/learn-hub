Principiile SOLID

Cinci euristici care ghidează design-ul claselor și modulelor spre cod ușor de modificat fără a-l strica.

De ce contează SOLID în era AI

AI va scrie cod care funcționează. SOLID este checklist-ul tău pentru a întreba: va mai funcționa acest cod când cerințele se schimbă? Cea mai mare parte din durerea de mentenanță vine nu din bug-uri în codul original, ci din schimbări care se propagă neașteptat prin componente strâns cuplate. SOLID este disciplina care previne această propagare.

S — Principiul Responsabilității Unice

S O clasă ar trebui să aibă un singur motiv de schimbare.

„Motiv de schimbare" înseamnă un grup de stakeholderi ale căror cerințe ar putea forța o modificare. O clasă care gestionează logica de business și formatarea e-mailului și persistența în baza de date are trei motive de schimbare — câte unul pentru fiecare preocupare.

Semnal de încălcare: o clasă sau funcție al cărei nume conține „și" sau „plus" (ex. UserRegistrationAndEmailService).

// Încalcă SRP — două responsabilități într-o singură clasă
class OrderService {
    public function processPayment(Order $order): void { ... }
    public function sendConfirmationEmail(Order $order): void { ... }
}

// Respectă SRP — fiecare clasă are un singur motiv de schimbare
class PaymentProcessor {
    public function process(Order $order): void { ... }
}

class OrderMailer {
    public function sendConfirmation(Order $order): void { ... }
}

O — Principiul Deschis/Închis

O Deschis pentru extensie, închis pentru modificare.

Odată ce o clasă este testată și funcțională, ar trebui să poți adăuga comportament nou fără a-i edita codul existent. În practică, asta înseamnă design cu interfețe și injecție de dependențe, astfel încât implementările noi să poată fi introduse fără a atinge originalul.

Semnal de încălcare: un lanț switch sau if/elseif lung care trebuie să crească de fiecare dată când se adaugă un tip nou.

// Încalcă OCP — adăugarea unei noi metode de plată necesită editarea acestei clase
class PaymentProcessor {
    public function process(string $method, float $amount): void {
        if ($method === 'stripe')  { /* logica stripe */ }
        if ($method === 'paypal')  { /* logica paypal */ }
        // Adăugarea 'apple_pay' înseamnă modificarea acestui fișier
    }
}

// Respectă OCP — adaugă o nouă metodă de plată adăugând o clasă nouă
interface PaymentGateway {
    public function charge(float $amount): void;
}
class StripeGateway implements PaymentGateway { ... }
class PaypalGateway implements PaymentGateway { ... }
// Gateway nou = clasă nouă, nicio schimbare în codul existent

L — Principiul Substituției Liskov

L Subtipurile trebuie să fie substituibile pentru tipurile lor de bază.

Dacă clasa B extinde clasa A, oriunde folosești A trebuie să funcționeze corect cu B substituit — același comportament, același contract, nicio surpriză. O ReadOnlyCollection care extinde Collection dar aruncă excepție la add() încalcă LSP.

Semnal de încălcare: o subclasă care suprascrie o metodă pentru a arunca o excepție sau returnează ceva ce părintele nu ar returna niciodată.

I — Principiul Segregării Interfețelor

I Clienții nu ar trebui să depindă de interfețe pe care nu le folosesc.

O interfață grasă care grupează metode necorelate forțează fiecare implementator să furnizeze implementări stub de care nu are nevoie. Împarte interfețele grase în interfețe de rol focalizate, astfel încât fiecare client să depindă doar de ce folosește efectiv.

Semnal de încălcare: clase care implementează o interfață dar lasă mai multe metode ca throw new NotImplementedException().

// Interfață grasă — încalcă ISP
interface Worker {
    public function work(): void;
    public function eat(): void;   // roboții nu mănâncă
    public function sleep(): void; // roboții nu dorm
}

// Segregată — respectă ISP
interface Workable { public function work(): void; }
interface Feedable { public function eat(): void; }
interface Restable { public function sleep(): void; }

class Human implements Workable, Feedable, Restable { ... }
class Robot implements Workable { ... }  // doar ce are nevoie

D — Principiul Inversiunii Dependențelor

D Depinde de abstracțiuni, nu de implementări concrete.

Modulele de nivel înalt (logica de business) nu ar trebui să importe direct modulele de nivel scăzut (drivere de baze de date, clienți HTTP, servicii de email). Ambele ar trebui să depindă de o interfață. Asta face testarea ușoară: înlocuiești baza de date reală cu un test double fără a schimba logica de business.

Semnal de încălcare: cuvântul cheie new în interiorul unei clase de logică de business pentru a instanția o dependență de infrastructură concretă.

// Încalcă DIP — OrderService este legat fix de MySQLOrderRepository
class OrderService {
    private MySQLOrderRepository $repo;

    public function __construct() {
        $this->repo = new MySQLOrderRepository(); // dependență concretă
    }
}

// Respectă DIP — depinde de o interfață, clasa concretă injectată din exterior
class OrderService {
    public function __construct(private OrderRepository $repo) {}
    // OrderRepository este o interfață; injectezi MySQL, SQLite sau fake în teste
}

SOLID în code review

Când revizuiești cod generat de AI, parcurge SOLID ca un checklist. Cele mai frecvente încălcări din output-ul AI sunt SRP (funcții care fac prea mult) și D (dependențe concrete hard-codate în constructori). Fă-ți un obicei să întrebi: ce se întâmplă cu această clasă dacă cerințele se schimbă?

Exercițiu — Audit SOLID pe output AI
  1. Cere unui AI să genereze o clasă UserService care înregistrează utilizatori, trimite e-mailuri de bun venit și loghează acțiunea.
  2. Identifică ce principii SOLID încalcă (vor fi cel puțin două).
  3. Refactorizează — separă responsabilitățile și injectează dependențele — până când satisface toate cele cinci principii.
  4. Notează câte clase ai avut nevoie față de originalul AI.

Quiz

Întrebarea 1 din 5 O clasă care gestionează autentificarea utilizatorilor ȘI trimite e-mailuri de bun venit ȘI loghează evenimente de audit încalcă ce principiu?
SRP spune că o clasă ar trebui să aibă un singur motiv de schimbare — adică un singur grup de stakeholderi care conduce schimbările. Logica de autentificare, șabloanele de e-mail și logarea auditului sunt controlate de echipe diferite cu ritmuri de schimbare diferite, astfel că gruparea lor într-o clasă creează trei motive de schimbare.
Întrebarea 2 din 5 Care tipar de cod este semnalul clasic al unei încălcări Open/Closed?
Un lanț switch/if-elseif în creștere este încălcarea clasică OCP: adăugarea unui caz nou necesită modificarea codului existent, testat, în loc să adaugi o clasă nouă care să se integreze într-o interfață existentă.
Întrebarea 3 din 5 Ce înseamnă pentru un subtip să satisfacă Principiul Substituției Liskov?
LSP este despre substituibilitate comportamentală. Dacă înlocuiești cu un subtip și apelantii se strică — deoarece aruncă excepție acolo unde părintele returna, sau returnează forme diferite — ai o încălcare LSP. Exemplul clasic este o ReadOnlyList care extinde List dar aruncă excepție la add().
Întrebarea 4 din 5 De ce Inversiunea Dependențelor face codul mai ușor de testat?
Când logica de business depinde de o interfață în loc de o clasă concretă, poți injecta un fake (repository în memorie, mailer mock, client HTTP stub) fără a schimba nicio linie din logica de business. Aceasta este câștigul principal de testabilitate al DIP.
Întrebarea 5 din 5 Care două încălcări SOLID sunt cele mai frecvente în codul generat de AI?
AI tinde să producă clase de servicii monolitice care gestionează multe preocupări (SRP) și hard-codează dependențe concrete cu cuvântul cheie new în constructori (DIP). Acestea sunt cele două tipare de verificat primele când revizuiești output-ul AI.
