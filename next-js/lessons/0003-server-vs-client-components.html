Server Components vs Client Components

Cel mai important concept al App Router: unde rulează codul, ce poate face fiecare tip și cum decizi.

Conceptul de bază

În Next.js App Router, toate componentele sunt Server Components implicit. Ele rulează pe server, au acces la baza de date, fișiere sistem, variabile de mediu secrete — și HTML-ul lor ajunge la browser gata generat.

Client Components sunt componente care trebuie să ruleze în browser — pentru că folosesc stare (useState), efecte (useEffect), event handlers sau API-uri disponibile numai în browser (localStorage, window). Le declari cu directiva 'use client' la începutul fișierului.

Regulă de bază
Implicit = Server Component. Adaugi 'use client' numai când componenta are nevoie de browser. Nu adăuga 'use client' preventiv — pierzi avantajele serverului.

Ce poate face fiecare tip

SRV

Server Component — poate face:

async/await direct în componentă, acces la DB/fișiere, variabile secrete, fetch fără CORS, reducere bundle JS

server only
SRV

Server Component — NU poate face:

useState, useEffect, event handlers (onClick), window, localStorage, hooks React

limitări
CLI

Client Component — poate face:

useState, useEffect, onClick, window, localStorage, toate hook-urile React

browser
CLI

Client Component — NU poate face:

Acces direct la DB, variabile secrete (le expui prin API sau Server Actions), async direct fără workaround

limitări

Exemple de cod

Server Component — fetch date direct, fără boilerplate:

// app/products/page.tsx — Server Component (implicit)
// Nu are nevoie de 'use client'

async function getProducts() {
  // Poți apela direct DB, API intern, etc.
  const res = await fetch('https://api.example.com/products', {
    next: { revalidate: 3600 }  // cache ISR: regenerează la o oră
  });
  return res.json();
}

export default async function ProductsPage() {
  const products = await getProducts();  // async/await direct în componentă!

  return (
    <ul>
      {products.map(p => (
        <li key={p.id}>{p.name} — {p.price} RON</li>
      ))}
    </ul>
  );
}

Client Component — interactivitate cu stare:

'use client';  // ← directiva obligatorie, prima linie din fișier

import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Contor: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Incrementează
      </button>
    </div>
  );
}

Granița Server–Client și cum o traversezi

Poți compune Server și Client Components împreună. Regula: datele curg de la server la client prin props. Un Server Component poate importa și folosi un Client Component, dar un Client Component nu poate importa un Server Component. 

// app/dashboard/page.tsx — Server Component
import { InteractiveChart } from './InteractiveChart';  // Client Component

export default async function DashboardPage() {
  // Fetch date pe server — fără a expune cheile API la client
  const data = await getChartData();

  return (
    <div>
      <h1>Dashboard</h1>
      <InteractiveChart data={data} />  {/* datele sunt serializate și trimise */}
    </div>
  );
}

// app/dashboard/InteractiveChart.tsx — Client Component
'use client';
import { useState } from 'react';

export function InteractiveChart({ data }: { data: ChartData[] }) {
  const [activePoint, setActivePoint] = useState(null);
  // ... interactivitate în browser
}
Pattern recomandat
Mută 'use client' cât mai jos în arborele de componente. Dacă numai butonul are nevoie de stare, extrage doar butonul ca Client Component — nu întreaga pagină.

Server Components și securitatea

Un avantaj major al Server Components: codul nu ajunge niciodată în browser. Cheile API, credentials, logica de business sensibilă — rămân pe server. 

// SIGUR: variabila secretă nu apare în bundle-ul clientului
const data = await fetch('https://api.example.com/data', {
  headers: {
    'Authorization': `Bearer ${process.env.SECRET_API_KEY}`
  }
});

// PERICULOS: variabila cu NEXT_PUBLIC_ prefix e vizibilă în browser!
const key = process.env.NEXT_PUBLIC_ANON_KEY;  // ← intenționat public
Variabile de mediu
Orice variabilă cu prefix NEXT_PUBLIC_ este inclusă în bundle-ul de browser — e publică. Fără prefix = disponibilă numai pe server. Niciodată nu pune chei secrete cu prefix NEXT_PUBLIC_.
Exercițiu — Clasifică componentele

Pentru fiecare componentă de mai jos, decide: Server sau Client Component?

  1. Un header cu navbar și un dropdown menu care se deschide la click
  2. O pagină care afișează lista de produse din baza de date
  3. Un formular de căutare cu input și debounce
  4. O componentă care afișează data și ora curentă (statică, fără actualizare)
  5. Un modal care se deschide/închide cu animație

Răspuns: 1→Client (onClick), 2→Server (fetch DB), 3→Client (useState, onChange), 4→Server (data e statică la render), 5→Client (useState pentru deschis/închis).

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Ce tip de componentă este implicit în App Router dacă nu specifici nimic?
Corect. Implicit toate componentele sunt Server Components în App Router. Trebuie să adaugi explicit 'use client' pentru a face o componentă să ruleze în browser.
Întrebarea 2 din 5 Un Server Component vrea să acceseze localStorage. Ce se întâmplă?
Corect. localStorage este o API de browser — nu există în Node.js. Un Server Component care încearcă să acceseze window, localStorage sau document va arunca o eroare. Mută logica în Client Component.
Întrebarea 3 din 5 Care este avantajul principal al Server Components din perspectiva securității?
Corect. Server Components nu trimit niciun cod JavaScript în browser — numai HTML. Cheile API, logica de business, credentials — rămân pe server și nu pot fi inspectate de utilizatori.
Întrebarea 4 din 5 Cum transmiți date dintr-un Server Component într-un Client Component copil?
Corect. Datele curg prin props. Server Component face fetch la date, le pasează ca props Client Component-ului copil. Datele trebuie să fie serializabile (JSON-friendly) — nu poți pasa funcții sau obiecte non-serializabile.
Întrebarea 5 din 5 Care variabilă de mediu este accesibilă și în browser, nu doar pe server?
Corect. Prefixul NEXT_PUBLIC_ face ca variabila să fie inclusă în bundle-ul JavaScript trimis browserului. Oricine poate vedea valoarea în DevTools. Niciodată nu pune chei secrete cu acest prefix.
