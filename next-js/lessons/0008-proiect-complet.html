Ce construim

Un Task Manager cu autentificare — utilizatorii se pot înregistra, autentifica și gestiona propriile task-uri. Fiecare utilizator vede numai task-urile lui. Datele sunt în Supabase, aplicația e deployată pe Vercel.

Pasul 1: Schema Supabase

În Supabase Dashboard → SQL Editor, rulează: 

-- Tabel tasks
CREATE TABLE tasks (
  id          BIGSERIAL PRIMARY KEY,
  user_id     UUID REFERENCES auth.users(id) ON DELETE CASCADE NOT NULL,
  title       TEXT NOT NULL,
  completed   BOOLEAN DEFAULT FALSE,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Activează RLS
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

-- Politică: utilizatorii văd/modifică NUMAI task-urile lor
CREATE POLICY "users_own_tasks" ON tasks
  FOR ALL
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

Pasul 3: Server Actions pentru autentificare

// lib/actions/auth.ts
'use server';

import { createClient } from '@/lib/supabase/server';
import { redirect } from 'next/navigation';
import { revalidatePath } from 'next/cache';

export async function login(formData: FormData) {
  const email    = formData.get('email') as string;
  const password = formData.get('password') as string;
  const supabase = createClient();

  const { error } = await supabase.auth.signInWithPassword({ email, password });
  if (error) return { error: error.message };

  revalidatePath('/', 'layout');
  redirect('/dashboard');
}

export async function register(formData: FormData) {
  const email    = formData.get('email') as string;
  const password = formData.get('password') as string;
  const supabase = createClient();

  const { error } = await supabase.auth.signUp({ email, password });
  if (error) return { error: error.message };

  redirect('/dashboard');
}

export async function logout() {
  const supabase = createClient();
  await supabase.auth.signOut();
  redirect('/login');
}

Pasul 4: Server Actions pentru tasks

// lib/actions/tasks.ts
'use server';

import { createClient } from '@/lib/supabase/server';
import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';

async function getAuthUser() {
  const supabase = createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect('/login');
  return { supabase, user };
}

export async function createTask(formData: FormData) {
  const title = (formData.get('title') as string)?.trim();
  if (!title) return { error: 'Titlul nu poate fi gol.' };

  const { supabase, user } = await getAuthUser();
  const { error } = await supabase
    .from('tasks')
    .insert({ title, user_id: user.id });

  if (error) return { error: error.message };
  revalidatePath('/dashboard');
}

export async function toggleTask(id: number, completed: boolean) {
  const { supabase } = await getAuthUser();
  await supabase.from('tasks').update({ completed }).eq('id', id);
  revalidatePath('/dashboard');
}

export async function deleteTask(id: number) {
  const { supabase } = await getAuthUser();
  await supabase.from('tasks').delete().eq('id', id);
  revalidatePath('/dashboard');
}

Pasul 5: Pagina Dashboard (Server Component)

// app/(app)/dashboard/page.tsx
import { createClient } from '@/lib/supabase/server';
import { redirect } from 'next/navigation';
import { createTask, toggleTask, deleteTask } from '@/lib/actions/tasks';
import { logout } from '@/lib/actions/auth';

export default async function DashboardPage() {
  const supabase = createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect('/login');

  const { data: tasks } = await supabase
    .from('tasks')
    .select('*')
    .order('created_at', { ascending: false });

  const done  = tasks?.filter(t => t.completed).length ?? 0;
  const total = tasks?.length ?? 0;

  return (
    <div style={{ maxWidth: '600px', margin: '0 auto', padding: '2rem' }}>
      <div style={{ display: 'flex', justifyContent: 'space-between', alignItems: 'center' }}>
        <h1>Task-urile mele ({done}/{total} completate)</h1>
        <form action={logout}>
          <button type="submit">Logout</button>
        </form>
      </div>

      {/* Formular adăugare */}
      <form action={createTask} style={{ display: 'flex', gap: '0.5rem', margin: '1.5rem 0' }}>
        <input
          name="title"
          placeholder="Task nou..."
          required
          style={{ flex: 1, padding: '0.5rem 0.8rem', borderRadius: '4px', border: '1px solid #ccc' }}
        />
        <button type="submit">+ Adaugă</button>
      </form>

      {/* Lista task-uri */}
      <ul style={{ listStyle: 'none', padding: 0 }}>
        {tasks?.map(task => (
          <li key={task.id} style={{ display: 'flex', alignItems: 'center', gap: '0.8rem', padding: '0.6rem 0', borderBottom: '1px solid #eee' }}>
            {/* Toggle completat */}
            <form action={toggleTask.bind(null, task.id, !task.completed)}>
              <button type="submit" style={{ fontSize: '1.2rem' }}>
                {task.completed ? '✅' : '⬜'}
              </button>
            </form>
            <span style={{ flex: 1, textDecoration: task.completed ? 'line-through' : 'none', color: task.completed ? '#999' : 'inherit' }}>
              {task.title}
            </span>
            {/* Ștergere */}
            <form action={deleteTask.bind(null, task.id)}>
              <button type="submit" style={{ color: 'red', background: 'none', border: 'none', cursor: 'pointer' }}>✕</button>
            </form>
          </li>
        ))}
      </ul>

      {tasks?.length === 0 && (
        <p style={{ color: '#999', textAlign: 'center', marginTop: '2rem' }}>
          Nicio sarcină. Adaugă prima ta sarcină mai sus!
        </p>
      )}
    </div>
  );
}

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 De ce este importantă această linie în politica RLS: USING (auth.uid() = user_id)?
Corect. USING definește condiția pentru operațiile de citire. Orice SELECT, UPDATE sau DELETE pe acel tabel va fi filtrat automat de PostgreSQL la rândurile unde user_id = auth.uid(). Chiar dacă un bug din cod face un SELECT *, DB-ul returnează numai rândurile utilizatorului autentificat.
Întrebarea 2 din 5 De ce folosim toggleTask.bind(null, task.id, !task.completed) în formular?
Corect. .bind(null, id, value) pre-aplică argumente la o funcție. Aceasta e o tehnică standard pentru Server Actions care primesc argumente dinamice (din lista randată de server) dar nu au câmpuri de formular corespunzătoare.
Întrebarea 3 din 5 În schema SQL, ce face ON DELETE CASCADE pe user_id?
Corect. ON DELETE CASCADE spune PostgreSQL: „când rândul referit (utilizatorul) este șters, șterge și rândurile care îl referențiază (task-urile)". Previne orfanarea datelor — nu rămân task-uri fără utilizator.
Întrebarea 4 din 5 De ce este util Route Group-ul (app) din structura proiectului?
Corect. (app)/layout.tsx poate conține navbar-ul cu butonul de logout — aplicat numai paginilor autentificate. Paginile de (auth)/login și (auth)/register au un alt layout (fără navbar). URL-urile rămân /dashboard și /login, nu /app/dashboard.
Întrebarea 5 din 5 De ce verificăm getUser() în fiecare Server Action, chiar dacă RLS e activ?
Corect. RLS protejează datele la nivel de DB, dar verificarea în Server Action oferă: 1) un mesaj de eroare clar pentru utilizatorul neautentificat, 2) posibilitatea de redirect la login, 3) un strat de securitate suplimentar. „Defense in depth" — nu te baza numai pe un singur mecanism.
