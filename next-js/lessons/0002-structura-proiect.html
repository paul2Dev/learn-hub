Structura unui proiect Next.js

Cum creezi un proiect nou, ce înseamnă fiecare folder și fișier special, și cum funcționează file-based routing în App Router.

Creare proiect nou

Comanda oficială creează un proiect cu toate dependențele configurate: 

npx create-next-app@latest my-app --typescript --tailwind --app --no-src-dir --import-alias "@/*"

cd my-app
npm run dev   # pornești serverul de dezvoltare pe http://localhost:3000

Opțiunile de mai sus: TypeScript activat, Tailwind CSS activat, App Router activat (--app), fără directorul src/ (structura flat), alias de import @/.

Notă Tailwind
Tailwind este opțional. Acest curs nu folosește Tailwind pentru a ține stilurile lizibile — dar în proiecte reale e alegerea implicită și recomandată cu Next.js.

Structura de foldere generată

my-app/
├── app/                    ← App Router — tot routing-ul e aici
│   ├── layout.tsx          ← layout rădăcină (aplat pe toate paginile)
│   ├── page.tsx            ← pagina / (home)
│   ├── globals.css         ← stiluri globale
│   └── favicon.ico
├── public/                 ← fișiere statice (imagini, fonts)
├── next.config.js          ← configurare Next.js
├── tsconfig.json           ← configurare TypeScript
├── package.json
└── .env.local              ← variabile de mediu locale (nu se comite)

File-based routing — cum funcționează

În App Router, structura de foldere definește rutele. Un folder în app/ cu un fișier page.tsx creează o rută accesibilă în browser. 

app/
├── page.tsx              → /
├── about/
│   └── page.tsx          → /about
├── blog/
│   ├── page.tsx          → /blog
│   └── [slug]/
│       └── page.tsx      → /blog/orice-slug
└── dashboard/
    ├── layout.tsx        ← layout aplicat doar la /dashboard/*
    ├── page.tsx          → /dashboard
    └── settings/
        └── page.tsx      → /dashboard/settings
Regulă de bază
Dacă un folder NU are un fișier page.tsx, ruta respectivă nu există. Folderele fără page.tsx pot conține layout.tsx, loading.tsx etc. fără să creeze rute.

Fișiere speciale în App Router

Next.js recunoaște câteva nume de fișiere cu roluri speciale. Trebuie să le cunoști pe de rost:

page

page.tsx

UI-ul paginii pentru acea rută. Singurul fișier care face ruta accesibilă public.

obligatoriu
layout

layout.tsx

UI wrapping aplicat la pagina curentă și toate sub-rutele. Persistent între navigări — nu se remontează.

opțional
load

loading.tsx

UI afișat automat cât timp pagina se încarcă (Suspense boundary automat). Ex: spinner, skeleton.

opțional
error

error.tsx

UI pentru erori în acea rută. Trebuie să fie Client Component ('use client').

opțional
404

not-found.tsx

UI pentru 404. Se activează când dai notFound() din Server Component.

opțional

Layouts nested — cum funcționează

Layouts se compun. Dacă ai layout în app/ și layout în app/dashboard/, cel de la dashboard este wrapat de cel din app/: 

// app/layout.tsx — layout rădăcină (OBLIGATORIU)
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ro">
      <body>
        <header>Nav global</header>
        {children}  {/* pagina sau layout-ul copil intră aici */}
        <footer>Footer global</footer>
      </body>
    </html>
  );
}

// app/dashboard/layout.tsx — layout pentru /dashboard/*
export default function DashboardLayout({ children }: { children: React.ReactNode }) {
  return (
    <div style={{ display: 'flex' }}>
      <aside>Sidebar dashboard</aside>
      <main>{children}</main>
    </div>
  );
}

Rezultat: pagina /dashboard va fi wrapată de ambele layout-uri: mai întâi cel de dashboard, apoi cel rădăcină. Navigarea între pagini din /dashboard/* nu remontează layout-ul de dashboard.

Route Groups — organizare fără a afecta rutele

Poți crea foldere cu paranteze (group) pentru a organiza fișierele fără a adăuga prefix la URL: 

app/
├── (auth)/
│   ├── login/page.tsx     → /login   (nu /auth/login)
│   └── register/page.tsx  → /register
└── (dashboard)/
    └── settings/page.tsx  → /settings

Route Groups sunt utile pentru a aplica layout-uri diferite unor grupuri de pagini fără a le schimba URL-ul.

Exercițiu — Mapare URL → fișier

Scrie calea fișierului page.tsx corespunzătoare fiecărui URL:

  1. /products/123/reviews
  2. /blog
  3. / (home)
  4. /users/[id]/posts

Răspuns: 1→app/products/[id]/reviews/page.tsx, 2→app/blog/page.tsx, 3→app/page.tsx, 4→app/users/[id]/posts/page.tsx.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Ce fișier face o rută accesibilă în browser în App Router?
Corect. Numai page.tsx creează o rută publică. Fișierele layout.tsx, loading.tsx, error.tsx sunt speciale dar nu creează rute accesibile direct.
Întrebarea 2 din 5 Cum creezi o rută dinamică pentru URL-ul /posts/[id]?
Corect. Parantezele pătrate în numele folderului ([id]) marchează un segment dinamic. Valoarea din URL devine disponibilă ca prop params.id în componenta paginii.
Întrebarea 3 din 5 Ce se întâmplă cu un layout când utilizatorul navighează între pagini din același grup?
Corect. Layouts sunt persistente — aceasta este una dintre marile avantaje față de Pages Router. Navigarea între /dashboard și /dashboard/settings nu remontează layout-ul de dashboard, deci starea componentelor din layout supraviețuiește.
Întrebarea 4 din 5 Ce este un Route Group (folder cu paranteze) în App Router?
Corect. Folderul (auth) nu apare în URL. app/(auth)/login/page.tsx creează ruta /login, nu /auth/login. Util pentru organizare și pentru a aplica layout-uri diferite unor grupuri de pagini.
Întrebarea 5 din 5 Care fișier special trebuie să fie obligatoriu un Client Component ('use client')?
Corect. error.tsx trebuie să fie Client Component deoarece React error boundaries funcționează numai pe client. Next.js îți va afișa o eroare dacă uiți să adaugi 'use client' în error.tsx.
