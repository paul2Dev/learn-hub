Deploy pe Vercel

De la repo la producție în 5 minute: conectezi GitHub, configurezi variabilele de mediu și fiecare git push devine un deploy automat.

De ce Vercel pentru Next.js?

Vercel este compania care creează și menține Next.js. Platformele lor sunt construite să lucreze perfect împreună — fără configurare manuală de server, Docker sau Nginx.

  • Zero-config deploy — Vercel detectează automat că e un proiect Next.js
  • Edge Network global — aplicația servită din locația geografică cea mai apropiată de utilizator
  • Preview deployments — fiecare PR primește un URL de preview unic
  • Rollback instant — poți reveni la orice deployment anterior cu un click
  • Tier gratuit generos — suficient pentru proiecte personale și prototipuri

Pasul 1: Pregătire repo GitHub

Vercel se conectează la GitHub (sau GitLab/Bitbucket). Asigură-te că:

  • Proiectul tău Next.js e într-un repo GitHub
  • Fișierul .env.local este în .gitignore (nu se comite niciodată)
  • package.json are script-ul "build": "next build"
# .gitignore (generat automat de create-next-app)
.env.local
.env.*.local
node_modules/
.next/

Pasul 3: Configurare variabile de mediu pe Vercel

Variabilele de mediu din .env.local nu se comit — trebuie adăugate manual în Vercel: 

# Adaugă în Vercel Dashboard → Settings → Environment Variables:
NEXT_PUBLIC_SUPABASE_URL        = https://xyzcompany.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY   = eyJhbGciOi...
SUPABASE_SERVICE_ROLE_KEY       = eyJhbGciOi...
Medii separate
Vercel suportă variabile diferite pentru „Production", „Preview" și „Development". Ideal, ai un proiect Supabase de producție și unul de staging/preview — nu folosi producția pentru teste.

Vercel CLI — deploy din terminal

# Instalare globală
npm install -g vercel

# Login
vercel login

# Deploy din directorul proiectului
vercel                  # preview deploy
vercel --prod           # production deploy

# Pull variabile de mediu din Vercel în local
vercel env pull .env.local

vercel env pull este extrem de util — îți sincronizează variabilele de mediu din Vercel în .env.local fără a le copia manual.

Workflow automat: git push → deploy

Odată conectat, workflow-ul devine: 

# Fiecare push pe branch-ul principal → deploy în producție
git add .
git commit -m "feat: add user dashboard"
git push origin main
# → Vercel primește webhook, pornește build-ul automat

# Fiecare PR → preview deployment cu URL unic
git checkout -b feature/new-feature
git push origin feature/new-feature
# → PR primeeste un comentariu cu URL de preview: https://my-app-git-feature-*.vercel.app

Domeniu personalizat

Din Vercel Dashboard → Settings → Domains, adaugi domeniul tău:

  1. Adaugi domeniul în Vercel (ex: myapp.ro)
  2. Vercel îți dă un DNS record CNAME sau A de adăugat la registrar
  3. Adaugi recordul la providerul DNS (ex: Cloudflare, GoDaddy)
  4. SSL (HTTPS) e configurat automat de Vercel prin Let's Encrypt
Supabase și domenii
Dacă folosești Supabase Auth cu OAuth (Google, GitHub), trebuie să adaugi domeniul de producție în Supabase Dashboard → Authentication → URL Configuration → Site URL și Redirect URLs. Altfel autentificarea va eșua în producție.

Verificare build local înainte de deploy

# Rulează exact ce face Vercel — detectezi erori înainte de deploy
npm run build

# Pornește serverul de producție local pentru testare
npm run start

npm run build poate eșua pe erori TypeScript sau import-uri greșite care nu apar la npm run dev. Rulează-l local înainte de fiecare push important.

Exercițiu — Primul deploy

Deploys proiectul tău Next.js pe Vercel urmând acești pași:

  1. Asigură-te că .env.local e în .gitignore
  2. Rulează npm run build local — verifică că trece fără erori
  3. Push repo-ul pe GitHub
  4. Importă proiectul pe vercel.com
  5. Adaugă toate variabilele de mediu
  6. Deploys și verifică URL-ul de producție
  7. Fă o modificare mică și push — verifică că deploy-ul automat funcționează

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 De ce nu trebuie commituit .env.local în Git?
Corect. .env.local conține chei API secrete, credențiale DB, etc. Dacă ajunge în repo (chiar și privat), riscul de scurgere crește enorm. Variabilele de producție se adaugă manual în dashboard-ul Vercel sau prin CLI.
Întrebarea 2 din 5 Ce se întâmplă automat la fiecare Pull Request pe GitHub conectat la Vercel?
Corect. Fiecare PR primește un deployment de preview la un URL unic (ex: my-app-git-pr-42-user.vercel.app). Poți testa feature-ul în producție înainte de merge. Vercel adaugă automat un comentariu în PR cu link-ul.
Întrebarea 3 din 5 Ce face comanda vercel env pull .env.local?
Corect. Extrem de util când lucrezi în echipă — cineva adaugă o variabilă nouă în Vercel, tu rulezi vercel env pull și o primești local fără copy-paste manual. Sincronizare în ambele sensuri menținută cu ușurință.
Întrebarea 4 din 5 De ce SSL/HTTPS e configurat automat pe Vercel și nu trebuie instalat manual?
Corect. Vercel gestionează totul prin Let's Encrypt: emitere, reînnoire înainte de expirare, configurare Nginx/proxy. Nu trebuie să faci nimic — HTTPS e activ imediat ce domeniul e configurat, inclusiv pentru domenii proprii.
Întrebarea 5 din 5 De ce trebuie adăugat domeniul de producție în Supabase Auth URL Configuration?
Corect. După autentificare, Supabase redirecționează utilizatorul înapoi la aplicație. Din motive de securitate, acceptă numai URL-uri pre-aprobate. Dacă uiți să adaugi domeniul de producție, utilizatorii vor vedea eroare la login — funcționează local dar nu în producție.
