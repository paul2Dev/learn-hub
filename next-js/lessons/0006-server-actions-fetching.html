Server Actions și fetching date

Cum muți mutațiile (create, update, delete) pe server fără a scrie un API REST separat, și cum gestionezi loading states și revalidarea.

Ce sunt Server Actions?

Server Actions sunt funcții async care rulează pe server dar pot fi apelate dintr-un Client Component sau dintr-un formular HTML. Elimină necesitatea de a scrie un API REST separat pentru operații simple de mutație.

Analogie Laravel
Gândește-te la Server Actions ca la Controller methods din Laravel — primesc date din formular, fac operații pe DB, redirecționează sau returnează un răspuns. Dar fără a defini explicit rutele API.

O Server Action se definește cu directiva 'use server' — fie la nivelul funcției, fie la nivelul fișierului: 

// Varianta 1: directivă la nivel de funcție (în Server Components)
export default function Page() {
  async function createTask(formData: FormData) {
    'use server';  // ← această funcție rulează pe server

    const title = formData.get('title') as string;
    const supabase = createClient();
    await supabase.from('tasks').insert({ title });
    revalidatePath('/dashboard');
  }

  return (
    <form action={createTask}>
      <input name="title" required />
      <button type="submit">Adaugă</button>
    </form>
  );
}
// Varianta 2: fișier separat lib/actions.ts (recomandat)
'use server';  // ← toate funcțiile din fișier sunt Server Actions

import { createClient } from '@/lib/supabase/server';
import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';

export async function createTask(formData: FormData) {
  const title = formData.get('title') as string;
  if (!title) return { error: 'Titlul este obligatoriu' };

  const supabase = createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect('/login');

  const { error } = await supabase
    .from('tasks')
    .insert({ title, user_id: user.id });

  if (error) return { error: error.message };

  revalidatePath('/dashboard');  // reîncarcă datele pe pagina /dashboard
}

useFormState și useFormStatus

React 19 + Next.js oferă hook-uri pentru a gestiona starea formularelor cu Server Actions: 

'use client';

import { useActionState } from 'react';
import { useFormStatus } from 'react-dom';
import { createTask } from '@/lib/actions';

function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? 'Se salvează...' : 'Adaugă task'}
    </button>
  );
}

export default function TaskForm() {
  const [state, formAction] = useActionState(createTask, null);

  return (
    <form action={formAction}>
      {state?.error && <p style={{ color: 'red' }}>{state.error}</p>}
      <input name="title" required />
      <SubmitButton />
    </form>
  );
}

Fetching date în Server Components

Cel mai simplu pattern: async/await direct în componentă. Nu ai nevoie de useEffect, loading state manual sau librării de fetching. 

// app/tasks/page.tsx
import { createClient } from '@/lib/supabase/server';

export default async function TasksPage() {
  const supabase = createClient();

  const { data: tasks } = await supabase
    .from('tasks')
    .select('id, title, completed, created_at')
    .order('created_at', { ascending: false });

  return (
    <div>
      <h1>Tasks ({tasks?.length ?? 0})</h1>
      {tasks?.map(task => (
        <div key={task.id}>
          <span>{task.completed ? '✓' : '○'}</span>
          <span>{task.title}</span>
        </div>
      ))}
    </div>
  );
}

Loading states cu Suspense și loading.tsx

Când un Server Component face fetch, Next.js poate afișa un fallback în timp ce se încarcă. Adaugă un fișier loading.tsx în același folder: 

// app/tasks/loading.tsx — afișat automat cât timp page.tsx se încarcă
export default function TasksLoading() {
  return (
    <div>
      {[1, 2, 3].map(i => (
        <div key={i} style={{ height: '40px', background: '#eee', borderRadius: '4px', marginBottom: '8px' }} />
      ))}
    </div>
  );
}

Pentru Suspense granular (mai mult control), poți wrapa componente individuale: 

import { Suspense } from 'react';
import { TaskList } from './TaskList';
import { TaskStats } from './TaskStats';

export default function DashboardPage() {
  return (
    <div>
      <Suspense fallback={<div>Se încarcă statistici...</div>}>
        <TaskStats />  {/* fetch async în interior */}
      </Suspense>

      <Suspense fallback={<div>Se încarcă lista...</div>}>
        <TaskList />   {/* fetch async în interior, paralel cu TaskStats! */}
      </Suspense>
    </div>
  );
}
Fetch paralel automat
Componentele în Suspense separate fac fetch în paralel, nu secvențial. TaskStats și TaskList ambele pornesc fetch la același timp — pagina se încarcă la fel de repede ca cel mai lent dintre ele, nu suma ambelor.

revalidatePath și revalidateTag

Când mutezi date (create/update/delete), trebuie să invalidezi cache-ul: 

import { revalidatePath, revalidateTag } from 'next/cache';

// Revalidare după path — toate paginile din /tasks/* se vor reîncărca
export async function deleteTask(id: number) {
  'use server';
  const supabase = createClient();
  await supabase.from('tasks').delete().eq('id', id);
  revalidatePath('/tasks');
}

// Sau folosind tag-uri — mai granular
// Fetch cu tag:
const data = await fetch('/api/tasks', { next: { tags: ['tasks'] } });

// Invalidare după tag:
export async function updateTask(id: number, data: Partial<Task>) {
  'use server';
  await db.update(tasks).set(data).where(eq(tasks.id, id));
  revalidateTag('tasks');  // invalidează toate fetch-urile cu tag 'tasks'
}
Exercițiu — CRUD complet

Implementează operațiile CRUD pentru un tabel notes (id, content, user_id):

  1. Server Action createNote(formData) — inserează o notă nouă
  2. Server Action deleteNote(id) — șterge o notă după id
  3. Server Action updateNote(id, content) — modifică conținutul
  4. Formular în Client Component care apelează createNote
  5. Buton de ștergere care apelează deleteNote

Asigură-te că fiecare Server Action verifică autentificarea cu supabase.auth.getUser() și apelează revalidatePath după fiecare mutație.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 O Server Action definită cu 'use server' rulează pe:
Corect. 'use server' marchează funcția să ruleze exclusiv pe server. Browserul trimite un request HTTP automat generat — nu vede codul funcției. Aceasta poate accesa DB, fișiere, variabile secrete.
Întrebarea 2 din 5 Ce face revalidatePath('/dashboard') apelat dintr-o Server Action?
Corect. revalidatePath invalidează cache-ul pentru acea rută. La următoarea vizită (sau imediat dacă utilizatorul e pe pagină), Next.js va re-randa Server Components și va face fetch nou la date. Fără revalidatePath, utilizatorul ar vedea date vechi din cache.
Întrebarea 3 din 5 Ce avantaj oferă loading.tsx față de a gestiona manual loading state cu useState?
Corect. loading.tsx creează automat un Suspense boundary în jurul paginii. La navigare, Next.js afișează imediat loading UI-ul din loading.tsx în timp ce Server Component-ul face fetch. Zero boilerplate de useState/isLoading.
Întrebarea 4 din 5 Două componente Server în blocuri Suspense separate fac fetch în:
Corect. Componentele în Suspense separate sunt independente — fiecare pornește fetch imediat la randare, în paralel. Pagina se încarcă în timpul celei mai lente, nu suma tuturor. Aceasta este una dintre marile diferențe față de useEffect secvențial.
Întrebarea 5 din 5 Hook-ul useFormStatus returnează pending: true când:
Corect. useFormStatus urmărește starea formularului părinte. pending: true înseamnă că formularul a fost submituit și Server Action-ul nu a terminat. Util pentru a dezactiva butonul de submit și a afișa un loading indicator în acel interval.
