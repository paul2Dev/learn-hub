Mapping — Schema indexului tău

Tipuri de câmpuri, mappings explicite și de ce contează

Mapping-ul este „schema" indexului OpenSearch — definește ce tip de date are fiecare câmp și cum îl procesează motorul de căutare. E echivalentul unei migrări Laravel, dar pentru OpenSearch.

1. De ce mapping-ul este critic

Dacă nu definești un mapping explicit, OpenSearch face dynamic mapping: ghicește tipul câmpului din primul document indexat. Asta creează probleme:

Probleme frecvente cu dynamic mapping
  • Un câmp status = "1" (string) → devine text, nu poți filtra exact
  • Un câmp pret = "29.99" (string) → nu poți face range query numeric
  • Un câmp titlu → devine text dar și keyword sub-field automat — confuz
  • Odată stabilit, nu poți schimba tipul unui câmp fără să recreezi indexul

Soluție: definești mapping-ul explicit înainte de a indexa primul document.

2. Tipurile de câmpuri esențiale

Tip Folosit pentru Poate fi căutat full-text? Poate fi filtrat exact?
text Conținut lung: titluri, descrieri, articole Da — analizat, tokenizat Nu
keyword Valori exacte: status, email, tag, slug Nu Da — filtru exact, sortare
integer Numere întregi: vârste, cantități, contoare Nu Da — range queries
float Numere zecimale: prețuri, rating-uri Nu Da — range queries
date Date calendaristice: created_at, published_at Nu Da — range, sortare cronologică
boolean Câmpuri true/false: is_published, is_active Nu Da — filtru exact
nested Array de obiecte cu relații interne Da — per obiect Da — per obiect

3. text vs keyword — cea mai importantă distincție

Câmpul "autor" = "Paul Maxineanu"
  • type: text full-text search
    • Analizat și tokenizat → ["paul", "maxineanu"]
    • ✓ match "paul" (case-insensitive)
    • ✓ match "maxineanu"
    • ✗ filtru exact / sortare / agregare
  • type: keyword filtru exact
    • Stocat exact → "Paul Maxineanu"
    • ✓ term "Paul Maxineanu"
    • ✓ sort, agregare (count per autor)
    • ✗ match "paul" (nu e analizat)
Multi-fields — cel mai bun din ambele lumi Poți defini un câmp cu ambele tipuri simultan. Laravel Eloquent face asta automat, dar cu clientul low-level îl definești explicit: 
// Câmpul „autor" disponibil atât ca text cât și ca keyword
'autor' => [
    'type'   => 'text',              // pentru full-text search
    'fields' => [
        'keyword' => [              // sub-câmp pentru filtru exact
            'type'         => 'keyword',
            'ignore_above' => 256,
        ],
    ],
],

// Accesare în query:
// - căutare full-text: câmpul 'autor'
// - filtru exact:      câmpul 'autor.keyword'

4. Mapping complet pentru indexul „articles"

// app/Services/OpenSearchService.php
public function createArticlesIndex(): array
{
    return $this->client->indices()->create([
        'index' => 'articles',
        'body'  => [

            'settings' => [
                'number_of_shards'   => 1,
                'number_of_replicas' => 0,
                'analysis' => [
                    'analyzer' => [
                        'romanian_analyzer' => [
                            'type'      => 'standard',
                            'stopwords' => '_romanian_',
                        ],
                    ],
                ],
            ],

            'mappings' => [
                'properties' => [

                    'titlu' => [
                        'type'     => 'text',
                        'analyzer' => 'romanian_analyzer',
                        'fields'   => [
                            'keyword' => ['type' => 'keyword'],
                        ],
                    ],

                    'continut' => [
                        'type'     => 'text',
                        'analyzer' => 'romanian_analyzer',
                    ],

                    'autor' => [
                        'type'   => 'text',
                        'fields' => [
                            'keyword' => ['type' => 'keyword'],
                        ],
                    ],

                    'slug' => [
                        'type' => 'keyword',  // doar filtru exact, nu full-text
                    ],

                    'status' => [
                        'type' => 'keyword',  // 'draft', 'published', 'archived'
                    ],

                    'este_publicat' => [
                        'type' => 'boolean',
                    ],

                    'vizualizari' => [
                        'type' => 'integer',
                    ],

                    'taguri' => [
                        'type' => 'keyword',  // array de string-uri exacte
                    ],

                    'publicat_la' => [
                        'type'   => 'date',
                        'format' => 'strict_date_optional_time||epoch_millis',
                    ],

                    'mongo_id' => [
                        'type' => 'keyword',  // ID-ul din MongoDB pentru referință
                    ],
                ],
            ],
        ],
    ]);
}

5. Comandă Artisan pentru creare index

Cel mai bun loc pentru crearea indexului este o comandă Artisan, echivalentul unui php artisan migrate: 

// app/Console/Commands/OpenSearchMigrate.php
class OpenSearchMigrate extends Command
{
    protected $signature   = 'opensearch:migrate {--force : Recreează indexul dacă există}';
    protected $description = 'Creează indexurile OpenSearch cu mapping-urile definite';

    public function handle(OpenSearchService $search): int
    {
        if ($search->indexExists('articles')) {
            if !$this->option('force')) {
                $this->warn('Indexul "articles" există deja. Folosește --force pentru recreare.');
                return 0;
            }
            $search->deleteIndex('articles');
            $this->info('Index șters.');
        }

        $search->createArticlesIndex();
        $this->info('✓ Index "articles" creat cu mapping complet.');
        return 0;
    }
}
# Prima dată
php artisan opensearch:migrate

# Dacă vrei să recreezi (ATENȚIE: șterge toate documentele!)
php artisan opensearch:migrate --force
