Lecția 01 · OpenSearch + Laravel

Ce este OpenSearch?

Arhitectura și conceptele fundamentale — de la zero

De ce OpenSearch în proiectul tău?

Proiectul tău are două baze de date cu roluri diferite:

Baza de date Rol Exemple de date
MongoDB Date operaționale — sursa de adevăr Utilizatori, comenzi, produse, tranzacții
OpenSearch Motor de căutare — date indexate pentru search Articole, produse (copie pentru search), logs
Regulă de bază Scrii întotdeauna în MongoDB. OpenSearch primește o copie a datelor pe care vrei să le poți căuta rapid și flexibil. Nu este sursa de adevăr — este un index de căutare.

Ce este OpenSearch?

OpenSearch este un motor de căutare distribuit, open-source, construit peste Apache Lucene. A pornit ca un fork din Elasticsearch (2021) și este menținut de Amazon și comunitate.

Poate fi folosit pentru:

  • Căutare full-text (ca Google, dar în datele tale)
  • Autocomplete și sugestii
  • Filtrare și faceting (filtre de tip e-commerce)
  • Analiză de loguri și metrici
  • Geo-search (căutare după locație)
OpenSearch vs. o bază de date relațională O bază de date SQL face LIKE '%termen%' — lent și inexact. OpenSearch face căutare full-text analizată: înțelege sinonime, variante morfologice, relevanță, typo-uri — și returnează rezultate ordonate după scor.

Arhitectura: Cluster → Node → Index → Shard → Document

Gândește arhitectura OpenSearch ca pe o succesiune de containere imbricate:

CLUSTER
  • NODE 1 server / mașină virtuală
    • INDEX "articles"
      • Shard 0 primar
      • Shard 1 primar
    • INDEX "products"
      • Shard 0 primar
  • NODE 2 server secundar — stochează replici
    • Replica Shard 0 replică
    • Replica Shard 1 replică

Conceptele fundamentale

Document

Un document este unitatea de bază de stocare — echivalentul unui rând din SQL sau al unui document din MongoDB. Datele sunt stocate în format JSON. 

// Un document OpenSearch
{
  "id": "123",
  "titlu": "Introducere în Laravel",
  "continut": "Laravel este un framework PHP...",
  "autor": "Paul",
  "publicat_la": "2024-03-15",
  "taguri": ["php", "laravel", "backend"]
}

Index

Un index este colecția de documente de același tip — echivalentul unui tabel din SQL sau al unei colecții din MongoDB. Exemplu: articles, products, logs.

Atenție la terminologie În OpenSearch, „a indexa un document" înseamnă a-l salva (nu a crea un index de bază de date). Contextul face diferența.

Mapping

Un mapping definește structura unui index — tipurile de date ale fiecărui câmp. Echivalentul unui schema/migrare din Laravel. 

// Mapping pentru indexul "articles"
{
  "mappings": {
    "properties": {
      "titlu":       { "type": "text" },        // câmp searchable, analizat
      "autor":       { "type": "keyword" },     // câmp exact (filtru, sort)
      "publicat_la": { "type": "date" },        // dată
      "vizualizari": { "type": "integer" }     // număr întreg
    }
  }
}
Tip câmp Când îl folosești Exemplu
text Căutare full-text (analizat, tokenizat) titlu, descriere, conținut
keyword Filtre exacte, sortare, agregări status, autor, categorie, tag
date Date calendaristice, range queries created_at, published_at
integer / float Numere, prețuri, scoruri price, rating, views
boolean Câmpuri true/false is_published, is_active

Shard

Un shard este o subdiviziune a unui index. OpenSearch distribuie shardurile pe mai multe noduri pentru a putea procesa căutări în paralel. Ca developer Laravel, nu ai nevoie să gestionezi sharduri manual în stadiul inițial — configurarea implicită (1 shard primar + 1 replică) este suficientă.

Node și Cluster

Un node este o instanță OpenSearch (un proces pe un server). Un cluster este mai multe noduri care lucrează împreună. În dezvoltare locală, lucrezi cu un singur node (cluster de 1).

Fluxul datelor în proiectul tău

POST /articles — scriere articol nou
  • ArticleController::store()
    • MongoDB save() sursa de adevăr — toate câmpurile
    • OpenSearch indexDocument() câmpurile searchable — titlu, conținut, taguri
GET /search?q=laravel — căutare
  • SearchController::index()
    • OpenSearch search() returnează ID-uri ordonate după relevanță
      • Răspuns JSON [ { _id, titlu, _score }, ... ]
Pattern recomandat Când salvezi un articol, salvezi simultan în MongoDB (complet) și în OpenSearch (doar câmpurile relevante pentru search). Când utilizatorul caută, interoghezi doar OpenSearch. Când afișezi detaliile unui articol, poți lua datele din MongoDB (sau din OpenSearch dacă ai indexat complet).
