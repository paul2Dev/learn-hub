Rebase vs. Merge

When to use each — and the one rule that protects your team

Why this lesson matters

Both git rebase and git merge take changes from one branch and incorporate them into another. They produce very different results. The wrong choice can rewrite history that your teammates depend on. By the end of this lesson you will have a simple rule that tells you which to reach for every time — no more guessing.

The starting point

Both operations begin from the same situation: two branches have diverged from a common ancestor. Here main moved to C while feature went to D then E, both branching off commit B.

B is the common ancestor. main has commit C. feature has commits D and E. You are on feature (HEAD).

Option A: git merge

git merge creates a new commit — a merge commit — that joins the two branch tips. It has two parents: the tip of your current branch and the tip of the branch you are merging. Both branches, and all their history, are preserved exactly as they happened.

$ git checkout main    $ git merge feature

Merge commit M (dark) has two parents: C and E. History shows the branching exactly as it happened. feature still points to E — unchanged.

Prefer merge when:

  • Bringing a completed feature into a shared branch (the PR merge button does this)
  • You want history to show exactly when and where branches diverged and rejoined
  • Others are also committing to the target branch

Option B: git rebase

git rebase takes your commits and replays them on top of a new base. Git creates brand-new commits — D′ and E′ — that contain the same code changes as D and E, but with new hashes and a new parent (C instead of B). The result looks as if you had started feature from C all along.

$ git checkout feature    $ git rebase main

D′ and E′ (purple outline) are new commits — same code changes as D and E, but new hashes and a new parent (C). main is unchanged at C. History is now perfectly linear.

Prefer rebase when:

  • Incorporating the latest main changes into your own feature branch
  • You want a clean, linear history that is easy to review in a PR
  • Only you are using the branch — nobody else has pulled it

Side by side

Property git merge git rebase
What it creates A merge commit with two parents New replayed commits (new hashes)
History shape Non-linear — branching is visible Linear — looks sequential
Rewrites history? No — safe on shared branches Yes — unsafe on shared branches
Best used for Integrating completed features via PR Updating your private feature branch

Conflicts during a rebase

Rebase replays commits one at a time, so you may hit a conflict on each replayed commit. The resolution is the same as a merge conflict (edit the file, remove the markers), but the command to continue is different: 

git add app.js             # mark the file as resolved
git rebase --continue      # continue replaying the next commit

You may need to do this more than once if multiple commits in your branch touch the same lines. To abandon the entire rebase and go back to how things were: 

git rebase --abort

The rules

The golden rule

Never rebase commits that others have already pulled.

When you rebase, the original commits (D, E) are replaced by new ones (D′, E′) with different hashes. If a teammate already built on top of the originals, their history breaks when they try to merge or pull. This is the one rule that makes rebase safe.

The decision rule

  • Private feature branch (only you use it): git rebase main to stay current
  • Integrating into a shared branch: git merge — or let the PR do it

Rebase privately. Merge publicly.

How most teams use both

Here is the workflow that puts both commands in the right places. Notice rebase is only ever used on a branch that only you have: 

# 1. Start your feature branch off main
git checkout -b feature/new-login

# 2. Do your work
git commit -m "Add login form"
git commit -m "Add validation"

# 3. Main has new commits. Update your branch cleanly:
git fetch origin
git rebase origin/main         # safe: only you have this branch

# 4. If a conflict appears during rebase:
git add auth.js
git rebase --continue          # not git commit — continue the replay

# 5. Push your clean, rebased branch
git push origin feature/new-login

# 6. Open a Pull Request on GitHub
#    When approved, click Merge — GitHub creates the merge commit

Why rebase in step 3 and not merge? If you used git merge origin/main here, your PR would show unnecessary merge commits that make the review harder to read. Rebasing keeps your PR diff clean — just the commits for your feature, nothing else.
