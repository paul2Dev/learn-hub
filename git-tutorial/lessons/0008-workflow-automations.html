Lesson 08 · Git & GitHub Fluency

Workflow Automations

Small habits and tools that save hours every week

You now have the full mental model: commits, branches, conflicts, rebase, Actions, Webhooks, CI/CD, and PRs. This final lesson is about using all of that faster. Each topic here is something you can set up in 10 minutes and benefit from every day.

⌘

1 — GitHub CLI (gh)

Do GitHub things from your terminal — no browser required

The gh CLI lets you create PRs, review code, check CI status, and manage issues without leaving your terminal. Once you start using it, going back to the browser for these tasks feels slow. 

# Install (Windows)
winget install GitHub.cli

# Authenticate once
gh auth login

# Create a PR from your current branch
gh pr create --title "Add login validation" --body "Fixes #42"

# See PR status + CI checks for current branch
gh pr status

# Open the PR in your browser (when you need the full view)
gh pr view --web

# List open PRs waiting on you to review
gh pr list --search "review-requested:@me"

# Check out someone else's PR locally to test it
gh pr checkout 123

The last command — gh pr checkout — is especially useful during reviews. Instead of pulling manually, one command puts you on their branch.

⚡

2 — Git aliases

Turn long commands into short ones you type every day

Git aliases live in your global ~/.gitconfig. You set them once and they work in every repository forever. 

# Set aliases from the terminal
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.lg "log --oneline --graph --decorate --all"
Type thisRuns thisWhy it helps
git st git status You run this dozens of times per day
git co main git checkout main Shorter than the full command
git lg git log --oneline --graph --decorate --all Visual branch graph — use this to orient yourself
git undo git reset HEAD~1 --soft Undo the last commit but keep your changes staged 
# Add the undo alias
git config --global alias.undo "reset HEAD~1 --soft"
📋

3 — PR & Issue templates

Pre-fill the description box so every PR has the right structure

Create the file .github/PULL_REQUEST_TEMPLATE.md in your repository. GitHub will auto-fill the PR description box with its content every time someone opens a new PR. This enforces a consistent format without reminding anyone. 

## What changed
<!-- One paragraph. What does this PR do? -->

## Why
<!-- Link the ticket: Fixes #123 -->

## How to test
- [ ] Step one
- [ ] Step two

## Screenshots
<!-- For UI changes: before / after -->

For issues, create .github/ISSUE_TEMPLATE/bug_report.md and .github/ISSUE_TEMPLATE/feature_request.md. GitHub shows a choice dialogue when someone clicks "New issue", and fills in the right template.

🏷️

4 — Auto-labeling PRs

Label PRs by which files changed — zero manual work

The actions/labeler action reads a config file and applies labels to PRs based on which paths were changed. Add this once and never manually label another PR. 

# .github/workflows/labeler.yml
name: Label PR
on: [pull_request]
jobs:
  label:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/labeler@v5
        with:
          repo-token: "${{ secrets.GITHUB_TOKEN }}"
# .github/labeler.yml  <-- the rules file
frontend:
  - src/components/**
  - src/styles/**

backend:
  - src/api/**
  - src/models/**

ci:
  - .github/workflows/**

PRs that touch only src/components/ get the "frontend" label automatically. Useful for filtering PRs, routing to the right reviewer, or tracking what kind of work is being merged.

⏰

5 — Scheduled workflows

Run jobs on a timer — no push required

The schedule trigger runs a workflow on a cron schedule, like a server cron job but managed in your repository. 

on:
  schedule:
    - cron: '0 9 * * 1'   # every Monday at 09:00 UTC

Common uses:

  • Dependency audit — run npm audit or pip-audit every night and open an issue if vulnerabilities are found.
  • Stale PR/issue bot — comment on issues that have had no activity for 30 days and close them after 60.
  • Weekly report — query the GitHub API for merged PRs, format a summary, and post it to a Slack channel every Monday morning.
  • Database backup — trigger a backup script on a schedule and upload the result to S3.
📦

6 — git stash

Park unfinished work and come back to it later

You are halfway through a feature when a colleague asks you to review their PR urgently. You cannot commit half-done work. git stash saves your changes to a temporary shelf so your working directory is clean. 

# Save current changes to the stash
git stash

# Now switch branches, review the PR, come back
git checkout main
# ... do whatever you need ...
git checkout feature/my-work

# Bring your stashed changes back
git stash pop

# See all stashes if you have more than one
git stash list

# Give a stash a name so you can find it later
git stash push -m "half-done login refactor"

git stash pop applies the most recent stash and removes it from the list. Use git stash apply if you want to keep it in the list (useful if you want to apply the same changes to multiple branches).
