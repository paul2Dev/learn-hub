GitHub Webhooks

What they are, how they differ from Actions, and when to reach for one

The one-sentence definition

A webhook is a URL on your server that GitHub calls — with an HTTP POST containing event data — whenever something happens in your repository.

Think of it this way: with GitHub Actions, GitHub runs code on its servers. With webhooks, GitHub notifies your server, and your code decides what to do. You provide the phone number; GitHub makes the call.

Actions vs. Webhooks: where the code runs

This is the single most important distinction between the two:

GitHub Actions

event push to main
↓
runs on GitHub's servers
↓
executes your .yml steps
↓
result ✓ / ✗ on commit

No server required. GitHub handles everything.

GitHub Webhooks

event push to main
↓
posts to https://yourapp.com/hook
↓
runs on your server
↓
does whatever you wrote

Requires a server that GitHub can reach over the internet.

What GitHub sends

Every webhook delivery is an HTTP POST to your URL. It includes two important headers and a JSON body describing exactly what happened:

Headers:

X-GitHub-Event: push
X-Hub-Signature-256: sha256=3a7bd3e2...  ← security signature
Content-Type: application/json

Body (the payload) — a push event example:

{
  "ref": "refs/heads/main",              ← which branch was pushed
  "repository": {
    "name": "my-project",
    "full_name": "paulm/my-project"
  },
  "pusher": {
    "name": "paulm"                       ← who pushed
  },
  "commits": [
    {
      "id": "a3f1c29",
      "message": "Add login validation",    ← commit message
      "author": { "name": "Paul" }
    }
  ]
}

Every event type (PR opened, issue created, release published…) has its own payload shape. The full list is in the Webhook Events and Payloads reference.

Security: always verify the signature

When you create a webhook, you set a secret. GitHub uses it to sign every payload and sends the signature in the X-Hub-Signature-256 header. Your server should compute the same HMAC-SHA256 hash and compare — if they do not match, reject the request. This prevents anyone else from sending fake events to your webhook URL. The official docs include code examples for Node, Python, Ruby, and others.

When to use each

Reach for GitHub Actions when… Reach for a Webhook when…
Running tests, linting, or builds on GitHub's servers You already have a server that must react to GitHub events
You don't want to run or maintain any server Integrating GitHub with an external system (Jira, custom Slack bot, analytics DB)
Deploying via the Actions marketplace (deploy to AWS, etc.) Triggering CI on an external server like Jenkins or TeamCity
Automating GitHub-internal tasks (labels, PR comments, releases) Building your own tool that processes GitHub event data in real time

Rule of thumb: if you don't have a server and the task lives entirely inside GitHub, use Actions. If you have a server and need GitHub events to flow into your own systems, use webhooks.

Setting one up: four steps

  1. 1 Go to your repository on GitHub → Settings → Webhooks → Add webhook.
  2. 2 Enter your server's Payload URL (e.g., https://yourapp.com/github-hook), set Content-Type to application/json, and enter a Secret for signature verification.
  3. 3 Choose which events to subscribe to — you can select Just the push event, Send me everything, or pick individual events. Only subscribe to what you actually handle.
  4. 4 Click Add webhook. GitHub immediately sends a ping event to your URL. If your server responds with HTTP 200, the webhook is active. You can inspect all deliveries (and redeliver any of them for debugging) in the Recent Deliveries tab.

Real-world examples

  • 💬
    Custom Slack notifications When a PR is merged, your server receives the event and posts a richly-formatted message to Slack with the author, PR title, and a direct link — formatted exactly the way your team wants, not limited by a Marketplace action.
  • 📋
    Jira / Linear sync When a GitHub issue is opened or closed, your server syncs the state to your project management tool automatically. Keeps both systems in agreement without manual updates.
  • 🏗️
    Trigger an external CI server Your company uses Jenkins or TeamCity. On every push to main, GitHub fires a webhook to the Jenkins API, which starts the build pipeline on your own infrastructure.
  • 📊
    Engineering metrics dashboard Every PR event (opened, reviewed, merged) writes a row to your database. Your team dashboard tracks mean-time-to-merge, review turnaround, and deployment frequency — data that Actions alone cannot produce.
