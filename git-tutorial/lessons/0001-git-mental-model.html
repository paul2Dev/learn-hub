The Git Commit Graph

What Git actually stores — and why it explains everything you feared

Why this lesson first

Every Git situation that feels scary — a merge conflict, a rebase you are not sure about, a "detached HEAD" warning — is scary for the same reason: you cannot see the structure underneath. Once you can, these things become logical operations on a simple graph. This lesson gives you that picture. Everything else in this course builds on top of it.

What a commit really is

A commit is not a "diff" or a list of changes. It is a complete snapshot — a full record of every tracked file at that exact moment. Along with the snapshot, each commit stores four things:

  • A unique identifier — a hash like a3f1c29
  • Author name, email, and a timestamp
  • A commit message
  • A pointer to its parent commit — the commit that came before it

That last item — the parent pointer — is what creates the commit graph: a chain of snapshots linked through time.

Commit B was built on A; commit C was built on B. Each stores a complete snapshot.

Real-world benefit: Because Git stores full snapshots (not diffs), it can instantly show you what any commit looked like — no "replaying" of changes needed. This is also why git checkout <hash> is instantaneous.

What a branch really is

Here is the insight that changes everything: a branch is nothing more than a named pointer to a single commit. That is all. It contains no files, no history — it is a label, 41 bytes on disk, that says "this branch is currently pointing here."

When you run git commit, Git does exactly two things:

  1. Creates a new commit, pointing back to the current commit as its parent.
  2. Moves the branch pointer forward to the new commit.

A branch "moves" because it is a pointer, and updating a pointer is trivially cheap. Creating a new branch in Git takes milliseconds because nothing is copied — a new pointer is just written.

What HEAD is

HEAD is a special pointer that tracks where you currently are in the repository. Usually it points to a branch name, which in turn points to a commit.

When you run git checkout feature, HEAD moves to point at feature. When you then make a commit, feature moves forward to the new commit — and HEAD follows it, because HEAD points to the branch.

HEAD points to main. main points to commit C. You are on the main branch at commit C.

Why this explains what scared you

With the commit graph in mind, two things that felt mysterious become logical:

  • Merge conflicts happen when two branches both changed the same lines of the same file. Git tries to combine two snapshots automatically, but when it cannot decide which change wins, it stops and asks you. It is not losing your work — it is asking for guidance on one specific collision. (Lesson 02 covers this step by step.)
  • Rebase takes the commits on your branch and replays them on top of a different commit — it rewrites the parent pointers to give your work a new starting point. Once you see the graph, it looks exactly like what the word describes. (Lesson 03 covers when and why.)

Nothing is lost. Nothing is random. Git is operating on a graph of pointers, and every command just moves or creates things in that graph.
