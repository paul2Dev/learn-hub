Merge Conflicts

Why they happen, what they look like, and how to fix them in three steps

Why this lesson matters

A merge conflict is not Git breaking. It is Git being honest — it found two versions of the same lines and does not know which one you want. The conflict markers are its way of asking you a question. Once you know the structure, you will never feel stuck in front of one again.

What merging is

From Lesson 01 you know that branches are pointers to commits. Merging is the act of combining the work from two branch tips into a single new commit — a merge commit — that has two parents.

When you run git merge feature while on main, Git looks at exactly three commits:

  • The common ancestor — the last commit both branches share (B below)
  • The tip of main — what your branch currently looks like (C)
  • The tip of feature — what the incoming branch looks like (E)

It compares what each branch changed relative to B, then tries to combine them.

Commits A and B are shared history. After B, main went to C and feature went to D then E. Both branches edited app.js — starting from the same B.

When Git merges automatically

If the two branches changed different files, or different parts of the same file, Git merges silently and creates the merge commit on its own. No action needed. This is the most common outcome — most merges just work.

A conflict only happens when both branches edited the same lines of the same file. Git cannot decide which version wins, so it pauses and marks the collision for you to resolve.

What a conflict looks like

When Git stops, run git status. It will list the conflicted files under "Unmerged paths": 

On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  both modified:   app.js

Open app.js in your editor. Inside it, Git has added conflict markers. Here is what the section looks like — both branches changed the same line:

your branch <<<<<<< HEAD
your version <title>My App — Dashboard</title>
divider =======
incoming <title>Dashboard — My App</title>
end >>>>>>> feature
  • <<<<<<< HEAD → ======= — your version (what main currently has)
  • ======= → >>>>>>> feature — the incoming version (what feature brought in)

Everything outside the markers is already merged successfully. You only need to resolve what is between them.

Resolving it: three steps

  1. Edit the file

    Decide what the final version should be. Delete all three marker lines (<<<<<<<, =======, >>>>>>>) and everything you do not want to keep. You can keep your version, keep the incoming version, or write something new entirely.

    <title>My App — Dashboard</title>

    Save the file. The conflict markers must be completely gone — Git rejects a commit if any marker remains.

  2. Stage the resolved file

    This is how you tell Git "I have resolved this file": 

    git add app.js

    Repeat steps 1 and 2 for every file listed under "Unmerged paths" in git status. Run git status again to confirm all conflicts are resolved — it will say "All conflicts fixed".

  3. Complete the merge

    Run git commit. Git will open your editor with a pre-filled merge commit message like: 

    Merge branch 'feature' into main

    Save and close. The merge commit is created. Your history now contains both lines of work, joined at that commit.

VS Code tip: When you open a conflicted file, VS Code highlights the conflict markers and shows clickable buttons above each section: Accept Current Change, Accept Incoming Change, and Accept Both Changes. Click the one you want instead of editing manually. It removes the markers for you.

If you want to abort the merge

Changed your mind mid-conflict? Run: 

git merge --abort

This puts everything back to how it was before you ran git merge. No changes are lost. You can start over when you are ready.
