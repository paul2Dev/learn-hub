Pull Requests

The human side of the pipeline — proposing, reviewing, and merging code as a team

What a pull request actually is

A pull request (PR) is a GitHub feature — not a Git feature — that wraps a branch in a conversation. It says: "Here is my branch. I want to merge it into main. Please look at my changes, discuss them, and approve."

Git just has branches and commits. GitHub adds the PR layer: a diff viewer, a comment thread, a review system, a CI status display, and a merge button — all in one place. The PR is how teams collaborate on code before it lands in the shared history.

Merge request vs. pull request: GitLab uses "merge request" (MR), GitHub uses "pull request" (PR). They are the same concept. At work you will hear both terms — they are interchangeable.

The PR lifecycle

1
Open the PR

Push your feature branch and open a PR targeting main. Write a clear title and description. Link the ticket or issue it solves.

2
CI runs automatically

The workflow triggers on pull_request. Every push to the branch reruns CI. The PR shows a live status — green means tests pass, red means something broke.

3
Team reviews

Reviewers leave inline comments on specific lines, suggest changes, or ask questions. You respond, push new commits to address feedback, and request re-review.

4
All gates pass

Required reviews are approved. Required CI checks are green. Branch protection unlocks the merge button. The PR is ready.

5
Merge

Someone clicks Merge. The branch lands in main. GitHub automatically closes the PR and (if configured) deletes the feature branch.

Anatomy of a good PR

A PR description is a gift to your reviewer. The better you explain your change, the faster and more useful the review will be.

Open Add password strength validation on signup
paul opened this · 2 commits · Fixes #142
What changed

Added real-time password strength indicator to the signup form. Uses zxcvbn to score strength and shows a coloured bar (red → green) under the input field.

Why

Support raised 14 tickets last month for compromised accounts. This reduces the chance users pick weak passwords without blocking them from signing up.

How to test
  • Go to /signup and type a short password → bar should be red
  • Type a longer mixed password → bar turns green
  • Existing unit tests: npm test -- signup
Notes for reviewer

Deliberately not blocking weak passwords — product decision. Kept the zxcvbn bundle to 12 kB by importing only the core function.

✓ CI / test (ubuntu-latest) — passed
✓ CI / build (ubuntu-latest) — passed
✓ 1 approving review — anna approved

The four sections that matter most: what changed, why, how to test, and any notes for the reviewer about non-obvious decisions. Skip the sections that don't apply — a one-line fix doesn't need all four.

Draft PRs

Open a PR as a Draft when your work is not ready for review but you want feedback on direction, or you want CI to run on it, or you want teammates to know what you are working on. GitHub marks it with a grey "Draft" badge and disables the merge button until you click Ready for review.

Common pattern: open a Draft PR on day one of a feature. Reviewers can leave early comments. You push more commits as you build. When done, mark it ready and request formal review. This prevents the "surprise large PR" problem.

The three merge strategies

When you click the merge button, GitHub gives you three options. Understanding them matters because they produce different histories — and once merged, you cannot change it.

Strategy What it produces When to use
Merge commit All original commits + one merge commit with two parents When you want full history preserved. Each PR is a visible bubble in the graph.
Squash and merge most common All PR commits collapsed into one commit on main Messy WIP commits ("fix typo", "fix fix") become one clean commit. Keeps main linear and readable.
Rebase and merge PR commits replayed on top of main — no merge commit When each commit on the PR is meaningful and you want linear history without squashing.

Most teams pick one strategy and stick to it so main's history is consistent. Squash and merge is the most popular choice because it keeps main clean without requiring developers to curate their WIP commits.

Review etiquette

As the PR author

  • Keep PRs small — one logical change per PR. Big PRs take longer and get shallower reviews.
  • Respond to every comment, even just "done" or "good catch".
  • Don't take feedback personally. The reviewer is reviewing code, not you.
  • If you disagree, explain why — don't silently ignore the comment.
  • Add a comment explaining non-obvious decisions before someone asks.

As a reviewer

  • Distinguish blocking issues from suggestions — use "nit:" for minor style points.
  • Ask questions instead of making demands: "What if X happens?" not "Fix X."
  • Acknowledge what the author did well. A review is not just a list of problems.
  • Approve promptly when satisfied — don't leave a PR waiting for days.
  • Suggest specific code changes using GitHub's "Suggest a change" feature.

How PRs connect to CI and branch protection

Everything from the last two lessons comes together here:

  • The CI workflow (Lesson 06) triggers on pull_request — every new commit to the PR branch reruns all checks.
  • Branch protection rules (Lesson 06) require those checks to be green and require at least one approving review before the merge button activates.
  • Webhooks (Lesson 05) fire on PR events — useful for syncing to Jira, posting a Slack notification, or triggering an external preview deployment.

The PR is the integration point where all the automation you've learned about comes together in a single human-readable interface.
