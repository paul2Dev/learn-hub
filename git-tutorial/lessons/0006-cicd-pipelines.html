CI/CD Pipelines

How code travels from your laptop to production — automatically and safely

What CI/CD means

CI — Continuous Integration means every developer's code is merged into a shared branch frequently, and an automated pipeline immediately verifies it (runs tests, checks formatting, looks for security issues). The goal: catch problems within minutes, not days.

CD — Continuous Delivery / Deployment means that once code passes CI, it is automatically packaged and either made ready to deploy (Delivery) or actually deployed to production without manual steps (Deployment).

The distinction that matters day-to-day: CI is about quality gates (nothing broken merges). CD is about shipping speed (verified code reaches users fast). You can have CI without CD, but CD without CI is dangerous.

A real CI workflow

This is what a team's CI workflow actually looks like. It runs on every push and on every pull request targeting main: 

name: CI  [1]

on:  [2]
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:  [3]
    runs-on: ubuntu-latest  [4]

    steps:
      - uses: actions/checkout@v4  [5]

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci  [6]

      - name: Lint
        run: npm run lint

      - name: Test
        run: npm test  [7]

  build:  [8]
    needs: test  # only runs if test job passes
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
  1. 1 Workflow name — shown in the GitHub Actions tab. Pick something descriptive.
  2. 2 Triggers — runs on push to main AND on any PR targeting main. This means the PR is tested before it merges, not after.
  3. 3 Job name — test. Jobs run in parallel by default. The build job below uses needs: test to force sequential order.
  4. 4 Runner — a fresh Ubuntu VM for each run. Nothing carries over between runs; every run starts clean.
  5. 5 Checkout action — clones your repository into the runner so subsequent steps can access your code.
  6. 6 npm ci installs exact versions from package-lock.json — faster and deterministic. Always prefer it over npm install in CI.
  7. 7 Test step — if this exits with a non-zero code, the job fails, the PR gets a red X, and merge is blocked (when branch protection is on).
  8. 8 Build job — only runs after test succeeds. This is job chaining: fail fast, don't waste build minutes on code that doesn't pass tests.

Branch protection: the gate that enforces CI

A workflow file alone does not prevent broken code from merging. Branch protection rules are what actually block the merge button until CI passes. You configure them in GitHub → repository → Settings → Branches → Add branch protection rule.

Developer opens PR
  • Pushes feature branch
  • Opens pull request to main
  • CI workflow triggers automatically
→
Branch protection gates
  • Require status check: CI (test job)
  • Require at least 1 approving review
  • No merge until all checks pass

The key setting is "Require status checks to pass before merging" — you select which workflow jobs must be green. Once this is on, GitHub physically disables the merge button if any required check is failing.

Environments: staging and production

Most teams deploy to multiple environments before shipping to users. GitHub supports this natively via Environments (Settings → Environments), which let you add required reviewers and secrets per environment:

Environment Triggered by Typical gate
development Push to any branch None — deploy fast, test freely
staging Merge to main CI must pass
production Manual trigger or tag push Required reviewer approves in GitHub UI

In your workflow file you reference an environment with environment: production on a job. GitHub then pauses that job and waits for a human to approve it in the UI before the deployment continues — a built-in manual gate with a full audit trail.

What "a green pipeline" means at work

When a colleague says "the pipeline is green", they mean every automated check passed — tests, lint, build, security scan. It is a signal that the code is safe to merge or deploy.

When someone says "the pipeline is red" or "CI is broken", fixing it is treated as the highest priority. A red pipeline blocks the whole team from merging, so convention is: whoever broke it fixes it immediately.

At your next PR: look at the checks section at the bottom of the PR page. Each row is one job from the workflow file. Click Details next to a failing check to see the exact step and log line that failed. This is how you debug CI in practice.
