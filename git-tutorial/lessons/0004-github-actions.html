GitHub Actions

What it is, how it works, and a real workflow you can use today

Why Actions exist

Every team has repetitive tasks that run after every commit: run the tests, check code style, build the project, deploy to staging. Before automation, someone had to remember to do these manually — or a broken commit would slip into the main branch.

GitHub Actions is the automation platform built directly into GitHub. You describe what you want to happen, and when — in a YAML file that lives inside your repository. GitHub runs it for you on its own servers, free for public repos and up to 2,000 minutes per month on private ones.

The vocabulary

Six terms cover everything you need to read and write a workflow:

workflow

The whole automated process. One YAML file = one workflow. A repo can have many workflows.

trigger (on:)

The event that starts the workflow — a push, a pull request, a schedule, or a manual click.

job

A group of steps that run together on the same machine. Jobs in one workflow can run in parallel or in sequence.

step

A single task inside a job. Either a shell command (run:) or a pre-built action (uses:).

action

A reusable, shareable step from the GitHub Marketplace. actions/checkout clones your repo. Thousands exist.

runner

The virtual machine that runs a job. GitHub provides Ubuntu, Windows, and macOS runners for free.

Where the files live

GitHub only reads workflow files from one place — the .github/workflows/ directory in your repository. You can have as many .yml files there as you like, one per workflow. 

my-project/
├── .github/
│   └── workflows/
│       ├── ci.yml          ← runs tests on every push
│       └── deploy.yml      ← deploys when main is updated
├── src/
└── package.json

Create the directory, add a file, push — GitHub picks it up immediately. No registration or setup required.

Anatomy of a real workflow

Here is a complete CI workflow that runs your test suite on every push and every pull request targeting main. Read the numbered comments, then check the explanations below. 

name: Run Tests                        # [1]

on:                                          # [2]
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:                                        # [3]
    runs-on: ubuntu-latest                   # [4]
    steps:
      - uses: actions/checkout@v4              # [5]
      - uses: actions/setup-node@v4            # [6]
        with:
          node-version: '20'
      - run: npm install                       # [7]
      - run: npm test                           # [8]
  1. 1 name — the display name shown in GitHub's Actions tab. Choose something descriptive.
  2. 2 on — the triggers. This workflow fires on a push to main and whenever a pull request targeting main is opened or updated.
  3. 3 test — the name of this job. You could name it anything. A workflow can have multiple jobs (e.g., test and lint running in parallel).
  4. 4 runs-on: ubuntu-latest — the virtual machine type. GitHub spins up a fresh Ubuntu VM for every run. It is discarded when the job finishes.
  5. 5 actions/checkout@v4 — a pre-built action that clones your repository into the runner. Without this, the VM has no access to your code. @v4 pins the version.
  6. 6 actions/setup-node@v4 — installs Node.js on the runner. The with: block passes configuration — here, the Node version.
  7. 7 run: npm install — a plain shell command. Runs in the repository directory. Any shell command works here.
  8. 8 run: npm test — runs your test suite. If this command exits with a non-zero code, the step fails, the job fails, and GitHub shows a red ✗ on the commit and blocks the PR from merging.

What happens when you push

1 You push a commit (or open a PR). GitHub detects the push or pull_request event.
2 GitHub reads .github/workflows/ci.yml and matches the event against the on: block.
3 A fresh virtual machine (runner) is provisioned. Your steps run in order.
4 If every step exits cleanly: green ✓ on the commit. If any step fails: red ✗. On a PR, GitHub can block merging until the check passes.
5 The VM is thrown away. The next run starts completely fresh.

Common triggers

Trigger When it fires Real-world use
push A commit is pushed to a branch Run tests on every commit to main
pull_request A PR is opened, updated, or merged Block merging until tests pass
schedule A cron schedule (e.g., every Monday at 9am) Nightly builds, weekly dependency audits
workflow_dispatch Manual — you click "Run workflow" in GitHub Deploy to staging on demand
release A GitHub Release is created Build binaries and attach to the release

What teams actually use this for

  • ✅
    Continuous Integration (CI) Run tests and linting on every push and PR. The most common use. Broken code cannot silently reach main.
  • 🚀
    Continuous Deployment (CD) When a PR merges to main, automatically deploy to production or staging. No manual deploy step.
  • 📦
    Publish packages When you create a Release on GitHub, automatically build and publish to npm, PyPI, or Docker Hub.
  • 💬
    Notifications Post a Slack message when a deploy succeeds or a build breaks. Keep the whole team informed without checking GitHub.
  • 🏷️
    Auto-labeling PRs Automatically add labels (e.g., "frontend", "breaking change") based on which files changed.
