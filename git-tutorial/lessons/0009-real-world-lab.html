Real-World Lab

Apply every lesson on your own repository — step by step

This lab walks you through all eight topics on your own GitHub repository: paul2Dev/git-tests. Each step references the lesson that taught the concept, so you can go back and re-read if anything feels unclear. By the end, your repository will have a working CI pipeline, branch protection, a PR template, and all the daily productivity tools in place.

8 labs — work through them in order

1

Setup — clone, configure, orient yourself

Covers: Lesson 01 · Lesson 08

Clone the repository and immediately add the aliases you will use throughout this lab.

# Clone your repo
git clone git@github.com:paul2Dev/git-tests.git
cd git-tests

# Orient yourself — see the current history graph
git log --oneline --graph --decorate --all

# Add the aliases from Lesson 08 (run these once, they are global)
git config --global alias.st status
git config --global alias.lg "log --oneline --graph --decorate --all"
git config --global alias.undo "reset HEAD~1 --soft"

# Verify your remote is set correctly
git remote -v
What to expect After git lg you should see your branch graph. After git remote -v you should see two lines — fetch and push — both pointing to git@github.com:paul2Dev/git-tests.git.
  • Repository cloned successfully
  • git lg alias works
  • Remote points to paul2Dev/git-tests
2

Merge conflict — create one and resolve it

Covers: Lesson 02

You will deliberately create a merge conflict by editing the same line on two different branches, then resolve it. This is the scenario you used to be afraid of — let's make it routine. 

# Make sure you are on main and it is clean
git checkout main
git pull

# Create a file to work with
echo "Hello from main" > greeting.txt
git add greeting.txt
git commit -m "add greeting.txt"

# Create a feature branch and change the same line
git checkout -b feature/greeting
echo "Hello from feature branch" > greeting.txt
git add greeting.txt
git commit -m "update greeting on feature branch"

# Go back to main and make a DIFFERENT change to the same line
git checkout main
echo "Hello from main — updated" > greeting.txt
git add greeting.txt
git commit -m "update greeting on main"

# Now try to merge — this will conflict
git merge feature/greeting
You will see this message CONFLICT (content): Merge conflict in greeting.txt — Automatic merge failed; fix conflicts and then commit the result. This is expected. The conflict is working as intended.

Open greeting.txt — you will see conflict markers:

<<<<<<< HEAD
Hello from main — updated
=======
Hello from feature branch
>>>>>>> feature/greeting

Edit the file to the version you actually want — for example:

Hello from main — updated and feature merged
# After editing the file, mark it resolved and complete the merge
git add greeting.txt
git commit -m "merge feature/greeting — resolve greeting conflict"

# Verify the history now shows a merge commit
git lg
What to expect git lg should show a branch diverging and then converging — two lines meeting at a merge commit (the dark node with two parents you saw in Lesson 03's diagram).
  • Conflict created and markers visible in file
  • Conflict resolved by editing the file manually
  • git lg shows the merge commit with two parents
3

Rebase — keep your branch up to date, linearly

Covers: Lesson 03

The real-world scenario: you are on a feature branch, main has moved forward, and you want to update your branch without a merge commit. 

# Start a new feature branch from current main
git checkout main
git checkout -b feature/about-page

# Make a commit on your feature branch
echo "About this project" > about.txt
git add about.txt
git commit -m "add about.txt"

# Simulate main moving forward (another developer pushed something)
git checkout main
echo "new file from colleague" > colleague-work.txt
git add colleague-work.txt
git commit -m "colleague adds colleague-work.txt"

# Back on your feature branch — rebase onto the updated main
git checkout feature/about-page
git rebase main
# See the result — notice your commit is now AFTER main's new commit
git lg
What to expect The graph should be completely linear — your feature/about-page commit sits on top of main's latest commit, with no merge bubble. Your commit has a new hash (it was replayed) but the same message and changes.
  • Feature branch rebased onto updated main
  • History is linear — no merge commit visible
  • about.txt and colleague-work.txt both exist in your branch

GitHub Actions — add a real CI workflow

Covers: Lesson 04 · Lesson 06

You will create the .github/workflows/ directory and add a CI workflow. Since git-tests may not have a specific test framework, we will use a shell check that always passes — the goal is to see the workflow run and produce a green check on GitHub. 

# Go back to main first
git checkout main

# Create the workflows directory
mkdir -p .github/workflows

Create the file .github/workflows/ci.yml with this content:

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Verify repository files
        run: |
          echo "Files in repo:"
          ls -la
          echo "CI passed!"
# Commit and push the workflow
git add .github/workflows/ci.yml
git commit -m "add CI workflow"
git push origin main
What to expect on GitHub Go to github.com/paul2Dev/git-tests → Actions tab. You should see a workflow run triggered by your push. Click it — you should see the check job with a green checkmark and the file list printed in the logs.
  • .github/workflows/ci.yml committed and pushed
  • Workflow run visible in the Actions tab
  • check job shows green

PR Template — every PR gets the right structure

Covers: Lesson 07 · Lesson 08

Create the PR template file so every new pull request is pre-filled with the right sections.

Create .github/PULL_REQUEST_TEMPLATE.md:

## What changed
<!-- One paragraph describing what this PR does -->

## Why
<!-- Link the issue: Fixes #123 -->

## How to test
- [ ] Step one
- [ ] Step two

## Notes for reviewer
<!-- Non-obvious decisions, trade-offs, or things to watch out for -->
# Commit the template
git add .github/PULL_REQUEST_TEMPLATE.md
git commit -m "add PR template"
git push origin main

Now open a pull request on GitHub to test it. Create a new branch, make any small change, push it, and open a PR — the description box should be pre-filled with your template. 

# Create a test branch to open a PR with
git checkout -b feature/test-pr-template
echo "testing PR template" > pr-test.txt
git add pr-test.txt
git commit -m "test: verify PR template appears"
git push origin feature/test-pr-template
What to expect Go to github.com/paul2Dev/git-tests → Pull Requests → New pull request, choose feature/test-pr-template as the source. The description box should already contain your template text.
  • PULL_REQUEST_TEMPLATE.md committed and pushed
  • Opened a test PR and saw the template pre-filled
  • CI workflow ran automatically on the PR

Branch protection — enforce CI before any merge

Covers: Lesson 06

This step is done entirely on GitHub — there is no command to run. You are configuring a gate that makes CI mandatory for every PR targeting main.

Steps on GitHub:

  1. Go to github.com/paul2Dev/git-tests → Settings → Branches
  2. Click Add branch protection rule
  3. In "Branch name pattern" type main
  4. Check Require status checks to pass before merging
  5. In the search box that appears, search for and select check (your CI job name)
  6. Check Require branches to be up to date before merging
  7. Optionally check Require a pull request before merging to prevent direct pushes to main
  8. Click Create at the bottom
One-person repo note If you are the only contributor and you also enable "Require pull request reviews", you will need to approve your own PRs — GitHub allows this for repo owners. You can leave the review requirement off for solo repos and only require CI.
  • Branch protection rule created for main
  • check job selected as required status check
  • Merge button on your test PR is now gated behind CI

GitHub CLI — manage PRs from the terminal

Covers: Lesson 08

If you have not installed the GitHub CLI yet, do it now. Then authenticate against your account and use it to interact with the PR you created in Lab 5.

# See the detailed status of your open test PR (CI checks included)

# Create a new PR from the current branch (if you are on one)
# gh pr create --title "your title" --body "Fixes #1"

What to expect gh pr status shows your open PRs with their CI check results inline — green or red — without opening a browser. gh pr merge --squash merges the PR using squash strategy and deletes the feature branch automatically.

  • gh pr status shows CI check results
  • PR merged from the terminal using gh pr merge

Auto-labeler — label PRs by what files changed

Covers: Lesson 08 · Lesson 04

Add the auto-labeler. This is a two-file setup: a workflow that runs the action, and a config file that defines the label rules.

First, create the labels on GitHub — go to github.com/paul2Dev/git-tests → Issues → Labels → New label and create these three: docs, ci, config.

Create .github/workflows/labeler.yml:

name: Label PR
on: [pull_request_target]
jobs:
  label:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/labeler@v5
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}

Create .github/labeler.yml:

docs:
  - changed-files:
    - any-glob-to-any-file: ["*.md", "*.txt"]

ci:
  - changed-files:
    - any-glob-to-any-file: [".github/workflows/**"]

config:
  - changed-files:
    - any-glob-to-any-file: [".github/**"]
# Commit both files and push
git add .github/workflows/labeler.yml .github/labeler.yml
git commit -m "add auto-labeler workflow"
git push origin main
What to expect The next time you open a PR that touches .md or .txt files, GitHub will automatically apply the docs label. PRs touching .github/workflows/ get the ci label.
  • Labels docs, ci, config created on GitHub
  • Labeler workflow and config committed and pushed
  • Opened a test PR touching .txt — saw docs label applied automatically

What your repository now has

  • .github/workflows/ci.yml — CI runs on every push and PR
  • .github/workflows/labeler.yml — PRs get labelled automatically
  • .github/labeler.yml — labeling rules
  • .github/PULL_REQUEST_TEMPLATE.md — every PR is pre-structured
  • Branch protection on main — CI must pass before merging
  • Git aliases globally configured — git st, git lg, git undo
  • GitHub CLI authenticated — create and merge PRs from the terminal

This is a real, production-grade setup. Larger teams add more steps to the CI workflow (linting, security scans, coverage gates), more labeler rules, and multiple environments for deployment — but the structure is exactly what you have now. You built it yourself, on your own repository, from scratch.
