Vector Databases: stochează și caută vectori la scară

De ce nu poți folosi un loop simplu în producție și cum rezolvă vector databases problema nearest neighbor.

Problema: O(n) nu scalează

În lecțiile anterioare am calculat similaritatea față de fiecare document din corpus. Asta funcționează pentru 100 documente. La 1 milion de documente:

Comparație timp de răspuns
1K documente ~2ms ✓ OK
100K documente ~200ms ~ acceptabil
1M documente ~2 secunde ✗ prea lent
100M documente ~200 secunde ✗ imposibil

O vector database rezolvă asta cu algoritmi approximate nearest neighbor (ANN): returnează top-K cei mai apropiați vecini în sub-milisecundă, chiar și la sute de milioane de vectori. Compromisul: nu garantează precizia 100% — poate rata 1–2% din cele mai bune rezultate. În practică, diferența este imperceptibilă.

ChromaDB în Python — exemplu complet

# pip install chromadb openai
import chromadb
from openai import OpenAI

openai_client = OpenAI()
chroma_client = chromadb.Client()  # in-memory, sau chromadb.PersistentClient("./chroma_db")

# Creează o colecție (echivalentul unui tabel)
collection = chroma_client.create_collection("faq")

# Documentele tale
documents = [
    "Cum resetez parola contului meu?",
    "Ghid de instalare a aplicației mobile.",
    "Politica de retur și refund pentru comenzi online.",
    "Cum schimb adresa de livrare după plasarea comenzii?",
    "Suport pentru plăți cu cardul în aplicație.",
]

# Calculează embeddings
def get_embedding(text):
    res = openai_client.embeddings.create(input=text, model="text-embedding-3-small")
    return res.data[0].embedding

embeddings = [get_embedding(doc) for doc in documents]

# Adaugă în ChromaDB
collection.add(
    documents=documents,
    embeddings=embeddings,
    ids=[f"doc_{i}" for i in range(len(documents))],
)

# Search
query = "vreau să returnez un produs cumpărat online"
query_emb = get_embedding(query)

results = collection.query(
    query_embeddings=[query_emb],
    n_results=3,
)

print("Rezultate pentru:", query)
for doc, dist in zip(results["documents"][0], results["distances"][0]):
    print(f"  [distanță: {dist:.4f}] {doc}")
Distance vs Similarity în ChromaDB
ChromaDB returnează distances (nu similarities) — cu metrica L2 (euclidiană) implicit. Valori mai mici = mai similar. Poți schimba la cosine la crearea colecției: chroma_client.create_collection("faq", metadata={"hnsw:space": "cosine"})

pgvector — PostgreSQL cu suport vectori

Dacă folosești deja PostgreSQL (Laravel cu Postgres sau orice altă aplicație), poți stoca vectorii direct: 

-- Activează extensia
CREATE EXTENSION IF NOT EXISTS vector;

-- Tabel cu embeddings
CREATE TABLE documents (
    id          SERIAL PRIMARY KEY,
    content     TEXT NOT NULL,
    embedding   vector(1536)  -- dimensiunile modelului OpenAI
);

-- Index pentru căutare rapidă (HNSW)
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops);

-- Inserare
INSERT INTO documents (content, embedding)
VALUES ('Politica de retur...', '[0.231, -0.847, ...]'::vector);

-- Semantic search — cele mai similare 5 documente
SELECT content, 1 - (embedding <=> '[0.123, ...]'::vector) AS similarity
FROM documents
ORDER BY embedding <=> '[0.123, ...]'::vector
LIMIT 5;

Operatorul <=> calculează distanța cosinus. Sortezi ASC pentru cele mai apropiate. Formula 1 - distanță convertește la similarity (0–1).

Metadata filtering — un feature esențial

Vector databases permit să combini căutarea semantică cu filtre structurate. De exemplu: „găsește documentele similare semantice, dar DOAR din categoria 'facturare' și cu data > 2024": 

# ChromaDB cu metadata
collection.add(
    documents=documents,
    embeddings=embeddings,
    ids=[f"doc_{i}" for i in range(len(documents))],
    metadatas=[
        {"category": "cont", "lang": "ro"},
        {"category": "tehnic", "lang": "ro"},
        {"category": "comenzi", "lang": "ro"},
        {"category": "comenzi", "lang": "ro"},
        {"category": "plati", "lang": "ro"},
    ]
)

# Search cu filtru
results = collection.query(
    query_embeddings=[query_emb],
    n_results=3,
    where={"category": "comenzi"},  # doar documente din categoria comenzi
)

Asta este esențial în aplicații reale: un user caută ceva, dar vrei să returnezi rezultate doar din departamentul lui, sau din ultimele 30 de zile, sau cu un anumit status.

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 De ce un loop simplu de cosine similarity nu scalează la 1 milion de documente?
Corect. Fiecare query compară vectorul cu FIECARE din cele 1M documente. La 1M comparații × 1536 operații floating-point = lent. ANN (HNSW, IVF) reduce la O(log n) sau sub-liniar.
Întrebarea 2 din 5 Ce compromis face Approximate Nearest Neighbor (ANN) față de căutarea exactă?
Corect. ANN este „aproximativ" — poate rata 1–5% din cele mai bune rezultate. În schimb, este de sute de ori mai rapid. În practică (semantic search, RAG), precizia de 95%+ este mai mult decât suficientă.
Întrebarea 3 din 5 Ce întoarce ChromaDB în câmpul distances la un query?
Corect. ChromaDB returnează distanțe, nu similarități. Valori mai mici = mai similar. Pentru cosine similarity, convertești cu 1 - distance. Asigură-te că colecția folosește metrica cosine dacă vrei valori între 0–1.
Întrebarea 4 din 5 Care vector database este cel mai ușor de integrat dacă folosești deja PostgreSQL?
Corect. pgvector este o extensie PostgreSQL — o activezi cu CREATE EXTENSION vector și stochezi vectorii în coloane de tip vector(n). Nu adaugi niciun serviciu nou. Ideal pentru echipe care vor să minimizeze complexitatea infrastructurii.
Întrebarea 5 din 5 La ce folosești metadata filtering într-o vector database?
Corect. Metadata filtering permite query-uri de forma „similar semantic cu X, dar DOAR din categoria Y și cu data > Z". Asta este esențial în aplicații reale: nu returnezi tot din baza de date, ci doar din contextul relevant al userului.
