RAG — Retrieval Augmented Generation

Cum construiești un chatbot care răspunde pe baza documentelor tale, nu din antrenamentul LLM-ului.

Problema pe care o rezolvă RAG

Un LLM (Claude, GPT-4) știe enorm de multe lucruri din antrenament — dar nu știe nimic despre:

  • Documentele interne ale companiei tale
  • Datele actualizate după data de training (knowledge cutoff)
  • Informații specifice domeniului/produsului tău
  • Baza ta de date de clienți, comenzi, produse

RAG (Retrieval Augmented Generation) adaugă LLM-ului un mecanism de „căutare" în documentele tale înainte să genereze răspunsul. Nu re-antrenezi modelul — îi dai contextul relevant în prompt.

RAG vs Fine-tuning
Fine-tuning modifică weights-urile modelului prin re-antrenare pe datele tale — scump, lent, greu de actualizat. RAG injectează datele ca context în prompt la inference time — ieftin, rapid, datele se actualizează instant. Pentru 95% din use cases, RAG este alegerea corectă.

Arhitectura RAG

RAG are două faze distincte:

A

Faza de indexare (offline)

Citești documentele, le fragmentezi în chunk-uri, calculezi embeddings, stochezi în vector database. Se face O SINGURĂ DATĂ sau la actualizarea documentelor.

offline
B

Faza de retrieval + generation (online)

La fiecare întrebare a userului: vectorizezi query-ul, retrievi top-K chunk-uri relevante, construiești promptul cu contextul, trimiți LLM-ului care generează răspunsul.

per request
1

User întreabă: „Care este politica de retur pentru produse electronice?"

input utilizator
↓
2

Retrieval: vectorizezi întrebarea și cauți top-3 chunk-uri similare în vector database.

embeddings + vector DB
↓
3

Construiești promptul: combini contextul găsit cu întrebarea userului.

prompt engineering
↓
4

LLM generează răspunsul pe baza contextului furnizat, nu din memoria sa de antrenament.

LLM inference
↓
5

Răspuns returnat userului cu posibile citate din documentele sursă.

output

Chunking — fragmentarea documentelor

Nu poți pune un document de 50 pagini în embedding. Modelele au limite de token-uri (ex. 8192 pentru text-embedding-3-small). Și chiar dacă ai putea, un chunk mai mic produce retrieval mai precis. 

def chunk_text(text: str, chunk_size: int = 500, overlap: int = 50) -> list[str]:
    """
    Fragmentează text în chunk-uri de chunk_size cuvinte cu overlap.
    Overlap-ul previne pierderea contextului la granița dintre chunk-uri.
    """
    words = text.split()
    chunks = []
    for i in range(0, len(words), chunk_size - overlap):
        chunk = " ".join(words[i:i + chunk_size])
        if chunk:
            chunks.append(chunk)
    return chunks

# Exemplu
text = "Politica de retur... (text lung de 5000 cuvinte)"
chunks = chunk_text(text, chunk_size=200, overlap=20)
# → 25 de chunk-uri de ~200 cuvinte fiecare
Chunk size — un parametru important
  • Chunk mic (100–200 cuvinte): retrieval precis, context puțin per chunk
  • Chunk mare (500–1000 cuvinte): mai mult context, dar retrieval mai puțin precis
  • Overlap (50–100 cuvinte): previne pierderea contextului la granițe
Valoare bună de start: 300–500 cuvinte cu overlap 50.

RAG complet în Python — de la zero

# pip install openai chromadb
import os
import chromadb
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
chroma = chromadb.Client()
collection = chroma.create_collection("docs", metadata={"hnsw:space": "cosine"})

# ──────────────────────────────
# FAZA 1: Indexare (offline)
# ──────────────────────────────

documents = {
    "doc1": """Politica de retur: Clienții pot returna produsele în termen de 30 de zile
    de la data achiziției. Produsele trebuie să fie în starea originală, cu ambalajul intact.
    Electronicele au o garanție de 2 ani.""",

    "doc2": """Metode de plată acceptate: card de credit/debit (Visa, Mastercard),
    PayPal, transfer bancar, plata la livrare. Rambursările se procesează în 5-7 zile lucrătoare.""",

    "doc3": """Livrare: Livrăm în toată România în 2-3 zile lucrătoare.
    Livrarea gratuită pentru comenzi peste 200 RON. Livrarea expresă (24h) costă 25 RON.""",
}

def embed(text: str) -> list[float]:
    res = client.embeddings.create(input=text, model="text-embedding-3-small")
    return res.data[0].embedding

# Indexează toate documentele
collection.add(
    ids=list(documents.keys()),
    documents=list(documents.values()),
    embeddings=[embed(doc) for doc in documents.values()],
)

# ──────────────────────────────
# FAZA 2: RAG la query time
# ──────────────────────────────

def rag_query(question: str, top_k: int = 2) -> str:
    # 1. Retrieval
    query_emb = embed(question)
    results = collection.query(query_embeddings=[query_emb], n_results=top_k)
    context_chunks = results["documents"][0]

    # 2. Construiește promptul
    context = "\n\n".join(context_chunks)
    prompt = f"""Ești un asistent de customer support. Răspunde DOAR pe baza contextului de mai jos.
Dacă informația nu se află în context, spune că nu știi.

Context:
{context}

Întrebarea clientului: {question}

Răspuns:"""

    # 3. LLM generează
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0,
    )
    return response.choices[0].message.content

# Test
answer = rag_query("Cât timp am la dispoziție să returnez un produs?")
print(answer)
# → "Aveți 30 de zile de la data achiziției pentru a returna produsele..."

Varianta hibridă — embeddings OpenAI + generare cu Claude

Dacă ai deja o subscripție Anthropic, poți folosi Claude pentru partea de generare din RAG — înlocuind gpt-4o-mini cu un model Claude. Embeddings-urile rămân la OpenAI (sau Sentence Transformers gratuit) deoarece Anthropic nu oferă un endpoint de embeddings.

Flux hibrid
Text → OpenAI embeddings → ChromaDB (retrieval) → Claude (generare răspuns)
Poți înlocui orice LLM în pasul de generare fără să schimbi nimic în pipeline-ul de embeddings/retrieval. 
# pip install openai chromadb anthropic
import os
import chromadb
from openai import OpenAI
import anthropic

openai_client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
claude_client = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
chroma = chromadb.Client()
collection = chroma.create_collection("docs", metadata={"hnsw:space": "cosine"})

# Faza 1: Indexare — identică, folosind OpenAI embeddings
documents = {
    "doc1": "Politica de retur: Clienții pot returna produsele în 30 de zile...",
    "doc2": "Metode de plată: card, PayPal, transfer bancar, plata la livrare...",
    "doc3": "Livrare în 2-3 zile lucrătoare. Gratuit peste 200 RON...",
}

def embed(text: str) -> list[float]:
    res = openai_client.embeddings.create(input=text, model="text-embedding-3-small")
    return res.data[0].embedding

collection.add(
    ids=list(documents.keys()),
    documents=list(documents.values()),
    embeddings=[embed(doc) for doc in documents.values()],
)

# Faza 2: RAG cu Claude pentru generare
def rag_query_claude(question: str, top_k: int = 2) -> str:
    # 1. Retrieval — identic cu varianta OpenAI
    query_emb = embed(question)
    results = collection.query(query_embeddings=[query_emb], n_results=top_k)
    context = "\n\n".join(results["documents"][0])

    # 2. Prompt — identic
    prompt = f"""Ești un asistent de customer support. Răspunde DOAR pe baza contextului de mai jos.
Dacă informația nu se află în context, spune că nu știi.

Context:
{context}

Întrebarea clientului: {question}

Răspuns:"""

    # 3. Generare cu Claude în loc de GPT
    response = claude_client.messages.create(
        model="claude-haiku-4-5",   # ieftin și rapid; sau claude-opus-4-8 pentru calitate maximă
        max_tokens=500,
        messages=[{"role": "user", "content": prompt}],
    )
    return response.content[0].text

# Test
answer = rag_query_claude("Cât timp am la dispoziție să returnez un produs?")
print(answer)
# → "Aveți 30 de zile de la data achiziției..."

Verifică-ți înțelegerea

Cinci întrebări. Dă clic pe un răspuns — feedback-ul este imediat.

Întrebarea 1 din 5 Faza de indexare în RAG (calculul embeddings-urilor documentelor) se execută:
Corect. Indexarea este offline — calculezi o dată și stochezi. La query time, calculezi DOAR embedding-ul întrebării (rapid, un singur call) și retrievi din index. Asta face RAG eficient.
Întrebarea 2 din 5 De ce RAG este preferat față de fine-tuning pentru knowledge base-uri care se actualizează frecvent?
Corect. Fine-tuning modifică weights-urile modelului — re-antrenarea costă bani și timp, și trebuie repetată la fiecare actualizare. RAG adaugă documentele în vector DB instantaneu. Noul document este disponibil la următorul query.
Întrebarea 3 din 5 Ce rol joacă overlap-ul în chunking-ul documentelor?
Corect. Dacă o frază importantă cade exact la granița a două chunk-uri, fără overlap ar fi fragmentată. Overlap de 50 cuvinte înseamnă că ultimele 50 cuvinte din chunk-ul N sunt și primele 50 din chunk-ul N+1 — contextul este prezent în ambele.
Întrebarea 4 din 5 De ce e important „Dacă informația nu se află în context, spune că nu știi" în promptul RAG?
Corect. Un LLM fără instrucțiuni clare va folosi training data când nu găsește răspunsul în context — și poate inventa fapte plauzibile dar false (hallucination). Instrucțiunea de fallback îl ancorează la sursele furnizate.
Întrebarea 5 din 5 Care setare a temperature-ului LLM este recomandată pentru RAG pe documente factuale?
Corect. Temperature = 0 face LLM-ul determinist — aceeași întrebare primește mereu același răspuns. Pentru fapte din documente, vrei consistență, nu creativitate. Temperature mai mare e bună pentru generare de text creativ, nu pentru Q&A factual.
